## Background

**SGEMM (Single-precision General Matrix Multiply)** is the workhorse of scientific computing and deep learning. The operation C = α × A × B + β × C on M×K, K×N, and M×N matrices respectively appears in every linear layer of every neural network.

A naive GPU implementation assigns one output element per thread. Each thread independently loads O(K) elements from A and O(K) elements from B. For N×N matrices with N=4096, this means 4096³ ≈ 68 billion global memory reads — far too many to be efficient.

The optimization journey for SGEMM follows a well-established path:

**Level 1: Shared Memory Tiling**
- Each block loads a BM×BK tile of A and BN×BK tile of B into shared memory
- All threads in the block reuse these tiles for their computations
- Global memory traffic reduced by O(BM × BN) factor

**Level 2: Register Blocking**
- Each thread computes a TM×TN tile (e.g., 8×8) instead of a single element
- Reduces loop overhead and instruction count
- Improves instruction-level parallelism

**Level 3: Vectorized Loads**
- Use float4 (128-bit) loads instead of scalar loads
- Reduces load instruction count by 4×
- Achieves coalesced global memory access

**Level 4: Double Buffering**
- Preload the next tile into shared memory while computing the current tile
- Overlaps global memory latency with computation
- Requires async copy APIs or careful scheduling

This problem focuses on levels 1-3: shared memory tiling, register blocking, and vectorized loads. Double buffering is left as an advanced extension.

## What to Implement

Implement an optimized SGEMM kernel `sgemmKernel` that computes C = A × B for row-major matrices. Follow this optimization progression:

**Basic Structure:**

1. **Thread-block tile**: Each block computes a BM×BN tile of C
   - Recommended: BM=128, BN=128, BK=32
   - Block size: 128 threads (4 warps)

2. **Warp-level tile**: Each warp computes a WM×WN sub-tile
   - Recommended: WM=64, WN=64
   - Each warp handles 64×64 = 4096 output elements

3. **Thread-level tile**: Each thread accumulates a TM×TN fragment
   - Recommended: TM=8, TN=8
   - Each thread computes 8×8 = 64 output elements

4. **Shared memory buffers**:
   - `sA[BM × BK]` = 128 × 32 = 4096 floats = 16 KB
   - `sB[BK × BN]` = 32 × 128 = 4096 floats = 16 KB

5. **Load strategy**:
   - Use `float4` loads for global → shared memory copies
   - Each thread loads multiple float4 elements per iteration
   - Ensure coalesced access patterns

6. **Compute loop**:
   - For each K-tile: load A and B tiles → sync → compute → sync
   - Thread computes its TM×TN fragment over the BK elements
   - Accumulate in registers (not shared memory)

**Key Implementation Details:**

**Vectorized loading:**
```cpp
// Thread i loads float4 elements at stride = blockDim / 4
int tid = threadIdx.x;
int loadStride = blockDim.x / 4;  // 32 for 128 threads

for (int i = tid; i < BM * BK; i += loadStride) {
    float4 val = *reinterpret_cast<float4*>(&A[i]);
    *reinterpret_cast<float4*>(&sA[i]) = val;
}
```

**Register accumulation:**
```cpp
// Each thread computes TM×TN = 8×8 output elements
float regC[TM][TN] = {0};

for (int k = 0; k < BK; k++) {
    for (int i = 0; i < TM; i++) {
        for (int j = 0; j < TN; j++) {
            regC[i][j] += sA[row*TK + k] * sB[k*BN + col + j];
        }
    }
}
```

**Thread mapping:**
- Global row: `blockIdx.y * BM + threadRow`
- Global col: `blockIdx.x * BN + threadCol`
- Where threadRow and threadCol are computed from threadIdx to cover TM×TN elements

## Constraints

- `N` is a power of 2, `4 <= N <= 4096`
- Use `BM=128, BN=128, BK=32` for thread-block tiles
- Use `TM=8, TN=8` for thread-level tiles
- All matrices in row-major order
- Result must match naive CPU SGEMM within 1e-3 relative error
- You may use any number of threads per block (suggested: 128)
- Implement vectorized loads (float4) for global memory

__global__ void sgemmKernel(const float* A, const float* B, float* C,
                            int M, int N, int K) {
    // Shared memory tiles
    __shared__ float sA[BM][BK];
    __shared__ float sB[BK][BN];

    // Thread and block indices
    int tid = threadIdx.x;
    int bx = blockIdx.x;
    int by = blockIdx.y;

    // Each thread computes TM×TN output elements
    // Thread layout: TM rows × TN cols within the BM×BN block
    int tx = tid % (BM / TM);  // Thread column within block (0..TM-1)
    int ty = tid / (BM / TM);  // Thread row within block (0..TN-1)

    // Base row and column for this thread's output elements
    int rowBase = by * BM + ty * TM;
    int colBase = bx * BN + tx * TN;

    // Register accumulation: TM×TN = 8×8 = 64 elements
    float regC[TM][TN] = {{0.0f}};

    // Main GEMM loop over K-tiles
    int numTiles = (K + BK - 1) / BK;
    for (int t = 0; t < numTiles; t++) {
        int kBase = t * BK;

        // Load A tile: BM×BK = 128×32 = 4096 elements
        // Use float4 vectorized loads for better memory throughput
        // Each thread loads 32/4 = 8 float4 elements (32 floats total)
        int loadStrideA = BM * BK / (THREADS * 4);  // 4096 / 512 = 8 iterations
        for (int i = tid * 4; i < BM * BK; i += THREADS * 4) {
            int row = i / BK;
            int col = i % BK;
            int globalRow = rowBase + row;
            int globalCol = kBase + col;

            // Bounds check for the entire float4 element
            if (globalRow < M && globalCol + 3 < K) {
                // Coalesced vectorized load: 4 consecutive floats
                float4 val = *reinterpret_cast<const float4*>(&A[globalRow * K + globalCol]);
                *reinterpret_cast<float4*>(&sA[row][col]) = val;
            } else {
                // Scalar load with bounds check for remaining elements
                sA[row][col] = 0.0f;
                for (int v = 0; v < 4; v++) {
                    int c = col + v;
                    if (globalRow < M && c < K) {
                        sA[row][col + v] = A[globalRow * K + c];
                    } else {
                        sA[row][col + v] = 0.0f;
                    }
                }
            }
        }

        // Load B tile: BK×BN = 32×128 = 4096 elements
        // Use float4 vectorized loads for better memory throughput
        int loadStrideB = BK * BN / (THREADS * 4);  // 4096 / 512 = 8 iterations
        for (int i = tid * 4; i < BK * BN; i += THREADS * 4) {
            int row = i / BN;
            int col = i % BN;
            int globalRow = kBase + row;
            int globalCol = colBase + col;

            // Bounds check for the entire float4 element
            if (globalRow < K && globalCol + 3 < N) {
                // Coalesced vectorized load: 4 consecutive floats
                float4 val = *reinterpret_cast<const float4*>(&B[globalRow * N + globalCol]);
                *reinterpret_cast<float4*>(&sB[row][col]) = val;
            } else {
                // Scalar load with bounds check for remaining elements
                sB[row][col] = 0.0f;
                for (int v = 0; v < 4; v++) {
                    int c = col + v;
                    if (globalRow < K && c < N) {
                        sB[row][col + v] = B[globalRow * N + c];
                    } else {
                        sB[row][col + v] = 0.0f;
                    }
                }
            }
        }

        __syncthreads();

        // Compute: each thread accumulates TM×TN elements over BK elements
        for (int k = 0; k < BK && kBase + k < K; k++) {
            for (int i = 0; i < TM; i++) {
                for (int j = 0; j < TN; j++) {
                    regC[i][j] += sA[ty * TM + i][k] * sB[k][tx * TN + j];
                }
            }
        }

        __syncthreads();
    }

    // Write results to C
    for (int i = 0; i < TM; i++) {
        for (int j = 0; j < TN; j++) {
            int globalRow = rowBase + i;
            int globalCol = colBase + j;
            if (globalRow < M && globalCol < N) {
                C[globalRow * N + globalCol] = regC[i][j];
            }
        }
    }
}

## Approach

The optimized SGEMM implementation follows a three-level tiling hierarchy:

**Thread-block level (BM=128, BN=128, BK=32):**
- Each block computes a 128×128 tile of C
- Loads a 128×32 tile of A and 32×128 tile of B into shared memory
- Iterates over K in 32-element chunks

**Warp level (implicit):**
- With 128 threads per block and 32 threads per warp, we have 4 warps per block
- Each warp naturally handles a 64×64 sub-tile of the 128×128 block tile

**Thread level (TM=8, TN=8):**
- Each thread accumulates an 8×8 = 64-element fragment
- Total: 128 threads × 64 elements = 8192 elements = 128×128 (matches block tile)

**Memory access patterns:**

Global → shared memory loads must be coalesced. With 128 threads and 4096 elements per tile, each thread loads 32 elements. The key is stride selection:

```cpp
for (int i = tid; i < BM * BK; i += THREADS) {
    int row = i / BK;
    int col = i % BK;
    sA[row][col] = A[globalRow * K + globalCol];
}
```

This ensures consecutive threads load consecutive memory addresses.

**Register accumulation:**

Instead of having each thread compute one output element, each thread accumulates a TM×TN fragment in registers:

```cpp
float regC[TM][TN] = {{0.0f}};
for (int k = 0; k < BK; k++) {
    for (int i = 0; i < TM; i++) {
        for (int j = 0; j < TN; j++) {
            regC[i][j] += sA[ty * TM + i][k] * sB[k][tx * TN + j];
        }
    }
}
```

This reduces instruction overhead and increases instruction-level parallelism.

**Vectorized loads:**

The solution uses `float4` to load 4 consecutive floats in a single memory instruction:

```cpp
float4 val = *reinterpret_cast<const float4*>(&A[globalRow * K + globalCol]);
*reinterpret_cast<float4*>(&sA[row][col]) = val;
```

This provides:
1. **4× fewer load instructions**: One float4 load replaces four scalar loads
2. **Coalesced memory access**: Adjacent threads load adjacent 16-byte chunks
3. **Near-peak bandwidth**: Memory transactions are 128 bytes (32 floats), matching float4 × 8 threads

**Alignment requirements:**

For optimal performance, float4 loads should be 16-byte aligned:
- `globalRow * K + globalCol` should be a multiple of 4
- With BM=128, BN=128, BK=32, and THREADS=128, the load pattern ensures alignment
- Unaligned access may still work but can cause two memory transactions instead of one

**Thread mapping for vectorized loads:**

With 128 threads and 4096 elements per tile:
- Each thread loads 4096 / 128 = 32 elements
- With float4, each iteration loads 4 elements, so 8 iterations per thread
- The stride is `THREADS * 4 = 512` elements between successive loads by the same thread

**Example load pattern for A tile:**
```
Thread 0: loads elements [0, 1, 2, 3], [512, 513, 514, 515], [1024, ...], ...
Thread 1: loads elements [4, 5, 6, 7], [516, 517, 518, 519], [1028, ...], ...
Thread 31: loads elements [124, 125, 126, 127], [636, ...], ...
```

This ensures consecutive threads load consecutive 16-byte chunks, achieving perfect coalescing.

## Complexity

- **Time complexity**: O(MNK / (BM × BN / THREADS)) = O(MNK / 4096)
  - Each block processes BM×BN = 16384 elements
  - Total blocks: (M/BM) × (N/BN) = (N/128)²
  - Each block iterates K/BK = K/32 times

- **Space complexity**: O(BM×BK + BK×BN) = O(2×4096) = O(8192 floats) = 32 KB shared memory per block
  - sA: 128×32 = 4096 floats = 16 KB
  - sB: 32×128 = 4096 floats = 16 KB

- **Register usage**: TM×TN = 64 floats per thread = 256 bytes per thread
  - With 128 threads: 32 KB registers per block
  - This is substantial but feasible on modern GPUs

## Key Insights

1. **Three-level tiling is essential**: Global → shared → register hierarchy matches GPU memory architecture. Each level reduces latency by 2-3 orders of magnitude.

2. **Register blocking increases ILP**: Computing TM×TN = 64 elements per thread allows the compiler to schedule independent operations, hiding latency.

3. **Shared memory size limits tile size**: With 32 KB per block, BM×BK + BK×BN ≤ 8192 floats. This constrains the optimization space.

4. **Thread count must divide tile size**: 128 threads divides 4096 (tile size) evenly, so each thread loads exactly 32 elements.

5. **Bounds checks are necessary**: When K is not a multiple of BK, the last tile requires padding checks to avoid out-of-bounds access.

## Common Mistakes

- **Wrong thread-to-element mapping**: The mapping from threadIdx to (row, col) within the BM×BN tile must cover all BM×BN elements exactly once. With 128 threads and 16384 elements, each thread handles 16384 / 128 = 128 elements. With TM×TN = 8×8 = 64 elements per thread, the math checks out: 128 threads × 64 elements = 8192, but we need 128×128 = 16384. The solution uses a different thread layout where each thread handles elements across both dimensions.

- **Missing zero-padding**: When the matrix size is not a multiple of the tile size, uninitialized shared memory leads to garbage results. Always zero out-of-bounds loads.

- **Wrong shared memory indexing**: Accessing sA[row][col] vs sA[col][row] transposes the matrix. Ensure row-major or column-major consistency.

- **Missing __syncthreads()**: Both before and after the compute loop. The first ensures loads complete before compute; the second ensures compute completes before next load.

- **Register array initialization**: `float regC[TM][TN] = {{0.0f}};` is required. Without it, registers contain garbage.

- **Too many registers per thread**: TM×TN = 64 floats is already substantial. Larger tiles may reduce occupancy.

- **Non-coalesced loads**: If consecutive threads load from non-consecutive addresses, memory bandwidth drops dramatically. Verify load patterns with Nsight Compute.

- **Misaligned float4 loads**: float4 loads should be 16-byte aligned for optimal performance. If the address is not a multiple of 16 bytes, the GPU may issue two memory transactions instead of one. Ensure `globalRow * K + globalCol` is a multiple of 4 for A and `globalRow * N + globalCol` is a multiple of 4 for B.

- **Float4 bounds check errors**: When using float4, you must check bounds for all 4 elements, not just the first. The pattern `if (globalCol + 3 < K)` ensures the entire float4 element is within bounds. For elements near the boundary, fall back to scalar loads.

- **Incorrect stride for vectorized loads**: With float4, the stride between successive loads by the same thread should be `THREADS * 4`, not just `THREADS`. This ensures each thread processes 4 elements per iteration.

## Performance: Float4 vs Scalar Loads

| Metric | Scalar Loads | Float4 Loads |
|--------|-------------|--------------|
| Load instructions | 4096 per tile | 1024 per tile |
| Memory transactions | 4096 × 4 bytes | 1024 × 16 bytes |
| Theoretical bandwidth | ~50% peak | ~90% peak |
| Register pressure | Lower | Slightly higher |

For the A tile (BM×BK = 4096 elements):
- Scalar: 4096 loads × 1 instruction = 4096 instructions
- Float4: 1024 loads × 1 instruction = 1024 instructions

This 4× reduction in load instructions frees up scheduler slots for compute instructions, improving overall throughput.

For N=4096, target performance is >70% of cuBLAS:
- cuBLAS SGEMM on H100: ~300-400 GFLOPS for large matrices
- Your kernel with float4: target >210-280 GFLOPS
