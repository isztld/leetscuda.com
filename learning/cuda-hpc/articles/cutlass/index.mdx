# CUTLASS Abstractions

## Why this matters for interviews

CUTLASS questions appear less frequently than GEMM fundamentals or Tensor Core questions, but when they do, they signal that the interviewer is probing deep into LLM infrastructure optimization. At NVIDIA, Meta AI, and Microsoft Azure ML, senior engineers are expected to understand why CUTLASS achieves ~95% of Tensor Core peak throughput and how its abstractions map to the GPU memory hierarchy. Expect "How would you implement a custom attention epilogue?" or "Why can't you just use cuBLAS for quantized inference?" These questions test whether you've worked with production LLM kernels or only high-level frameworks.

CUTLASS is NVIDIA's open-source C++ template library for high-performance GEMM. It decomposes matrix multiplication into reusable components at every level of the GPU hierarchy: threadblock tiling, warp-level MMA pipelines, and thread-level register blocking. Version 4 adds CuTe DSL, a Python-native interface that exposes the same abstractions without C++ template metaprogramming complexity. Understanding CUTLASS is essential for anyone building custom LLM inference kernels, quantized training pipelines, or fused attention operators.

## The Problem CUTLASS Solves

Writing high-performance GEMM kernels from scratch requires managing three independent concerns simultaneously: (1) **Algorithmic correctness** — the mathematical GEMM computation C = αA×B + βC, (2) **Memory hierarchy optimization** — loading tiles from global memory to shared memory to registers, and (3) **Hardware-specific instructions** — Tensor Core MMA, async copy, warpgroup operations. CUTLASS separates these concerns into composable C++ templates.

Before CUTLASS, engineers either used cuBLAS (closed-source, fixed epilogues) or wrote custom kernels (high maintenance, architecture-specific). CUTLASS enables custom epilogues (bias addition, activations, quantization) while maintaining near-peak throughput across architectures from Volta to Blackwell. This is critical for LLM inference, where attention kernels fuse Q@K^T with softmax, bias, and dropout in a single kernel launch.

**The key insight:** High-performance GEMM follows a universal three-level tiling hierarchy, regardless of GPU generation. CUTLASS expresses this hierarchy as template parameters:

1. **Threadblock tile** (BM×BN×BK): Each CUDA thread block computes one output tile by streaming over the K dimension in BK-sized steps. For H100, typical values are BM=128, BN=128, BK=32, loading 128×32×4B = 16KB per matrix into shared memory.

2. **Warp tile** (WM×WN): Each warp within the block computes a subtile using register blocking. For H100 with 16 threads/warp, a WM=64, WN=64 warp tile means each thread handles 64×8 elements in registers.

3. **Thread tile** (TM×TN): Each thread accumulates a TM×TN fragment. With Tensor Cores, the instruction shape is M16×N16×K16 (or variants like M64×N4×K16 for WGMMA).

CUTLASS maps these levels to collective abstractions: `GemmShape<BM, BN, BK, WM, WN, WN, TM, TN, TK>` becomes template parameters for `cutlass::gemm::device::Gemm`. The compiler instantiates the exact instruction sequence for the target architecture.

## CUTLASS Architecture Decomposition

CUTLASS decomposes GEMM into four main components, each corresponding to a level in the GPU execution hierarchy.

### GemmKernel: Threadblock-Level Scheduling

The `GemmKernel` class orchestrates the threadblock-level computation. Each thread block:

1. **Prologue**: Load the first K-tile of A and B matrices from global memory to shared memory using coalesced loads. Vectorized loads (float4 / 128-bit) maximize memory throughput.

2. **Mainloop**: Iterate over remaining K-tiles. For each K-step:
   - Load next K-tile into shared memory (using async copy on Hopper)
   - Compute current tile using warp-level MMA
   - Double-buffer: overlap load of tile N+1 with computation of tile N

3. **Epilogue**: Write the final output tile C to global memory, applying post-GEMM operations (bias, activation, quantization).

**Example threadblock tiling for H100 BF16 GEMM:**
```cpp
using ThreadblockShape = cutlass::gemm::GemmShape<128, 128, 32,  // BM, BN, BK
                                                   8, 8, 2>;       // warp M, N, K
using WarpShape = cutlass::gemm::GemmShape<64, 64, 32, 64, 8>;   // WM, WN, TM, TN, TK
using InstructionShape = cutlass::gemm::GemmShape<16, 8, 16>;     // m16n8k16 for BF16
```

The `ThreadblockShape` defines the global memory access pattern. A 128×128 output tile means each block is responsible for 128×128 = 16,384 output elements. With BK=32 and 32 threads per block (16 warps), each K-step computes 128×32 + 32×128 = 8,192 intermediate products per matrix. Over 128 K elements, this accumulates to the full GEMM.

### Mma: Warp-Level MMA Pipeline

The `Mma` abstraction exposes the warp-level matrix multiply pipeline. On Ampere and Hopper, this uses Tensor Core instructions:

- **Ampere (cc 8.0+)**: `mma.sync.aligned.m16n8k16.row.col.f32.f16.f16.f32` — 16×8×16 BF16→FP32 MMA
- **Hopper (cc 9.0+)**: WGMMA `wgmma.mma_async.sync.aligned.m64n16k16.f32.bf16.bf16` — 64×16×16 across 4 warps

**Hopper WGMMA example (4 warps = 128 threads):**
```cpp
// WGMMA operates on 4 warps (128 threads) collectively
// Each warp contributes a 16×16 fragment
cute::gemm::collective::MmaOp<
  cute::Shape<cute::Int<64>, cute::Int<16>, cute::Int<16>>,  // M, N, K
  cute::CutlassArch::Sm100                                   // Blackwell
>::apply(src_A, src_B, dst_C);
```

The `Mma` abstraction is architecture-aware: for Ampere it emits WMMA instructions, for Hopper it emits WGMMA instructions, for Blackwell it emits UMMA instructions. The caller does not need to know which instruction is used — CUTLASS selects the optimal variant based on `cute::CutlassArch`.

### Epilogue: Post-GEMM Operations

The epilogue is where CUTLASS differs most from cuBLAS. cuBLAS's `cublasGemmEx` supports only fixed epilogues: C = αA×B + βC. CUTLASS's epilogue is composable via the **Epilogue Visitor Tree (EVT)**:

- `BiasAdder`: Add bias vector to output
- `ActivationApplicator`: Apply ReLU, GELU, SILU in-place
- `Quantizer`: Convert FP32→FP16/INT8/FP8 with scale
- `ResidualFuser`: Combine with skip connection

**Example: GELU-activated GEMM epilogue:**
```cpp
using EpilogueOutputOp = cutlass::epilogue::thread::Gelu;

using Gemm = cutlass::gemm::device::Gemm<
  bf16_t,                    // A type
  cutlass::layout::RowMajor, // A layout
  bf16_t,                    // B type
  cutlass::layout::ColumnMajor, // B layout
  float,                     // C type
  cutlass::layout::RowMajor, // C layout
  float,                     // accumulation type
  cutlass::arch::OpClassTensorOp,
  cutlass::SM80,
  ThreadblockShape,
  WarpShape,
  InstructionShape,
  EpilogueOutputOp  // <-- Custom epilogue
>;
```

The EVT allows composition: `Gelu(BiasAdder(BiasVector))` fuses bias addition and GELU into a single pass over the output tile. This is critical for attention kernels, where each Q@K^T must be softmax-normalized before the V@softmax multiplication.

### CuTe: Layout and Tensor Abstractions

CUTLASS 3 introduced **CuTe**, a C++ template library for expressing tensor layouts and operations independently of data types. A CuTe `Tensor` is a pair: `(pointer, Layout)` where `Layout = (shape, stride)`.

**CuTe layout example:**
```cpp
#include <cute/layout.hpp>
using namespace cute;

// 2D layout: shape (M, N), column-major strides (1, M)
auto layout = make_layout(make_shape(Int<128>{}, Int<64>{}),
                          make_stride(Int<1>{}, Int<128>{}));

// Access element (i, j)
auto element = layout(i, j);  // Returns pointer + offset
```

CuTe provides the algebraic operations needed for tiling:
- `tile_shape(shape, Int<M>, Int<N>)`: Partition a large shape into tiles
- `partition_tile(layout, tile_id)`: Compute the layout for a specific tile
- `make_tensor(pointer, layout)`: Create a typed view over memory

**CuTe tiling example for GEMM:**
```cpp
auto problem_shape = make_shape(Int<M>{}, Int<N>{}, Int<K>{});
auto threadblock_tile = make_shape(Int<128>{}, Int<128>{}, Int<32>{});

// Partition problem into threadblock tiles
auto tiles = partition<2>(problem_shape, threadblock_tile);
// tiles.shape() = (M/128, N/128, K/32)
```

CuTe DSL (CUTLASS 4) exposes these abstractions in Python, enabling kernel development without C++ template compilation.

## CuTe DSL: Python-Native Kernel Programming

CUTLASS 4.0 introduces **CuTe DSL**, a Python framework for writing CUDA kernels that compiles to PTX. The key innovation is that CuTe DSL uses the same abstractions as CUTLASS C++ — `Layout`, `Tensor`, `MmaOp` — but without C++ template metaprogramming.

**CuTe DSL HGEMM example:**
```python
import cutlass.cute as cute

@cute.kernel
def hgemm_kernel(A, B, C, M, N, K):
    # Threadblock tile: 128x128x32
    tm, tn, tk = 128, 128, 32

    # Compute this block's tile coordinates
    bx, by = cute.block_idx(), cute.block_y()
    row_start = by * tm
    col_start = bx * tn

    # Load tiles from global memory
    A_tile = cute.load_gm(A + row_start * K + col_start * K, (tm, tk))
    B_tile = cute.load_gm(B + col_start * K + row_start * K, (tk, tn))

    # Compute MMA using WMMA
    C_tile = cute.zeros((tm, tn), dtype=cute.float32)
    cute.mma(A_tile, B_tile, C_tile)

    # Store result
    cute.store_sm(C + row_start * N + col_start * N, C_tile)

# Launch kernel
hgemm_kernel[grid_dim](dA, dB, dC, M, N, K)
```

The benefits of CuTe DSL:
- **Faster iteration**: Python compiles in milliseconds vs. C++ minutes
- **Native PyTorch integration**: Use PyTorch tensors directly
- **No template metaprogramming**: Layout arithmetic is explicit, not inferred
- **Architecture-generic**: Same code runs on Ampere, Hopper, Blackwell

## CUTLASS Version Evolution

**CUTLASS 2.x (2019-2021):** Introduced the three-level tiling hierarchy. Heavy use of C++ template metaprogramming made kernels hard to read. Epilogue support limited to simple operations.

**CUTLASS 3.x (2022-2023):** Refactored to use CuTe abstractions. Threadblock, warp, and instruction shapes unified under `GemmShape`. Introduced **Collective** abstractions: `GemmCollection`, `MmaOp`, `EpilogueVisitor`.

**CUTLASS 4.x (2024-present):** Added CuTe DSL (Python). Introduced **EVT (Epilogue Visitor Tree)** for composable epilogues. Added Blackwell support: FP4/FP6/FP8 block-scaled GEMM.

## Common Misconceptions

**❌ Wrong:** CUTLASS is just a wrapper around cuBLAS.
**✓ Correct:** CUTLASS is source-level C++ templates. It implements GEMM from first principles using Tensor Core instructions (WMMA/WGMMA). It can outperform cuBLAS for small matrices (<256) or custom epilogues.

**❌ Wrong:** CUTLASS is too complex to use directly.
**✓ Correct:** CUTLASS 4's CuTe DSL provides Python-native interfaces. Even in C++, the high-level `cutlass::gemm::device::Gemm` API requires only 20 lines of code.

**❌ Wrong:** CUTLASS only works with specific matrix sizes.
**✓ Correct:** CUTLASS handles arbitrary sizes via tilizing and padding. For M not divisible by BM, CUTLASS launches a partial grid and masks out-of-bounds threads.

**❌ Wrong:** CUTLASS is slower than cuBLAS for large matrices.
**✓ Correct:** For large square GEMM (N>2048), CUTLASS and cuBLAS achieve ~95% of Tensor Core peak. The difference is <5% for well-tuned configurations.

**❌ Wrong:** CUTLASS is only for GEMM.
**✓ Correct:** CUTLASS supports convolution (via im2col GEMM), reduction, batched GEMM, sparse GEMM (2:4 structured sparsity), and Flash Attention variants.

**❌ Wrong:** CuTe DSL is a different algorithm from CUTLASS C++.
**✓ Correct:** CuTe DSL compiles to the exact same PTX as CUTLASS C++. The only difference is the frontend language.

**❌ Wrong:** CUTLASS automatically uses Tensor Cores for all data types.
**✓ Correct:** Tensor Core usage requires matching `InstructionShape` to data type. FP32 GEMM uses CUDA Cores unless explicitly configured for TF32 Tensor Cores.

**❌ Wrong:** CUTLASS 4 replaces CUTLASS C++.
**✓ Correct:** CUTLASS 4 adds CuTe DSL as an alternative frontend. C++ templates remain the primary production path; DSL is for research and rapid prototyping.

## Key Numbers to Memorize

| Parameter | H100 (BF16) | A100 (BF16) | RTX 4090 (FP16) |
|-----------|--------------|---------------|-------------------|
| Block tile (BM×BN×BK) | 128×128×32 | 128×128×32 | 128×128×16 |
| Warp tile (WM×WN) | 64×64 | 64×64 | 64×64 |
| Instruction shape | m64n16k16 (WGMMA) | m16n8k16 (WMMA) | m16n8k16 (WMMA) |
| Shared memory per block | 128 KB | 64 KB | 64 KB |
| Max blocks per SM | 8-16 | 8-16 | 8-16 |
| Throughput (% of peak) | ~95% | ~93% | ~90% |

**Throughput comparison (H100, M=N=K=4096):**
- cuBLAS `cublasGemmEx` BF16: 989 TFLOPS (100% of Tensor Core peak)
- CUTLASS 3.8 BF16: 934 TFLOPS (~95% of peak)
- Custom FP32 GEMM: 67 TFLOPS (~100% of CUDA Core peak, but 15x slower)

**Build time comparison:**
- CUTLASS C++ (full kernel): 5-15 minutes (template instantiation)
- CuTe DSL kernel: <100 milliseconds
- cuBLAS integration: <10 seconds (header-only)

**Memory bandwidth requirements:**
- H100 GEMM (N=4096): ~3.0 TB/s required for 100% utilization
- A100 GEMM (N=4096): ~2.0 TB/s required for 100% utilization
- RTX 4090 GEMM (N=4096): ~1.0 TB/s required for 100% utilization

For N < 512, GEMM is **memory-bandwidth limited** — all implementations achieve similar throughput. For N > 2048, GEMM becomes **compute-limited** — Tensor Core throughput dominates.

## When to Use CUTLASS vs cuBLAS

**Use cuBLAS when:**
- You need simple GEMM without custom epilogues
- Your team lacks C++ template expertise
- You need rapid prototyping (hours vs. days)
- Matrix sizes are large (N > 2048) and fixed

**Use CUTLASS when:**
- You need custom epilogues (fusion with activation, quantization)
- You're implementing new architectures (e.g., Blackwell FP4)
- You need small-matrix optimization (N < 256)
- You're building LLM inference kernels (attention, MoE routing)

**Use CuTe DSL when:**
- You're experimenting with new algorithms
- You need fast iteration (research, prototyping)
- You're integrating with PyTorch directly
- You want architecture-generic code without C++ complexity

## Interview Patterns

**"How would you implement a custom attention kernel with fused softmax?"**

Start with the math: attention = softmax(Q@K^T / sqrt(d)) @ V. Explain that naive implementation materializes the N×N attention matrix, which is O(N^2) memory. CUTLASS enables tiling the Q@K^T computation into blocks, computing softmax incrementally, and fusing the V multiplication. This is the FlashAttention algorithm.

Describe the CUTLASS structure: `GemmKernel` computes Q@K^T tiles, `EpilogueVisitor` applies softmax normalization in register, then another `GemmKernel` multiplies with V. The key insight is that softmax can be computed online (without materializing the full matrix) using the "online softmax" algorithm from FlashAttention paper.

**"Why does CUTLASS achieve higher throughput than naive GEMM?"**

Three reasons: (1) **Memory hierarchy optimization** — CUTLASS loads tiles to shared memory once, reusing them across 128×128 = 16,384 output elements. Naive GEMM loads each element O(N) times. (2) **Tensor Core utilization** — CUTLASS emits WGMMA/WMMA instructions for 16×8×16 or 64×16×16 MMA. Naive GEMM uses scalar FP32, 15x slower. (3) **Occupancy tuning** — CUTLASS configures register usage and shared memory to maximize warps per SM, hiding memory latency. Naive GEMM often spills registers or underutilizes SMs.

**"What is the Epilogue Visitor Tree and why does it matter?"**

The EVT is a composable expression tree of post-GEMM operations. Each node is a functor: `BiasAdder`, `Gelu`, `Quantizer`. CUTLASS fuses them into a single kernel launch, avoiding redundant memory passes. For LLM inference, this matters because attention kernels fuse 4+ operations: Q@K^T, scale, softmax, dropout, @V. Without fusion, each operation requires a separate kernel launch with its own global memory pass, which is bandwidth-limited.

**"How would you add Blackwell FP4 support to CUTLASS?"**

Describe the steps: (1) Define `FP4` type (4-bit, block-scaled), (2) Implement `MmaOp<FP4, FP4, FP32>` using Blackwell UMMA instructions, (3) Add dequantization logic in epilogue (FP4→FP16→FP32), (4) Tune block tile size for FP4's reduced memory footprint (16× larger than FP32), (5) Verify against CPU reference and benchmark vs FP16.
