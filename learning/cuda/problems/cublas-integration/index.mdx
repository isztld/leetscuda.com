## Background

**cuBLAS** is NVIDIA's production-grade BLAS library for GPU. For matrix multiplication (GEMM), cuBLAS achieves 90-95% of the theoretical Tensor Core peak on modern GPUs. Understanding how to correctly integrate cuBLAS is essential for any GPU engineer building ML systems.

The core GEMM operation is:

**C = α × A × B + β × C**

where A is M×K, B is K×N, and C is M×N. The total FLOPs is 2MNK (one multiply + one accumulate per output element).

The critical insight for cuBLAS usage is **column-major convention**. cuBLAS follows Fortran's column-major layout, while C/C++ uses row-major. This means:

- Row-major A[M][K] is stored as A[K][M] in cuBLAS
- To compute C = A × B in row-major, compute C^T = B^T × A^T in cuBLAS
- The "transpose trick" swaps A and B, avoiding any data copy

This problem teaches you:
1. cuBLAS handle lifecycle (create, configure, destroy)
2. FP16 inputs with FP32 accumulation (mixed precision)
3. The row-major transpose trick
4. Comparing cuBLAS vs custom kernel performance

## What to Implement

Implement `cublasMatrixMultiply` that multiplies two N×N matrices using cuBLAS. The matrices use FP16 (half precision) for storage but FP32 for accumulation.

**Key implementation details:**

**Step 1: cuBLAS handle setup**
- Create handle with `cublasCreate()`
- Set math mode to `CUBLAS_TENSOR_OP_MATH` to enable Tensor Cores
- Destroy handle with `cublasDestroy()`

**Step 2: Memory allocation**
- Allocate device memory for A, B, C matrices (N×N × sizeof(half))
- Use `cublasMalloc` or regular `cudaMalloc` (both work)

**Step 3: Copy input data**
- Convert FP32 host matrices to FP16 for device
- Copy A and B to device using `cudaMemcpy`

**Step 4: Call cublasGemmEx**
- Use `cublasGemmEx()` for explicit control over data types
- Data types: `CUDA_R_16F` for FP16 inputs, `CUDA_R_32F` for FP32 accumulation
- Compute mode: `CUBLAS_COMPUTE_32F` for FP32 accumulation
- For Tensor Cores: `CUBLAS_GEMM_DEFAULT_TENSOR_OP`

**Step 5: Handle row-major input**
- Input matrices A (M×K) and B (K×N) are row-major
- Compute C = A × B by computing C^T = B^T × A^T
- This means: pass B as first matrix, A as second, with swapped dimensions
- Output is C^T, so transpose back to get C

**Step 6: Copy result and cleanup**
- Copy C from device to host
- Convert FP16 back to FP32
- Free all device memory, destroy handle

## Constraints

- `1 <= N <= 4096` (square matrices only for this problem)
- Input matrices are FP32 on host
- Use FP16 storage on device, FP32 accumulation
- Result must match naive FP32 matrix multiply within 1e-2 relative error
- Use `CUBLAS_TENSOR_OP_MATH` mode to enable Tensor Cores
- Properly manage cuBLAS handle lifecycle

void cublasMatrixMultiply(const float* hA, const float* hB, float* hC, int M, int N, int K) {
    cublasHandle_t handle;
    cublasCreate(&handle);

    // Set Tensor Core mode for better performance
    cublasSetMathMode(handle, CUBLAS_TENSOR_OP_MATH);

    // Device pointers for FP16 matrices
    half *dA, *dB, *dC;
    cudaMalloc(&dA, M * K * sizeof(half));
    cudaMalloc(&dB, K * N * sizeof(half));
    cudaMalloc(&dC, M * N * sizeof(half));

    // Host buffers for FP16 conversion
    std::vector<half> hA_fp16(M * K);
    std::vector<half> hB_fp16(K * N);

    // Convert FP32 to FP16
    fp32ToFp16(hA, hA_fp16.data(), M * K);
    fp32ToFp16(hB, hB_fp16.data(), K * N);

    // Copy to device (row-major layout)
    cudaMemcpy(dA, hA_fp16.data(), M * K * sizeof(half), cudaMemcpyHostToDevice);
    cudaMemcpy(dB, hB_fp16.data(), K * N * sizeof(half), cudaMemcpyHostToDevice);

    // Alpha and beta scalars
    const float alpha = 1.0f;
    const float beta = 0.0f;

    // Row-major GEMM: C = A × B (row-major)
    // cuBLAS is column-major, so compute C^T = B^T × A^T
    // This means: first matrix = B (N×K), second = A (K×M), result = C^T (N×M)
    cublasGemmEx(
        handle,
        CUBLAS_OP_N, CUBLAS_OP_N,  // No transpose
        N, M, K,                   // ldb, lda, ldc (note: N, M, K swapped for row-major trick)
        &alpha,
        dB, CUDA_R_16F, K,         // B is K×N in row-major, so N×K in column-major
        dA, CUDA_R_16F, M,         // A is M×K in row-major, so K×M in column-major
        &beta,
        dC, CUDA_R_16F, N,         // C^T is N×M in column-major
        CUBLAS_COMPUTE_32F,        // FP32 accumulation
        CUBLAS_GEMM_DEFAULT_TENSOR_OP
    );

    // Copy result back and convert to FP32
    std::vector<half> hC_fp16(M * N);
    cudaMemcpy(hC_fp16.data(), dC, M * N * sizeof(half), cudaMemcpyDeviceToHost);
    fp16ToFp32(hC_fp16.data(), hC, M * N);

    // Cleanup
    cudaFree(dA);
    cudaFree(dB);
    cudaFree(dC);
    cublasDestroy(handle);
}

static void referenceMatMul(const std::vector<float>& A, const std::vector<float>& B,
                           std::vector<float>& C, int N) {
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < N; j++) {
            float sum = 0.0f;
            for (int k = 0; k < N; k++) {
                sum += A[i * N + k] * B[k * N + j];
            }
            C[i * N + j] = sum;
        }
    }
}

## Approach

The cuBLAS GEMM integration demonstrates the "boring but essential" GPU engineering skill: using production libraries correctly and efficiently.

**cuBLAS handle lifecycle:**
```cpp
cublasHandle_t handle;
cublasCreate(&handle);
cublasSetMathMode(handle, CUBLAS_TENSOR_OP_MATH);
// ... use handle ...
cublasDestroy(handle);
```

**The row-major transpose trick:**

cuBLAS uses column-major layout (Fortran convention). To compute C = A × B where A, B, C are row-major:

1. In column-major: A^T is K×M, B^T is N×K
2. C^T = B^T × A^T (column-major multiplication)
3. The result C^T is M×N in column-major, which is C in row-major

This means passing B as the first matrix and A as the second, with dimensions swapped:
```cpp
cublasGemmEx(handle, CUBLAS_OP_N, CUBLAS_OP_N,
             N, M, K,  // ldb, lda, ldc (note the swap!)
             &alpha, dB, CUDA_R_16F, K,  // B is now first
             dA, CUDA_R_16F, M,          // A is now second
             &beta, dC, CUDA_R_16F, N,
             CUBLAS_COMPUTE_32F,
             CUBLAS_GEMM_DEFAULT_TENSOR_OP);
```

**Mixed precision:**

Using FP16 storage with FP32 accumulation:
- `CUDA_R_16F` for input matrices A, B, C
- `CUBLAS_COMPUTE_32F` for FP32 accumulation

This gives the memory bandwidth benefits of FP16 while maintaining numerical stability from FP32 accumulation.

**Tensor Core usage:**

The `CUBLAS_GEMM_DEFAULT_TENSOR_OP` flag tells cuBLAS to use Tensor Cores when beneficial. For square matrices M=N=K >= 64, Tensor Cores are typically faster than FP32 CUDA cores.

## Complexity

- **Time complexity**: O(MNK) for the GEMM operation
  - cuBLAS achieves O(1) constant factor via Tensor Cores and highly optimized microkernels
  - For M=N=K=2048 on H100: ~3-5ms (90-95% of Tensor Core peak)

- **Space complexity**: O(MN + KN + KM) for device memory
  - FP16 storage halves memory compared to FP32
  - No additional workspace needed for small GEMM

- **Numerical accuracy**: FP32 accumulation ensures relative error <= 1e-2 vs FP32 reference
  - FP16 has ~4 significant decimal digits
  - FP32 accumulation prevents catastrophic cancellation

## Key Insights

1. **Row-major is free**: The transpose trick costs nothing—just swap A and B arguments. No data copying, no transposition overhead.

2. **Tensor Core threshold**: For small matrices (N < 64), Tensor Cores may not help due to overhead. cuBLAS automatically chooses the best algorithm.

3. **FP16 vs FP32**: Use FP16 inputs when memory bandwidth is the bottleneck (large matrices). Use FP32 everywhere when numerical accuracy is critical.

4. **Handle reuse**: Create the cuBLAS handle once and reuse it across many GEMM calls. Handle creation has non-trivial overhead (~100s of microseconds).

## Common Mistakes

- **Wrong dimension order**: cuBLAS expects (m, n, k) for C = A × B where A is m×k and B is k×n. With the row-major trick, you pass (N, M, K) not (M, N, K).

- **Missing Tensor Core mode**: Without `cublasSetMathMode(handle, CUBLAS_TENSOR_OP_MATH)`, cuBLAS may use FP32 CUDA cores instead of Tensor Cores.

- **Wrong data types**: Using `CUDA_R_32F` for everything defeats the purpose of mixed precision. Use `CUDA_R_16F` for storage, `CUBLAS_COMPUTE_32F` for accumulation.

- **Not destroying handle**: `cublasDestroy(handle)` must be called to avoid memory leaks. In long-running servers, handle leaks accumulate.

- **Wrong matrix layout in memory**: Row-major A[i][j] is stored at A[i*N + j]. Column-major A[i][j] is stored at A[j*M + i]. Be consistent!

- **Assuming exact FP32 results**: FP16 accumulation introduces rounding error. Accept relative error <= 1e-2, not exact match.
