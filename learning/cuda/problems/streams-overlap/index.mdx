## Background

In a naive CUDA program, operations execute sequentially:

```
1. Copy input data from Host to Device (H→D)
2. Run kernel on GPU
3. Copy output data from Device to Host (D→H)
```

Each stage must complete before the next begins. However, modern GPUs have **separate hardware engines** for:
- Kernel execution (the SMs running your kernel)
- H2D DMA (copying data to GPU)
- D2H DMA (copying data back to CPU)

These engines can operate **simultaneously**. CUDA **streams** let you schedule work so that, for example, while kernel A is running, the DMA engine can already be copying data for kernel B.

This problem teaches you to build a **pipeline**: split your data into chunks, then interleave H→D, compute, and D→H across multiple streams to achieve maximum overlap.

## What to Implement

You are given a skeleton that processes a large array by applying a simple transformation (`output[i] = input[i] * 2 + threadIdx.x`). Your task is to:

1. **Create multiple CUDA streams** (NUM_STREAMS = 4)
2. **Split the data into chunks** — each stream processes one chunk
3. **Issue async operations** in each stream:
   - `cudaMemcpyAsync` for H→D transfer
   - Kernel launch with stream parameter
   - `cudaMemcpyAsync` for D→H transfer
4. **Use pinned memory** for async transfers (allocated with `cudaMallocHost`)
5. **Measure timing** using CUDA events to verify overlap

### Key API Usage

Create streams:
```cpp
cudaStream_t streams[NUM_STREAMS];
for (int i = 0; i < NUM_STREAMS; i++) {
    cudaStreamCreate(&streams[i]);
}
```

Allocate pinned memory:
```cpp
float* h_pinned;
cudaMallocHost(&h_pinned, bytes);  // Page-locked, required for async memcpy
```

Launch kernel in a stream:
```cpp
myKernel<<<grid, block, 0, streams[i]>>>(args...);
```

Async memory copy:
```cpp
cudaMemcpyAsync(d_ptr, h_ptr, bytes, cudaMemcpyHostToDevice, streams[i]);
```

Record timing events:
```cpp
cudaEvent_t start, stop;
cudaEventCreate(&start);
cudaEventCreate(&stop);

cudaEventRecord(start, streams[0]);
// ... do work ...
cudaEventRecord(stop, streams[0]);
cudaEventSynchronize(stop);

float ms;
cudaEventElapsedTime(&ms, start, stop);
```

### Constraints

- `1 ≤ n ≤ 50,000,000` (large enough to see timing differences)
- `NUM_STREAMS = 4` (fixed)
- Use 256 threads per block
- All async transfers require pinned memory
- Chunk size = `n / NUM_STREAMS` (assume n is divisible)
- Do not modify the test harness

### Expected Behaviour

With proper pipelining:
- **Single stream**: Total time ≈ H→D + Kernel + D→H (sequential)
- **Four streams**: Total time should be significantly less due to overlap
- Target: ~3x speedup vs sequential for large n (N=10M+)

__global__ void transformKernel(const float* input, float* output, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        output[i] = input[i] * 2 + threadIdx.x;
    }
}

void solve_pipelined(const float* h_input, float* h_output, int n) {
    const int blockSize = 256;
    int chunkSize = n / NUM_STREAMS;
    int gridChunk = (chunkSize + blockSize - 1) / blockSize;

    // Create streams
    cudaStream_t streams[NUM_STREAMS];
    for (int i = 0; i < NUM_STREAMS; i++) {
        cudaStreamCreate(&streams[i]);
    }

    // Allocate device memory for all chunks
    float *d_input[NUM_STREAMS], *d_output[NUM_STREAMS];
    for (int i = 0; i < NUM_STREAMS; i++) {
        cudaMalloc(&d_input[i], chunkSize * sizeof(float));
        cudaMalloc(&d_output[i], chunkSize * sizeof(float));
    }

    // Allocate pinned host memory for async transfers
    float* h_pinned_input[NUM_STREAMS];
    float* h_pinned_output[NUM_STREAMS];
    for (int i = 0; i < NUM_STREAMS; i++) {
        cudaMallocHost(&h_pinned_input[i], chunkSize * sizeof(float));
        cudaMallocHost(&h_pinned_output[i], chunkSize * sizeof(float));
    }

    // Copy input data to pinned buffers
    for (int i = 0; i < NUM_STREAMS; i++) {
        for (int j = 0; j < chunkSize; j++) {
            h_pinned_input[i][j] = h_input[i * chunkSize + j];
        }
    }

    // Pipeline: issue all work in parallel across streams
    for (int i = 0; i < NUM_STREAMS; i++) {
        int offset = i * chunkSize;

        // H2D async transfer
        cudaMemcpyAsync(d_input[i], h_pinned_input[i],
                        chunkSize * sizeof(float),
                        cudaMemcpyHostToDevice, streams[i]);

        // Kernel launch in stream
        transformKernel<<<gridChunk, blockSize, 0, streams[i]>>>(
            d_input[i], d_output[i], chunkSize);

        // D2H async transfer
        cudaMemcpyAsync(h_pinned_output[i], d_output[i],
                        chunkSize * sizeof(float),
                        cudaMemcpyDeviceToHost, streams[i]);
    }

    // Wait for all streams to complete
    for (int i = 0; i < NUM_STREAMS; i++) {
        cudaStreamSynchronize(streams[i]);
    }

    // Copy results from pinned buffers to output
    for (int i = 0; i < NUM_STREAMS; i++) {
        for (int j = 0; j < chunkSize; j++) {
            h_output[i * chunkSize + j] = h_pinned_output[i][j];
        }
    }

    // Cleanup
    for (int i = 0; i < NUM_STREAMS; i++) {
        cudaFree(d_input[i]);
        cudaFree(d_output[i]);
        cudaFreeHost(h_pinned_input[i]);
        cudaFreeHost(h_pinned_output[i]);
        cudaStreamDestroy(streams[i]);
    }
}

## Approach

The key insight is **pipelining**: instead of processing one chunk at a time, we split the data into NUM_STREAMS chunks and issue work for all chunks simultaneously. The GPU's hardware can then overlap:

- DMA engine copying chunk 1 (H→D)
- SM executing kernel on chunk 0
- DMA engine copying chunk 2 (D→H)

This is the same principle as CPU instruction pipelining: keep all execution units busy by feeding them work in advance.

### Pipeline Timeline

```
Time →
Stream 0: [H2D_0][Kernel_0][D2H_0]
Stream 1:         [H2D_1][Kernel_1][D2H_1]
Stream 2:                 [H2D_2][Kernel_2][D2H_2]
Stream 3:                         [H2D_3][Kernel_3][D2H_3]
```

### Key Requirements

1. **Pinned memory**: `cudaMemcpyAsync` requires page-locked host memory because the DMA engine needs physically contiguous memory that won't be swapped out during transfer. Use `cudaMallocHost()` instead of `malloc()`.

2. **Explicit streams**: The default stream (stream 0) synchronizes with all other streams. Using user-created streams allows true concurrent execution.

3. **Kernel stream parameter**: `kernel<<<grid, block, 0, stream>>>` — the fourth parameter specifies which stream the kernel belongs to.

4. **Synchronization**: After issuing all work, call `cudaStreamSynchronize()` or `cudaDeviceSynchronize()` to wait for completion before accessing results.

### Why Not Infinite Streams?

You might wonder: why not use 100 streams instead of 4? There are limits:

- **Hardware concurrency**: GPUs have a finite number of work queues (HyperQ allows ~32 concurrent operations on modern GPUs)
- **PCIe bandwidth**: All H2D transfers share the same bus; more streams don't increase bandwidth
- **Memory pressure**: Each chunk needs device memory; too many chunks may exceed GPU memory
- **Diminishing returns**: Beyond 4-8 streams, additional streams add overhead without improving overlap

### Overhead Sources

Stream-based pipelining introduces several overhead sources:

1. **Stream creation/destruction**: ~1-5 µs per stream
2. **Async API calls**: ~0.1-1 µs per `cudaMemcpyAsync` or kernel launch
3. **Synchronization**: ~5-10 µs for `cudaStreamSynchronize()`
4. **Pinned memory allocation**: ~10-50 µs per `cudaMallocHost()`
5. **Staging copies**: CPU copies to/from pinned buffers add ~N × 0.01 µs

For small N, these overheads dominate:
- N = 1K: Overhead ≈ 100 µs, actual work ≈ 5 µs (20x overhead)
- N = 1M: Overhead ≈ 100 µs, actual work ≈ 500 µs (5x overhead reduced)
- N = 10M: Overhead ≈ 100 µs, actual work ≈ 5000 µs (2% overhead)

### When to Use Streams

| Scenario | Recommended Streams | Expected Speedup |
|------|--|--|
| Small batch (N < 1M) | 1 (sequential) | N/A |
| Medium batch (N = 1-10M) | 4 | 1.2-2x |
| Large batch (N > 10M) | 4-8 | 2-4x |
| Streaming (continuous) | 4-8 + CUDA Graphs | 2-3x steady state |
| Multi-GPU | NUM_STREAMS × num_gpus | Near-linear scaling |

For production code, consider:
- **Dynamic stream count**: Adjust based on data size
- **CUDA Graphs**: Reduce overhead for repeated workloads
- **Unified Memory**: Simplifies staging at slight performance cost

## Complexity

- **Time**: O(n/P) where P is the effective parallelism from stream overlap
  - Sequential: T_seq ≈ T_H2D + T_kernel + T_D2H
  - Pipelined: T_pipe ≈ T_H2D + T_kernel + T_D2H + T_overhead
    - With ideal overlap: T_pipe ≈ max(T_H2D, T_kernel, T_D2H) + T_overhead
- **Memory**: O(n) — same as sequential, but with extra pinned host buffers (2× for input and output staging)
- **Stream overhead**: O(NUM_STREAMS) — creating/destroying streams and synchronization

### Overhead vs Benefit Analysis

Stream pipelining introduces overhead that may not be worth it for small workloads:

| Data Size | Sequential Time | Pipelined Time | Speedup | Verdict |
|------|--|--|--|-|
| N = 1K | ~0.1 ms | ~0.15 ms | 0.67x | **Negative** — overhead dominates |
| N = 1M | ~5 ms | ~4 ms | 1.25x | **Marginal** — slight benefit |
| N = 10M | ~50 ms | ~20 ms | 2.5x | **Good** — clear overlap |
| N = 50M | ~250 ms | ~80 ms | 3.1x | **Excellent** — near-ideal overlap |

**When streams help:**
- Large data sizes (N > 5M for typical PCIe bandwidth)
- Compute-bound kernels (T_kernel dominates)
- Multiple independent workloads (batch processing)
- Streaming workloads (continuous data feed)

**When streams don't help:**
- Small data sizes (N < 1M)
- Latency-bound operations (quick kernels)
- One-off computations (setup cost > benefit)
- Memory-bound kernels with poor coalescing

The following metrics indicate successful stream overlap:

1. **PCIe Utilization**: Should be high (>80%) during H2D and D2H phases
2. **SM Occupancy**: Should remain high during kernel phases

Expected speedup formula:

For typical H100 + PCIe 4.0:

For N = 10M:
- T_H2D ≈ 2500 µs = 2.5 ms
- T_D2H ≈ 2500 µs = 2.5 ms
- T_kernel ≈ 10-20 ms (depends on computation)

- Speedup ≈ 1.5-2x

## Key Insights

1. **Async transfers need pinned memory**: `cudaMallocHost()` creates page-locked memory that the DMA engine can access directly.

2. **Streams enable overlap**: Operations in different streams can run concurrently if hardware resources permit.

3. **Timing reveals overlap**: Use CUDA events (`cudaEventRecord`, `cudaEventElapsedTime`) to measure actual speedup.

4. **Nsight Systems for debugging**: Run `nsys profile --stats=true ./program` to see the timeline and verify overlap visually.

## Common Mistakes

- **Using malloc() instead of cudaMallocHost()**: Async memcpy will fail or serialize with synchronous behavior.
- **Forgetting stream parameter**: `cudaMemcpyAsync` without a stream goes to the default stream, which blocks everything.
- **Not synchronizing**: Results won't be ready if you don't call `cudaStreamSynchronize()` before reading.
- **Too many streams**: 16+ streams often don't improve performance due to hardware limits.
- **Chunk size too small**: If chunks are too small, the overhead of stream management exceeds the benefit.
