## Background

The softmax function is a cornerstone operation in deep learning, particularly in Transformer models where it computes attention probabilities. Given an input vector x = (x_1, x_2, ..., x_C), softmax computes:

    softmax(x)_i = exp(x_i) / sum(exp(x_j) for j = 1 to C)

While mathematically simple, implementing softmax efficiently on GPUs reveals subtle challenges that distinguish novice from expert CUDA programmers.

### Mathematical Formulation and Numerical Stability

A naive implementation of softmax faces a critical numerical stability problem: if any x_i is large (e.g., x_i = 1000), then exp(x_i) overflows to infinity. The standard remedy is the log-sum-exp trick: subtract the maximum value before exponentiating:

    maxval = max(x_1, x_2, ..., x_C)
    softmax(x)_i = exp(x_i - maxval) / sum(exp(x_j - maxval))

This transformation is mathematically equivalent (since exp(a - max) / sum(exp(b - max)) = exp(a) / sum(exp(b))), but ensures all exponents are non-positive, preventing overflow.

### The Three-Pass Naive Algorithm

A straightforward GPU implementation requires three passes over the data:

    Pass 1: Reduce to find maxval = max(x_i)
    Pass 2: Compute exp(x_i - maxval) and store intermediate results
    Pass 3: Reduce to find sumval = sum(exp(x_i - maxval)), then divide

This three-pass approach has a critical flaw: it reads the input data three times from global memory (HBM). For a row of length C, this means 3 × C × 4 bytes of memory traffic (assuming FP32). In memory-bound kernels like softmax, reducing memory traffic is the primary optimization lever.

### Why Softmax is Challenging

Softmax couples all elements in a row through the global sum, preventing simple parallelization. Unlike elementwise operations where each thread works independently, softmax requires:

1. A global reduction to find the maximum
2. A global reduction to find the sum of exponentials
3. A final broadcast to divide each element by the sum

The coupling means you cannot compute softmax(x_i) without knowing information about all other x_j values. This data dependency forces careful synchronization and reduction patterns.

## What to Implement

You will implement an optimized fused softmax kernel that uses the **online softmax** algorithm to compute the result in a single pass over the data.

### The Online Softmax Algorithm

The key insight, first described by Milakov & Gimelshein (2018), is that you can maintain running statistics (m, d) where:
- m = running maximum
- d = running normalized sum

When processing a new element x_j:
```
m_new = max(m, x_j)
d_new = d * exp(m - m_new) + exp(x_j - m_new)
```

The correction factor `exp(m - m_new)` rescales the previous sum when a new maximum is found. This works because:
- If x_j <= m: m stays the same, d simply accumulates exp(x_j - m)
- If x_j > m: all previous terms must be rescaled by exp(m - x_j), then add exp(0) = 1 for x_j itself

### Implementation Strategy

Your kernel should:

1. **Assign one warp per row**: Each warp (32 threads) processes one complete row of C elements. This is efficient because warps naturally support parallel reduction via shuffle instructions.

2. **Use thread coarsening**: Each thread in the warp processes C/32 elements in a strided pattern: thread t handles indices t, t+32, t+64, ...

3. **Maintain running (m, d) statistics**: Each thread maintains its own partial (m, d) pair while scanning its assigned elements.

4. **Reduce within the warp**: Use `__shfl_down_sync(0xFFFFFFFF, ...)` to combine partial results across all 32 threads in the warp.

5. **Broadcast final result**: Use `__shfl_sync(0xFFFFFFFF, ...)` to broadcast m and d to all threads, then compute the final softmax output.

### Online Softmax Reduction Pattern

The reduction combines two (m, d) pairs using:
```
combine((m1, d1), (m2, d2)):
  m_new = max(m1, m2)
  d_new = d1 * exp(m1 - m_new) + d2 * exp(m2 - m_new)
  return (m_new, d_new)
```

This associative operation allows parallel reduction across threads.

## Constraints

- **Input format**: Matrix of shape (N, C) stored in row-major order
- **N**: Number of rows (batch size × sequence length), 1 <= N <= 65536
- **C**: Number of columns (feature dimension), 32 <= C <= 65536
- **Compute capability**: Target sm_120 (Hopper) but must work on sm_80+ (Ampere)
- **Data type**: FP32 (float) throughout
- **Memory**: Do not use additional global memory beyond input and output arrays
- **Shared memory**: Optional; the warp-based approach does not require it
- **Block size**: Must support any block size that is a multiple of 32

### Rules

- Do not modify the `solve()` function signature or the harness main() function
- Implement only the `softmax_forward_kernel` and any helper device functions
- The harness will call your kernel with appropriate grid/block dimensions
- Ensure numerical accuracy: output must match reference CPU implementation within 1e-4 absolute error

## Example

For input row [1.0, 2.0, 3.0]:
- maxval = 3.0
- exp([1-3, 2-3, 3-3]) = exp([-2, -1, 0]) = [0.135, 0.368, 1.0]
- sum = 1.503
- softmax = [0.090, 0.245, 0.666]

The online algorithm produces the same result by maintaining (m, d) across elements:
- Process 1.0: m=1.0, d=1.0
- Process 2.0: m=2.0, d=1*exp(1-2) + 1 = 0.368 + 1 = 1.368
- Process 3.0: m=3.0, d=1.368*exp(2-3) + 1 = 0.503 + 1 = 1.503
- Final: exp([1-3, 2-3, 3-3]) / 1.503 = [0.090, 0.245, 0.666]

#include <cuda_runtime.h>
#include <math_constants.h>
#include <cstdio>

// Helper: ceiling division
__device__ __forceinline__ int ceil_div(int a, int b) {
    return (a + b - 1) / b;
}

// Warp-level reduction for combining (max, sum) pairs
// Uses the online softmax update rule
__device__ __forceinline__ float warpReduceMax(float val) {
    for (int offset = 16; offset > 0; offset >>= 1) {
        val = fmaxf(val, __shfl_down_sync(0xFFFFFFFF, val, offset));
    }
    return val;
}

__device__ __forceinline__ float warpReduceSum(float val) {
    for (int offset = 16; offset > 0; offset >>= 1) {
        val += __shfl_down_sync(0xFFFFFFFF, val, offset);
    }
    return val;
}

// Kernel: Fused online softmax
__global__ void softmax_forward_kernel(float* output, const float* input, int N, int C) {
    int tid = threadIdx.x;
    int warpId = threadIdx.x / 32;
    int laneId = threadIdx.x % 32;
    int warpsPerBlock = blockDim.x / 32;

    // Each warp processes one row; calculate global row index
    int row = blockIdx.x * warpsPerBlock + warpId;

    if (row >= N) {
        return;
    }

    const float* x = input + row * C;
    float* y = output + row * C;

    // Online softmax: maintain running (max, sum) statistics
    float m = -INFINITY;  // running maximum
    float d = 0.0f;       // running sum of exp(x - m)

    // Thread coarsening: each thread processes C/warpSize elements
    for (int i = laneId; i < C; i += 32) {
        float x_i = x[i];
        float m_new = fmaxf(m, x_i);
        d = d * expf(m - m_new) + expf(x_i - m_new);
        m = m_new;
    }

    // Reduce (m, d) across all threads in the warp
    // First, reduce m to find the global maximum
    float global_m = warpReduceMax(m);

    // Then rescale d to account for the new maximum
    float rescaled_d = d * expf(m - global_m);
    float global_d = warpReduceSum(rescaled_d);

    // Broadcast final (m, d) to all threads in the warp
    float final_m = __shfl_sync(0xFFFFFFFF, global_m, 0);
    float final_d = __shfl_sync(0xFFFFFFFF, global_d, 0);

    // Compute softmax output: exp(x_i - m) / d
    for (int i = laneId; i < C; i += 32) {
        y[i] = expf(x[i] - final_m) / final_d;
    }
}

// Host function: called by harness to launch the kernel
void solve(float* d_output, const float* d_input, int N, int C) {
    int block_size = 128;  // 4 warps per block
    int warpsPerBlock = block_size / 32;
    int grid_size = ceil_div(N, warpsPerBlock);

    softmax_forward_kernel<<<grid_size, block_size>>>(d_output, d_input, N, C);

    cudaError_t err = cudaGetLastError();
    if (err != cudaSuccess) {
        printf("Kernel launch failed: %s\n", cudaGetErrorString(err));
    }

    cudaDeviceSynchronize();
}

// Reference CPU implementation of softmax
void softmax_reference(float* output, const float* input, int N, int C) {
    for (int i = 0; i < N; i++) {
        const float* row = input + i * C;
        float* out_row = output + i * C;

        // Find max for numerical stability
        float maxval = -INFINITY;
        for (int j = 0; j < C; j++) {
            maxval = fmaxf(maxval, row[j]);
        }

        // Compute exp(x - max) and sum
        double sum = 0.0;
        for (int j = 0; j < C; j++) {
            out_row[j] = expf(row[j] - maxval);
            sum += out_row[j];
        }

        // Normalize
        float norm = 1.0f / (float)sum;
        for (int j = 0; j < C; j++) {
            out_row[j] *= norm;
        }
    }
}

        float sum = 0.0f;

        }

## Approach Explanation

The fused online softmax problem teaches three key CUDA optimization patterns:

### 1. Online Algorithms for Single-Pass Computation

Traditional softmax requires three passes: (1) find max, (2) compute exp and sum, (3) divide. The online algorithm maintains running statistics (m, d) where:

- m = running maximum
- d = running sum with proper rescaling

When a new element x_j exceeds the current maximum m, we must rescale all previous accumulated values. The insight is that this rescaling is simply multiplication by exp(m_old - m_new).

### 2. Warp-Level Parallelism with Shuffle Instructions

Using one warp per row (32 threads) is ideal for softmax because:
- Warps have a natural leader (lane 0) for broadcasting results
- `__shfl_down_sync` enables communication without shared memory
- Each thread handles C/32 elements, providing good parallelism for large C

### 3. Memory-Bound Optimization

Softmax is memory-bound for all practical sequence lengths. The FLOP count is O(C) but memory traffic dominates:
- Naive: 3 passes (3 reads, 1 write) = 4 × C × 4 bytes
- Online: 2 passes (1 read, 1 write) = 2 × C × 4 bytes

This 2× reduction in memory traffic directly translates to 2× speedup on modern GPUs where bandwidth is the bottleneck.

## Complexity Analysis

**Time Complexity**: O(N × C / 32) per block, O(N × C) total work across all warps
- Each element is visited once by one thread
- Warp reductions add O(log(32)) = O(1) overhead per row

**Space Complexity**: O(1) additional memory
- No shared memory or global memory allocations beyond I/O
- Only register-local (m, d) variables needed

**Memory Bandwidth**: 2 × N × C × 4 bytes (1 read, 1 write)
- Compared to 4 × N × C × 4 bytes for naive 3-pass algorithm
- 50% reduction in memory traffic

## Key Insights

1. **Online softmax is numerically exact**: The rescaling maintains mathematical equivalence with the standard 2-pass algorithm, within floating-point precision.

2. **Warp shuffles beat shared memory**: For small reductions (32 threads), `__shfl_down_sync` has lower latency than shared memory with barriers.

3. **One warp per row is optimal when C >= 32**: Each thread processes at least one element, and all 32 threads stay busy. For very large C (e.g., 4096+), each thread processes 128+ elements, maximizing compute density.

4. **The correction factor is the key**: `d * exp(m_old - m_new)` elegantly rescales all previous contributions in O(1) time.

## Common Mistakes

1. **Using separate max and sum reductions**: Computing max first, then sum separately defeats the purpose of online softmax. The running (m, d) pair must be maintained simultaneously.

2. **Incorrect rescaling order**: When combining (m1, d1) and (m2, d2), you must rescale d1 by exp(m1 - max(m1,m2)) before adding.

3. **Not synchronizing warp lanes**: After the reduction, lane 0 holds the final result. All other lanes must read via `__shfl_sync`, not just read shared memory without proper synchronization.

4. **Ignoring edge cases**: When C is not divisible by 32, threads with indices >= C must not write invalid data. Use boundary checks or structure loops to avoid out-of-bounds access.

5. **Using INFINITY instead of -INFINITY**: The initial max must be -INFINITY, otherwise the first comparison fails and you get incorrect results.

## Further Optimization Opportunities

For production use, consider:

- **Multi-warp per row**: For very large C, split the row across multiple warps with shared memory for inter-warp reduction
- **Vectorized loads**: Use float4 or float8 loads to reduce load instructions
- **PTX instructions**: Use `.cs` (streaming) load hints to bypass L1 cache for read-once data
- **Pipelining**: Overlap computation with memory fetches using async copy on Hopper

The online softmax algorithm is fundamental to FlashAttention, which extends these same principles (maintaining running (m, l) statistics) to compute attention without materializing the N×N attention matrix.
