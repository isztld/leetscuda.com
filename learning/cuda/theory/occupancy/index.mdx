# Occupancy & Latency Hiding

## Why this matters for interviews

Occupancy questions separate CUDA practitioners who understand the execution model from those who just write kernels that happen to work. At NVIDIA, Meta, Google, and any company optimizing GPU kernels, expect "what occupancy do you target?", "when is high occupancy bad?", and "how do you calculate the occupancy of a kernel?" These are not academic questions — occupancy directly controls whether your kernel achieves peak throughput or stalls waiting for memory. This article gives you the precise mental model to answer them.

## What is Occupancy?

**Occupancy** is the ratio of active warps per SM to the SM's maximum supported warps. On H100 (and all modern NVIDIA GPUs), the maximum is **64 warps per SM** (2048 threads).

**Key insight:** Occupancy is a *proxy* for latency hiding potential. Higher occupancy means more warps available to switch to when one stalls on a memory access. But occupancy is not a goal in itself — it's a means to keep the execution units fed.

**The three resource constraints:**
Occupancy is determined by the *minimum* of three independent limits:

1. **Warp slots:** Maximum 64 warps/SM
2. **Register file:** 65,536 registers/SM on H100/A100
3. **Shared memory:** 228 KB/SM on H100, 164 KB on A100

**Example calculation for H100:**
Consider a kernel with:
- 32 threads/block (1 warp/block)
- 8 registers/thread
- 12 KB shared memory/block

```
Warp limit: 64 warps/SM → 64 blocks
Register limit: 65536 / (32 × 8) = 256 blocks
Shared memory limit: 228 KB / 12 KB = 19 blocks
```

The shared memory is the bottleneck: **19 blocks per SM** = 19 warps = **29.7% occupancy**.

Even though the register file could support 256 blocks, the shared memory limits us to 19. This is the common case: register pressure is usually low compared to shared memory requirements for working sets.

## The Purpose of Occupancy: Latency Hiding

The primary purpose of occupancy is **latency hiding**. When a warp stalls waiting for a memory load (400 cycles for L2 miss), the warp scheduler immediately switches to another eligible warp. If you have enough warps resident, the SM never idles.

**Little's Law determines how much concurrency you need:**
```
concurrency = latency × throughput
```

For a 400-cycle memory latency at 1 issue per cycle, you need 400 concurrent operations. Since warps are the unit of scheduling:
```
warps needed = 400 cycles / 32 threads/warp = 13 warps
```

So **13 warps** are sufficient to hide one memory access per warp. In practice, you need more because warps may have multiple outstanding memory requests.

**The occupancy sweet spot:** For most memory-bound kernels, 30-50% occupancy (20-32 warps/SM) is sufficient to hide latency. Beyond 50%, marginal returns diminish rapidly. For compute-bound kernels with high instruction-level parallelism (ILP), even lower occupancy can be fine.

## The Occupancy Trap: Why 100% Is Not Always Better

**High occupancy can hurt performance.** Here's why:

1. **Resource contention:** Higher occupancy means fewer resources per warp. If you push occupancy to 100% by reducing registers per thread, you may trigger register spilling, which is catastrophic.

2. **Diminishing returns:** Once you have enough warps to hide latency, adding more warps provides no benefit. The SM is already issuing every cycle.

3. **Compute-bound kernels don't need high occupancy:** A well-tuned GEMM kernel on Hopper often runs at **single-digit occupancy** (e.g., 4 warps/SM = 6.25%) because it doesn't need many warps to saturate Tensor Cores. The goal is compute unit utilization, not warp count.

**Practical guidance:**
- **Memory-bound kernels:** Target 30-50% occupancy. Enough to hide latency, not so much that you waste resources.
- **Compute-bound kernels:** Occupancy doesn't matter. Focus on arithmetic intensity and Tensor Core utilization.
- **Never sacrifice correctness for occupancy:** Don't reduce block size just to increase occupancy if it hurts algorithmic efficiency.

## The Occupancy API

CUDA provides `cudaOccupancyMaxActiveBlocksPerMultiprocessor` to calculate theoretical occupancy:

```cpp
#include <cuda_runtime.h>

int maxBlocks;
cudaError_t err = cudaOccupancyMaxActiveBlocksPerMultiprocessor(
    &maxBlocks,
    my_kernel,
    blockDim,      // threads per block
    dynamicSharedMemSize  // 0 if no dynamic shared memory
);

if (err != cudaSuccess) {
    // Handle error
}

int maxWarps = maxBlocks * (blockDim / 32);
float occupancy = maxWarps / 64.0f;  // 64 is H100 max warps/SM
printf("Theoretical occupancy: %.1f%%\n", occupancy * 100);
```

**Limitations:** This calculates *theoretical* occupancy based on resource constraints. It does not predict *achieved* occupancy, which depends on actual runtime behavior (e.g., how many blocks actually get scheduled, whether warps stall early).

**Auto-tuning with `cudaOccupancyMaxPotentialBlockSize`:**
```cpp
int minGridSize, blockSize;
cudaOccupancyMaxPotentialBlockSize(&minGridSize, &blockSize, my_kernel, 0, 0);
```

This returns a block size that typically maximizes occupancy for the given kernel. It's a good starting point for tuning.

## Measuring Achieved Occupancy

Nsight Compute provides the **Achieved Occupancy** metric:
- **Theoretical Occupancy:** Maximum possible based on resource constraints
- **Achieved Occupancy:** Actual average during execution

**Common gap:** Achieved occupancy is often lower than theoretical. Reasons:
- Kernel doesn't use all resources (early return, small data set)
- Blocks finish at different times, leaving SM idle
- Scheduling decisions by the runtime

**Querying metrics:**
```bash
nsight compute --metrics smsp__warps_active.avg.per_cycle,sm__warps_active.avg.per_cycle ./your_kernel
```

Or in the Nsight Compute UI, look for "Occupancy" under the "SmMetrics" section.

## Occupancy vs Other Performance Metrics

Occupancy must be interpreted in context:

**With Issue Efficiency:**
- Low occupancy + low issue efficiency = latency not hidden, need more warps
- High occupancy + low issue efficiency = warps present but not eligible (e.g., all waiting on same dependency)

**With Pipe Utilization:**
- Low occupancy + low compute utilization = likely memory-bound, latency hiding broken
- High occupancy + low compute utilization = kernel is compute-bound but underutilizing units (not an occupancy problem)

**With Branch Efficiency:**
- Low branch efficiency + low occupancy = divergence is the bottleneck, not occupancy
- High branch efficiency + low occupancy = genuine latency hiding problem

**Rule of thumb:** If issue efficiency is >90% and pipe utilization is high, occupancy is fine. Don't optimize it further.

## Common Misconceptions

**❌ Wrong:** 100% occupancy always means best performance.
**✓ Correct:** Many high-performance kernels run at single-digit occupancy. GEMM kernels on Hopper often achieve peak throughput at 4-8 warps/SM (6-12% occupancy). The goal is resource utilization (compute or memory), not maximizing warp count.

**❌ Wrong:** Occupancy and performance are linearly correlated.
**✓ Correct:** The relationship is sublinear with diminishing returns. Going from 10% to 20% occupancy may double performance if latency was the bottleneck. Going from 50% to 60% has negligible effect. Beyond a certain point, more occupancy can hurt performance by starving individual warps of resources.

**❌ Wrong:** Low occupancy always hurts performance.
**✓ Correct:** For compute-bound kernels with high ILP (instruction-level parallelism), low occupancy is fine. A single warp with many independent instructions can keep execution units busy without needing many concurrent warps. Vasily Volkov's thesis demonstrated that high-ILP kernels achieve similar latency hiding with fewer warps.

**❌ Wrong:** The occupancy API tells me if my kernel is optimal.
**✓ Correct:** The API calculates *theoretical* occupancy based on static resource analysis. It does not account for actual scheduling behavior, divergence patterns, or memory access patterns. Always verify with Nsight Compute's *achieved* occupancy and other metrics.

**❌ Wrong:** I should always maximize occupancy by reducing registers or shared memory.
**✓ Correct:** Reducing registers may trigger spilling (catastrophic). Reducing shared memory may increase memory traffic. The goal is the *minimum occupancy that hides latency*, not maximum occupancy. Tune for overall throughput, not occupancy in isolation.

## When to Worry About Occupancy

**Worry about occupancy when:**
- Issue efficiency is below 80%
- Pipe utilization is low (<70%)
- Nsight Compute shows many warps in "Stalled" state with no eligible warps to switch to
- Theoretical occupancy is below 25% and kernel is memory-bound

**Don't worry about occupancy when:**
- Issue efficiency is above 90%
- Pipe utilization is high
- Kernel is compute-bound (high arithmetic intensity)
- The kernel uses Tensor Cores or other specialized units that require few warps to saturate

## Key Numbers to Memorize

| Property | H100 | A100 |
|-----|-|---|
| Max warps/SM | 64 | 64 |
| Max threads/SM | 2048 | 2048 |
| Register file/SM | 65536 regs (256 KB) | 65536 regs (256 KB) |
| Shared memory/SM (max) | 228 KB | 164 KB |
| Memory latency (L2 miss) | ~400 cycles | ~400 cycles |
| Warps needed to hide 400 cycles | ~13 warps | ~13 warps |

**Practical occupancy targets:**
- **Memory-bound kernels:** 30-50% (20-32 warps/SM)
- **Compute-bound kernels:** 10-25% (6-16 warps/SM)
- **Tensor Core GEMM:** 6-12% (4-8 warps/SM)
- **Saturation point:** ~50% beyond this, diminishing returns

## Interview Patterns

**"What occupancy do you target for a CUDA kernel?"**
It depends on the bottleneck. For memory-bound kernels, I target 30-50% occupancy (20-32 warps/SM), which is typically enough to hide L2 miss latency (~400 cycles). For compute-bound kernels, occupancy doesn't matter — I focus on arithmetic intensity and Tensor Core utilization. Many high-performance GEMM kernels run at single-digit occupancy (6-12%) because they don't need many warps to saturate Tensor Cores. I never optimize for occupancy in isolation — I look at issue efficiency and pipe utilization first.

**"How do you calculate the occupancy of a kernel?"**
Use the CUDA Occupancy API: `cudaOccupancyMaxActiveBlocksPerMultiprocessor` returns the maximum blocks per SM given the kernel's register and shared memory usage. Then: `occupancy = (blocks × warps_per_block) / 64`. But this is theoretical — achieved occupancy requires profiling with Nsight Compute. The three constraints are: warp slots (64), registers (65536), and shared memory (228 KB on H100). The minimum of these limits occupancy.

**"When is high occupancy bad for performance?"**
When it forces resource contention that hurts per-warp performance. Examples: (1) Reducing registers to increase occupancy triggers spilling to L1 — catastrophic slowdown. (2) Reducing shared memory forces more global memory traffic. (3) Pushing occupancy beyond 50% provides diminishing returns while complicating the kernel. For compute-bound kernels, high occupancy is irrelevant — Tensor Core throughput dominates.

**"Explain Little's Law in the context of GPU latency hiding."**
Little's Law states: `concurrency = latency × throughput`. For a 400-cycle memory latency at 1 instruction per cycle throughput, you need 400 concurrent operations. Since warps are the scheduling unit, that's 400 / 32 = ~13 warps. This is the minimum to hide one memory access per warp. In practice, kernels need more (20-32 warps) to handle multiple outstanding requests. But once you have enough, adding more warps provides no benefit — the SM is already issuing every cycle.
