# Compute Capability & GPU Architecture Generations

## Why this matters for interviews

When you're optimizing a kernel, you need to know which GPU it runs on. Features like Tensor Cores, async memory copies, and TMA are architecture-specific. In interviews, you'll be asked: "Can you use WGMMA on an A100?" or "What's new in Hopper?" or "Why does your kernel crash on a V100?"

Understanding compute capability helps you write portable code, choose the right features, and debug architecture-specific issues.

## What is compute capability?

**Compute capability** is NVIDIA's versioning scheme for GPU architectures. It's encoded as a number `major.minor` (e.g., 8.0 for A100, 9.0 for H100).

### Why it matters

- **Feature availability**: New features appear at specific compute capabilities
- **Compilation**: PTX code must target a specific compute capability
- **Runtime checks**: You can query the device and adapt behavior

Example:
```cpp
cudaDeviceProp prop;
cudaGetDeviceProperties(&prop, 0);

int major = prop.major;  // e.g., 9 for H100
int minor = prop.minor;  // e.g., 0

if (major >= 9) {
    // Hopper features available (TMA, WGMMA)
} else {
    // Fall back to Ampere/Volta features
}
```

## Architecture progression: Volta to Blackwell

### Volta (cc 7.0, 2017)

**Products**: V100 (datacenter), GTX 1080 Ti (consumer, Pascal actually, cc6.1)

**Key features**:
- **First Tensor Cores** (Volta TF32) — mixed-precision matrix operations
- **Independent thread scheduling** — each thread has its own program counter (PC)
- **Simultaneous FP32 and INT32** — separate pipelines for floating-point and integer

**Impact on programming**:
- Before Volta, threads in a warp were tightly coupled (SIMD). Volta introduced true SIMT with independent PCs.
- This enabled more flexible control flow but introduced subtle divergence behavior.

### Turing (cc 7.5, 2018)

**Products**: RTX 2080 Ti, T4

**Key features**:
- **2nd-gen Tensor Cores** — support for INT8, INT4
- **RT Cores** — hardware ray tracing (not directly relevant to CUDA compute)
- **Improved FP16** — double FP16 throughput

**Note**: Turing is a bridge architecture; many production servers still use V100 (Volta) or A100 (Ampere).

### Ampere (cc 8.0, 2020)

**Products**: A100 (datacenter), RTX 3090 (consumer, cc8.6)

**Key features**:
- **3rd-gen Tensor Cores** — BF16, TF32, INT8 sparsity
- **Async memory copies** — `cp.async` instruction, decoupling data movement from computation
- **Split arrive/wait barriers** — advanced synchronization for async pipelines
- **HBM2e** — up to 2TB/s memory bandwidth (A100)
- **Multi-instance GPU (MIG)** — partition A100 into 7 isolated GPU instances

**Programming impact**:
```cpp
// Async copy (Ampere+)
// Copy data from global to shared memory without blocking
cp.async.cg.shared.global [%0], [%1], [%2];

// Wait for all async copies to complete
cp.async.commit_group();
cp.async.wait_group(0);  // Wait for all groups
```

### Ada Lovelace (cc 8.9, 2022)

**Products**: RTX 4090, L40

**Key features**:**:
- **4th-gen Tensor Cores** — FP8 support
- **Third-party async copies** — improved async memory pipeline
- **Optimized ray tracing** — consumer-focused features

**Note**: Primarily a consumer GPU; limited datacenter adoption.

### Hopper (cc 9.0, 2022)

**Products**: H100, H200

**Key features**:
- **4th-gen Tensor Cores (WGMMA)** — warpgroup MMA, explicit warp-group operations
- **TMA (Tensor Memory Accelerator)** — hardware async DMA engine
- **Thread Block Clusters** — groups of blocks that can cooperate
- **HBM3** — up to 3.35TB/s (H100), 4.8TB/s (H200)
- **Native FP8** — E4M3 and E5M2 formats
- **Transformer Engine** — specialized FP8 matrix unit

**Programming impact**:
```cpp
// WGMMA (Hopper only, cc9.0+)
// Requires warpgroup (64 threads), not single warp
// Direct MMA without load/store intermediates

// TMA (Hopper only, cc9.0+)
// Asynchronous bulk memory transfers with metadata
// No manual copy loops needed
```

**Warning**: Hopper features **do not work** on Ampere GPUs. TMA, WGMMA, and Thread Block Clusters require cc9.0+.

### Blackwell (cc 10.0, 2024)

**Products**: B200, GB200

**Key features**:
- **5th-gen Tensor Cores** — 2CTA (two cluster thread array), FP4 block-scaled quantization
- **5th-gen NVLink** — 1.8TB/s bidirectional between GPUs
- **HBM3e** — up to 8TB/s
- **Optical interconnect** — next-gen GPU-to-GPU networking

**Programming impact**:
```cpp
// FP4 block-scaled quantization (Blackwell only)
// New quantization format for LLM inference
// Requires B200 or later

// 2CTA UMMA (unified MMA)
// Two thread arrays can cooperate on large matrix ops
```

## Common misconceptions

❌ **Wrong**: "Hopper features work on Ampere GPUs"

✓ **Correct**: TMA, WGMMA, and Thread Block Clusters **require cc9.0+**. They will cause illegal instruction errors on A100 (cc8.0). Always check compute capability at runtime or compile for the target architecture.

❌ **Wrong**: "Upgrading to a newer GPU automatically speeds up all kernels"

✓ **Correct**: New features (TMA, WGMMA, FP8) require **rewriting kernels** to use them. A kernel that doesn't explicitly use TMA will not benefit from it. Old kernels run on new GPUs but don't automatically get speedups from new hardware features.

❌ **Wrong**: "All Ampere GPUs are equal"

✓ **Correct**: Ampere includes multiple variants:
- A100 (cc8.0) — datacenter, HBM2e, MIG support
- RTX 3090 (cc8.6) — consumer, GDDR6X, no MIG
- Jetson AGX Orin (cc8.7) — embedded, lower power

Each has different memory bandwidth, core counts, and feature support.

## Practical implications for CUDA programmers

### Portability

Write portable code that adapts to the available hardware:

```cpp
void launchOptimizedKernel(...) {
    cudaDeviceProp prop;
    cudaGetDeviceProperties(&prop, 0);

    if (prop.major >= 9) {
        // Hopper+ path: use TMA, WGMMA
        launchHopperKernel(...);
    } else if (prop.major >= 8) {
        // Ampere path: use async copies
        launchAmpereKernel(...);
    } else {
        // Fallback: classic CUDA
        launchClassicKernel(...);
    }
}
```

### Compilation targets

Specify compute capability when compiling:

```bash
# Compile for specific architectures
nvcc -arch=sm_80 kernel.cu   # A100
nvcc -arch=sm_90 kernel.cu   # H100

# Compile for multiple architectures (fat binary)
nvcc -gencode arch=compute_80,code=sm_80 \
     -gencode arch=compute_90,code=sm_90 \
     kernel.cu
```

### PTX vs SASS

- **PTX** (Parallel Thread Execution) — virtual ISA, portable across generations
- **SASS** (Shader ASSembly) — actual GPU machine code, architecture-specific

```bash
# Generate PTX (portable)
nvcc -ptx kernel.cu

# Generate SASS (architecture-specific)
nvcc -cubin -arch=sm_80 kernel.cu
```

## Interview patterns

**Question 1**: "Can you use WGMMA on an A100?"

**Answer**: No. WGMMA (warpgroup MMA) is a Hopper feature (cc9.0+). A100 is Ampere (cc8.0) and only supports regular MMA via Tensor Cores.

**Question 2**: "What's the difference between Volta and Ampere?"

**Answer**: Volta (cc7.0) introduced Tensor Cores and independent thread scheduling. Ampere (cc8.0) added async memory copies (`cp.async`), BF16 support, and improved Tensor Cores with sparsity.

**Question 3**: "Why does my kernel crash on a V100 but work on H100?"

**Answer**: You're likely using a Hopper-specific feature (TMA, WGMMA) that doesn't exist on V100 (Volta, cc7.0). Check your code for cc9.0+ features and add runtime or compile-time checks.

**Question 4**: "What compute capability does an A100 have?"

**Answer**: A100 has compute capability 8.0 (Ampere). H100 has 9.0 (Hopper). B200 has 10.0 (Blackwell).

**Question 5**: "How do you write portable CUDA code that works on multiple GPU generations?"

**Answer**: Use compile-time directives (`#if __CUDA_ARCH__ >= 900`) or runtime checks (`cudaGetDeviceProperties` to check `prop.major`) to adapt behavior. Compile for multiple architectures with `-gencode` to create fat binaries.
