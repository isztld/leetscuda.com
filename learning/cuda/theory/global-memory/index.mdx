# Global Memory & Coalescing

## Why this matters for interviews

Global memory coalescing is the first optimization most interviewers probe when they ask about CUDA performance. Questions like "why is your matrix transpose 10× slower than cuBLAS?", "what access pattern should this kernel use?", or "how many memory transactions does this warp issue?" all bottom out here. At principal level, you are expected to reason precisely: not just "use coalesced access" but how many transactions an access pattern generates, why the hardware degrades gracefully with stride-2 but catastrophically with stride-32, and what role L1/L2 caches play in ameliorating poor access. This article gives you that precision.

## Global Memory: What It Is and Where It Lives

Global memory is the primary, large-capacity off-chip storage on the GPU, backed by High Bandwidth Memory (HBM) in current data-center GPUs. Every thread in every block across the entire grid can read and write it. Its lifetime spans the program's execution — data persists across kernel launches until explicitly freed with `cudaFree`.

Two API calls define the basic workflow: `cudaMalloc` allocates device memory and returns a device pointer; `cudaMemcpy` moves data between host and device. These calls involve the CUDA driver and incur nontrivial overhead — `cudaMalloc` can take microseconds to milliseconds. Allocate once, reuse across kernels; never `cudaMalloc` inside a hot path.

The word "global" here has nothing to do with the `__global__` keyword on kernel functions. `__global__` marks a function that runs on the device but is launched from the host. Global *memory* is a storage tier. CUDA architect Nicholas Wilt noted this collision was made "for maximum developer confusion" — expect this to be a trap in whiteboard questions.

## The HBM Tier: Bandwidth and Latency

Modern data-center GPUs attach global memory via HBM stacks integrated directly on the package, achieving memory bandwidths that CPUs cannot match:

| GPU | Memory | Capacity | Peak BW |
|-----|--------|----------|---------|
| A100 SXM | HBM2e | 80 GB | 2.0 TB/s |
| H100 SXM | HBM3 | 80 GB | 3.35 TB/s |
| B200 SXM | HBM3e | 192 GB | 8.0 TB/s |

Despite this bandwidth, global memory access latency is substantial: approximately **400 clock cycles** for an L2 miss that reaches DRAM, and approximately **200 cycles** for an L2 hit. At 1.5 GHz, 400 cycles is ~267 ns — 50× longer than L1 latency on a modern CPU. The GPU's mechanism for tolerating this latency is warp switching, not caching (see the Occupancy & Latency Hiding article). The memory system's job is to maximize *throughput*, not minimize *latency*.

**ECC.** When error-correcting code is enabled (the default in data-center configurations), effective bandwidth decreases by approximately 6% because ECC parity bytes consume a fraction of the HBM transfer capacity.

## Cache Line Mechanics: How DRAM Bursts Work

DRAM does not transfer single bytes or single words. Each access fetches an entire *burst* of consecutive bytes in a single physical transfer. On CUDA GPUs, that burst maps to a **128-byte cache line**. When the SM issues a memory load, the memory controller fetches the 128-byte aligned segment containing the requested address and deposits it in L1 (and L2).

128 bytes divided by 4 bytes per `float` equals **32 floats** — exactly one warp's worth of data if each of the 32 threads requests one float at a consecutive address. This alignment is not an accident: SIMT execution and DRAM burst width were co-designed.

The L1 data cache on Hopper SMs is part of a unified 228 KB pool shared with shared memory (the programmer controls the split). L2 is shared across all SMs: 40 MB on A100, 50 MB on H100. L2 hit latency is ~200 cycles; L1 hit latency is ~30 cycles. Working sets that exceed 50 MB on H100 will thrash L2 and every load goes all the way to HBM.

## The Coalescing Contract

**Memory coalescing** occurs when the memory controller can service all load/store requests from a warp with a minimum number of 128-byte transactions. The ideal case:

- 32 threads in a warp, each accessing one `float`
- Addresses are consecutive and the first address is 128-byte-aligned
- Result: **one** memory transaction services the entire warp

The worst case:

- 32 threads each accessing a `float` at an address in a *different* 128-byte cache line
- Result: **32** separate transactions — the full 32× throughput degradation

Real-world degradation follows the stride. From benchmarks on a Tesla T4 (SM 7.5) using a strided-read kernel:

```
stride    GB/s   relative
    1    206.0   100%
    2    130.5    63%
    4     68.8    33%
    8     33.8    16%
   16     16.8     8%
   32     15.2     7%
   64     13.6     7%
  128     11.2     5%
```

Stride 2 cuts throughput approximately in half because each warp now needs two transactions. Stride 4 halves it again. The degradation flattens at stride ≥ 16 because at that point you are hitting TLB and other memory subsystem limits rather than purely adding transactions.

The mechanism: when consecutive thread indices (threadIdx.x increasing by 1) access consecutive memory addresses (pointer + thread_id), the hardware recognizes the 32 accesses all fall within one 128-byte window and issues one transaction. When thread 0 reads address 0, thread 1 reads address 512 (stride 128), each hits a different cache line — 32 transactions.

## Access Patterns in Practice

### Row-major 2D arrays (coalesced)

```cpp
// A is stored row-major: A[row][col] = A[row * width + col]
// Thread (blockIdx.x * blockDim.x + threadIdx.x) processes one element per row
__global__ void process_rows(const float* __restrict__ A,
                              float* __restrict__ B,
                              int width)
{
    int col = blockIdx.x * blockDim.x + threadIdx.x;  // consecutive threads → consecutive cols
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    if (col < width)
        B[row * width + col] = A[row * width + col] * 2.0f;
}
```

Thread 0 reads `A[row*width + 0]`, thread 1 reads `A[row*width + 1]`, …, thread 31 reads `A[row*width + 31]`. These are 32 consecutive floats in a single 128-byte window. One transaction. Full bandwidth.

### Column access (non-coalesced)

```cpp
__global__ void process_cols(const float* __restrict__ A,
                              float* __restrict__ B,
                              int width, int height)
{
    int row = blockIdx.x * blockDim.x + threadIdx.x;  // consecutive threads → consecutive rows
    int col = blockIdx.y;
    if (row < height)
        B[row * width + col] = A[row * width + col] * 2.0f;  // stride = width
}
```

Thread 0 reads `A[0 * width + col]`, thread 1 reads `A[1 * width + col]`. The stride between accesses is `width` elements (typically 1024 or 4096). Each thread hits a different cache line. 32 transactions per warp. This is the canonical non-coalesced pattern — it appears whenever you process a column of a row-major matrix.

**The fix** is always the same: use shared memory as a staging buffer. Load a tile cooperatively (coalesced reads from global), perform the operation in shared memory (no coalescing needed — shared memory has banks, not cache lines), then write the result back (coalesced writes to global). The matrix transpose is the canonical exercise for this pattern.

### Write coalescing

Coalescing applies equally to stores. A kernel that reads coalesced but writes non-coalesced wastes write bandwidth. The lecture 008 `copyDataNonCoalesced` kernel demonstrates this:

```cpp
__global__ void copyDataNonCoalesced(float* in, float* out, int n) {
    int index = blockIdx.x * blockDim.x + threadIdx.x;
    if (index < n)
        out[index] = in[(index * 2) % n];  // read stride-2: non-coalesced reads
}
```

Thread 0 reads `in[0]`, thread 1 reads `in[2]`, thread 2 reads `in[4]`… — stride 2 reads, ~50% bandwidth efficiency. The writes to `out` are coalesced (sequential index), so the write side is fine. In practice, both the read and write sides must be analyzed independently.

## Nsight Compute Metric: Global Load Efficiency

Nsight Compute reports **Global Load Efficiency** (also called **Global Memory Load Efficiency**) as the ratio of requested bytes to actually-transferred bytes. An efficiency of 100% means every byte fetched from DRAM was used by a thread. A stride-2 pattern shows ~50% efficiency; stride-32 shows ~3%.

To find: open Nsight Compute → Sections → Memory Workload Analysis → L1 Cache Transactions per Request. A ratio above 1.0 for 32-bit loads indicates non-coalesced access.

## Common Misconceptions

❌ **Wrong:** Any sequential memory access is coalesced.
✓ **Correct:** Coalescing requires all threads in a *warp* to access addresses within the same 128-byte cache line. A loop where each thread increments its own pointer is sequential per thread, but if the threads are accessing different cache lines, the warp's access is not coalesced.

---

❌ **Wrong:** Coalescing only matters for reads.
✓ **Correct:** Write coalescing is equally critical. A kernel that accumulates results into a non-sequential output array pays the same 32× penalty on writes as on reads. Profile stores as carefully as loads.

---

❌ **Wrong:** Padding arrays never helps coalescing.
✓ **Correct:** Padding each row to a multiple of 128 bytes (32 floats) aligns row starts to cache-line boundaries. For column access after a transpose, this makes previously non-coalesced accesses perfectly coalesced. The overhead is at most 128 bytes per row — negligible for large matrices.

---

❌ **Wrong:** Global memory is shared with the CPU.
✓ **Correct:** CPU and GPU have separate DRAM. Data moves between them only via explicit `cudaMemcpy`, DMA (direct memory access via pinned memory), or Unified Memory page migration. A pointer valid on the CPU will segfault if dereferenced by a kernel without migration.

---

❌ **Wrong:** All global memory accesses go all the way to HBM.
✓ **Correct:** Global accesses go through L1 (per-SM, ~30-cycle hit) and L2 (shared, ~200-cycle hit) before reaching HBM (~400 cycles). For working sets smaller than L2 (50 MB on H100), repeated accesses will hit L2 and the effective bandwidth is limited by L2 throughput, not HBM bandwidth. L2 bandwidth is significantly higher than HBM bandwidth.

---

❌ **Wrong:** `cudaMalloc` is equivalent to `malloc` in performance.
✓ **Correct:** `cudaMalloc` involves a CUDA driver call that may touch the GPU's page table and interact with the OS. On first use, device memory pages must be mapped. In latency-sensitive paths, pre-allocate all buffers at initialization and use a pool allocator.

## Interview Patterns

**Pattern 1 — Coalescing diagnosis.** Interviewer shows a kernel and asks you to identify the access pattern. Protocol: (1) identify the outermost dimension of the data structure; (2) identify which thread index (threadIdx.x, y, z) maps to that dimension; (3) check if consecutive `threadIdx.x` values produce consecutive memory addresses. If threadIdx.x drives the innermost/fastest dimension, accesses are coalesced. If threadIdx.y or threadIdx.z drives it, they are not.

**Pattern 2 — Transaction count.** "How many memory transactions does this warp issue for a load with stride S?" Answer: `min(32, ceil(32 * S * sizeof(float) / 128))`. For S=1: 1. For S=2: 2. For S=32: 32.

**Pattern 3 — Fix non-coalesced access.** The answer is nearly always: load into shared memory with coalesced threads, `__syncthreads()`, then operate on shared memory in the required layout.

**Pattern 4 — Bandwidth calculation.** "What is the effective bandwidth of this kernel?" Formula: `(bytes_read + bytes_written) / kernel_time_seconds`. Compare to peak HBM bandwidth (3.35 TB/s on H100). Efficiency = achieved / peak. A copy kernel should achieve >90% of peak with coalesced access.

**Pattern 5 — Roofline placement.** "Is this kernel memory-bound or compute-bound?" Compute arithmetic intensity (FLOPs / bytes). If below the H100 BF16 ridge point of 295 FLOPs/byte, it is memory-bound and coalescing dominates performance. If above, compute units are the bottleneck and coalescing matters less.
