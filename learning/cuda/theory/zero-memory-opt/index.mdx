# ZeRO: Zero Redundancy Optimizer for Large-Scale Training

## Why this matters for interviews

ZeRO is the technology that made training 100B+ parameter models feasible on commodity GPU clusters. When interviewing for AI infrastructure roles, you'll be asked: "How do you train a 175B model on 1000 GPUs?" or "What's the memory savings of ZeRO Stage 3?"

Understanding ZeRO requires grasping both the memory layout (what each GPU stores) and the communication protocol (how GPUs coordinate). It's a perfect example of trading communication for memory—a fundamental systems tradeoff.

## The memory problem: training state blows up

For data parallelism (DP), each GPU stores a complete copy of the model. For a model with Ψ parameters:

**Mixed-precision Adam training state per GPU:**
- FP16 weights: 2Ψ bytes
- FP16 gradients: 2Ψ bytes
- FP32 master weights: 4Ψ bytes
- Adam momentum: 4Ψ bytes
- Adam variance: 4Ψ bytes
- **Total: 16Ψ bytes per GPU**

For GPT-3 (175B parameters):
- 16 × 175B = 2.8 TB per GPU

Even with 1000 GPUs, you're storing 2.8 TB × 1000 = 2.8 PB of redundant data!

**The ZeRO insight**: In standard DP, each GPU computes the same forward pass on different data. The model is identical. Why store 1000 copies when you can store 1?

## ZeRO fundamentals: partitioning model states

ZeRO (Zero Redundancy Optimizer) eliminates redundancy by **partitioning** model states across GPUs instead of replicating them. Each GPU stores only its partition, then **all-gathers** the full model when needed.

### Standard data parallelism (baseline)

```
GPU 0: [full model] [full gradients] [full optimizer state]
GPU 1: [full model] [full gradients] [full optimizer state]
GPU 2: [full model] [full gradients] [full optimizer state]
...

Each step:
1. Forward pass (each GPU uses full model)
2. Backward pass (each GPU computes full gradients)
3. All-reduce gradients across GPUs
4. Update full model on each GPU
```

Memory per GPU: 16Ψ bytes

### ZeRO Stage 1: partition optimizer state

```
GPU 0: [full model] [full gradients] [optimizer state partition 0]
GPU 1: [full model] [full gradients] [optimizer state partition 1]
GPU 2: [full model] [full gradients] [optimizer state partition 2]
...

Each step:
1. Forward pass (each GPU uses full model)
2. Backward pass (each GPU computes full gradients)
3. Reduce-scatter gradients to responsible GPU
4. Each GPU updates its optimizer partition
5. All-gather updated parameters
```

Memory per GPU: 2Ψ + 2Ψ + 4Ψ/Nd + 4Ψ/Nd = (4Ψ + 8Ψ/Nd) bytes

For 64 GPUs:
- Standard DP: 16Ψ
- ZeRO-1: 4Ψ + 8Ψ/64 = 4.125Ψ (≈4× reduction)

### ZeRO Stage 2: partition optimizer state + gradients

```
GPU 0: [full model] [gradients partition 0] [optimizer state partition 0]
GPU 1: [full model] [gradients partition 1] [optimizer state partition 1]
GPU 2: [full model] [gradients partition 2] [optimizer state partition 2]
...

Each step:
1. Forward pass (each GPU uses full model)
2. Backward pass (each GPU computes gradients for its partition only)
3. Reduce-scatter gradients to responsible GPU
4. Each GPU updates its optimizer partition
5. All-gather updated parameters
```

Memory per GPU: 2Ψ + 2Ψ/Nd + 8Ψ/Nd = 2Ψ + 10Ψ/Nd bytes

For 64 GPUs:
- Standard DP: 16Ψ
- ZeRO-2: 2Ψ + 10Ψ/64 = 2.156Ψ (≈8× reduction)

### ZeRO Stage 3: partition optimizer state + gradients + parameters

```
GPU 0: [model partition 0] [gradients partition 0] [optimizer state partition 0]
GPU 1: [model partition 1] [gradients partition 1] [optimizer state partition 1]
GPU 2: [model partition 2] [gradients partition 2] [optimizer state partition 2]
...

Each step:
1. All-gather full model (before forward)
2. Forward pass (each GPU uses full model temporarily)
3. Release model (before backward)
4. All-gather full model (before backward)
5. Backward pass (each GPU computes gradients for its partition)
6. Reduce-scatter gradients to responsible GPU
7. Release model
8. Each GPU updates its optimizer partition
9. All-gather updated parameters
```

Memory per GPU: 16Ψ/Nd bytes

For 64 GPUs:
- Standard DP: 16Ψ
- ZeRO-3: 16Ψ/64 = 0.25Ψ (64× reduction)

## Memory savings by stage

| Stage | What's partitioned | Memory per GPU (64 GPUs) | Savings |
|-------|-------------------|--------------------------|---------|
| 0 (baseline) | Nothing | 16Ψ | 1× |
| 1 | Optimizer state | ~4.125Ψ | ~4× |
| 2 | Optimizer + gradients | ~2.156Ψ | ~8× |
| 3 | Optimizer + gradients + params | ~0.25Ψ | 64× |

### Real-world example: 7.5B parameter model

Using K=12 (constant factor accounting for activation overhead):

| Stage | Memory required (64 GPUs) |
|-------|--------------------------|
| 0 | 120 GB |
| 1 | 31.4 GB |
| 2 | 16.6 GB |
| 3 | 1.9 GB |

With ZeRO-3, a 7.5B model fits on a single GPU! For larger models, ZeRO-3 enables training 100B+ models on commodity clusters.

## Communication overhead analysis

A common misconception: ZeRO-3 must be much slower due to all-gathering parameters every step. Let's analyze the communication volume.

### Standard data parallelism

Each step:
- All-reduce gradients: 2Ψ elements (each gradient is 2 bytes, but we communicate the full gradient)

Communication volume: **2Ψ per step**

### ZeRO Stage 1 and 2

Each step:
- Reduce-scatter gradients: Ψ elements
- All-gather parameters: Ψ elements

Communication volume: **2Ψ per step** (same as baseline!)

The key insight: reduce-scatter + all-gather = same total data as all-reduce.

### ZeRO Stage 3

Each step:
- All-gather parameters (forward): Ψ elements
- All-gather parameters (backward): Ψ elements
- Reduce-scatter gradients: Ψ elements

Communication volume: **3Ψ per step**

ZeRO-3 adds 50% communication overhead vs baseline DP (3Ψ vs 2Ψ). This is the price for Nd-fold memory reduction.

### Practical performance

The 50% overhead is often **amortized**:
- Larger DP degree allows larger global batch size
- Each GPU computes more, hiding communication latency
- With high-bandwidth networking (NVLink, InfiniBand HDR), overhead drops

DeepSpeed benchmarks show near-linear scaling up to 64 GPUs for many workloads.

## Combining ZeRO with tensor parallelism

ZeRO and tensor parallelism are **complementary**, not alternatives:

**Within a node (NVLink):**
- Use tensor parallelism for low-latency communication
- Each "super-GPU" appears as one unit to ZeRO

**Across nodes (InfiniBand):**
- Use ZeRO-DP for memory efficiency
- Minimize cross-node all-gathers

### Example: training a 175B model on 512 GPUs

```
8 nodes × 8 GPUs/node = 64 "super-GPUs" (tensor parallelism within each)
ZeRO-3 across 64 super-GPUs

Per super-GPU memory:
- Model partition: 175B/64 = 2.7B params
- With tensor parallelism (4-way): each GPU stores 2.7B/4 = 675M params
```

This is how Megatron-LM + ZeRO trains GPT-3.

## Common misconceptions

❌ **Wrong**: "ZeRO and model parallelism (tensor/pipeline) are alternatives"

✓ **Correct**: ZeRO is a **data parallelism variant**, not a replacement for model parallelism. ZeRO partitions optimizer states/gradients/parameters within data parallelism. You can (and do!) combine ZeRO with tensor parallelism: tensor parallelism within a node, ZeRO across nodes.

❌ **Wrong**: "Stage 3 always has 50% communication overhead"

✓ **Correct**: The 50% overhead is **relative to baseline DP all-reduce**. In practice, overhead is often lower because:
- Larger DP degree allows larger batch per GPU (better utilization)
- Communication can be overlapped with computation
- High-bandwidth networks reduce absolute latency

Super-linear speedup is observed at 64-400 GPUs!

❌ **Wrong**: "ZeRO requires model code changes"

✓ **Correct**: ZeRO is implemented as an **optimizer/DDP wrapper**. You don't need to change model code. DeepSpeed's ZeRO integrates with PyTorch DDP transparently. The model sees the same API; ZeRO handles all-gathering parameters behind the scenes.

## Interview patterns

**Question 1**: "What's the memory savings of ZeRO-3 with 64 GPUs?"

**Answer**: 64× reduction. Each GPU stores 1/64 of the model state (optimizer, gradients, parameters). A 175B model that needs 2.8 TB per GPU in standard DP needs only 44 GB per GPU with ZeRO-3.

**Question 2**: "Compare ZeRO-1, ZeRO-2, and ZeRO-3."

**Answer**:
- ZeRO-1: partitions optimizer state only (4× savings)
- ZeRO-2: partitions optimizer + gradients (8× savings)
- ZeRO-3: partitions optimizer + gradients + parameters (Nd× savings)
Communication overhead: ZeRO-1/2 same as DP, ZeRO-3 adds 50%.

**Question 3**: "What's the communication volume of ZeRO-3 per step?"

**Answer**: 3Ψ elements (two all-gathers for params + one reduce-scatter for grads). Baseline DP is 2Ψ (one all-reduce). So ZeRO-3 adds 50% communication.

**Question 4**: "Why does ZeRO-3 all-gather params twice (forward and backward)?"

**Answer**: Each GPU only stores its parameter partition. For the forward pass, it needs the full model, so all-gather. After forward, it releases the model to save memory. For backward, it needs the full model again to compute gradients, so all-gather again.

**Question 5**: "How do you combine ZeRO and tensor parallelism?"

**Answer**: Use tensor parallelism within a node (NVLink) and ZeRO-DP across nodes (InfiniBand). Each "super-GPU" (group of GPUs doing tensor parallelism) appears as one unit to ZeRO.
