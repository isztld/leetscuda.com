# Warp Divergence & Branch Efficiency

## Why this matters for interviews

Warp divergence questions appear in every performance-focused CUDA interview at NVIDIA, AMD, and any company building GPU kernels. Expect "why is my kernel slower with this if-statement?", "how can I measure branch efficiency?", and "what is the difference between pre-Volta and Volta+ behavior?" These are not theoretical questions — divergence is the silent performance killer in many production kernels. This article gives you the precise mental model to answer them and write divergence-free code.

## The SIMT Model and Execution Context

To understand divergence, you must first understand the SIMT (Single Instruction, Multiple Thread) model. A warp of 32 threads executes the same instruction at the same time. This is hardware SIMT: the warp scheduler issues one instruction, and all 32 threads execute it in parallel (each on its own data).

**The critical constraint:** When threads within the same warp encounter a conditional (if/else, loop with varying trip counts), they may need to take different execution paths. The hardware cannot execute two different instructions in the same cycle on the same warp.

**Divergence resolution:** The warp must serialize the different paths. Threads in the inactive path are "masked off" (predicated) while the active threads execute. Then the paths swap. The warp only "converges" when all threads have completed both paths.

**Example: A divergent if/else**
```cpp
__global__ void divergent_kernel(float* data, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        if (data[idx] > 0.5f) {
            // Path A: 40% of threads
            data[idx] = data[idx] * 4.0f;
        } else {
            // Path B: 60% of threads
            data[idx] = data[idx] + 2.0f;
        }
    }
}
```

In this kernel, if a warp contains threads with `data[idx] > 0.5f` mixed with threads with `data[idx] <= 0.5f`, the warp executes both branches: first Path A with 12 threads (masked: 20 off), then Path B with 20 threads (masked: 12 off). Throughput is reduced to ~40% of ideal (60% branch efficiency).

## Pre-Volta vs Volta+ Independent Thread Scheduling

**Pre-Volta (cc < 7.0):** Each warp has a single program counter (PC). When divergence occurs, the warp literally serializes: one path at a time, no parallelism. The performance penalty is severe — a 2-way divergence cuts throughput by exactly 2×.

**Volta+ (cc >= 7.0):** Each thread has its own program counter and independent scheduling. This enables more flexible divergence handling: the scheduler can interleave execution from different paths more efficiently. However, **this does not eliminate serialization**. If threads in the same warp need to execute different instructions, the hardware still must issue those instructions separately.

**Key insight:** Independent thread scheduling enables *more* divergence patterns (e.g., threads within a warp can now make independent scheduling decisions), but it does not remove the fundamental SIMT constraint. Divergence still costs throughput.

## How the Compiler Handles Divergence

The NVIDIA compiler (ptxas) is clever about divergence. Consider this SASS output from a simple if/else:

```nasm
LDG.E.SYS R4, [R2]                       // L1: Load data[idx]
FSETP.GT.AND P0, PT, R4.reuse, 0.5, PT   // L2: Set predicate P0 = (data[idx] > 0.5)
FADD R0, R4, 2                           // L3: ALL threads compute 2 + data[idx]
@P0 FMUL R0, R4, 4                       // L4: Only P0=true threads compute 4 * data[idx]
FMUL R5, R0, R0                          // L5: ALL threads compute R0 * R0
STG.E.SYS [R2], R5                       // L6: All threads store
```

Notice the compiler's optimization: **L3 is executed by all threads**, even those that will take Path B. Then **L4 is predicated** — only threads with P0=true overwrite R0. This is a trade-off: waste a cheap FADD on 60% of threads to avoid a full branch serialization.

**The takeaway:** Modern compilers often use "JIT fusion" to avoid serialization when the cost of extra computation is less than the cost of flow control. But this only works when both branches compute similar operations.

## Common Divergence Patterns

**Pattern 1: Bounds check (usually safe)**
```cpp
int idx = blockIdx.x * blockDim.x + threadIdx.x;
if (idx < n) {
    data[idx] = ...;  // Only last warp diverges
}
```
This has high branch efficiency. Most warps are all-in or all-out. Only the last partial warp has divergence.

**Pattern 2: Data-dependent branches (divergent)**
```cpp
if (data[idx] > threshold) {
    // Different threads take different paths based on data
}
```
This is the classic divergent pattern. Every warp with mixed data will serialize.

**Pattern 3: Loops with varying trip counts**
```cpp
while (data[idx] > threshold) {
    data[idx] -= step;
}
```
If different threads need different iterations, the warp diverges every loop. Some threads finish early (executing no-op or waiting), others continue.

**Pattern 4: Grid-stride loops (safe)**
```cpp
for (int i = idx; i < n; i += blockDim.x * gridDim.x) {
    data[i] = ...;  // All threads in warp have same trip count
}
```
This is safe because all threads in a warp have the same stride and converge at the loop condition.

## Measuring Branch Efficiency

Nsight Compute exposes the `Branch Efficiency` metric:
- **100%:** No divergence — all threads in every warp take the same path
- **<100%:** Some divergence occurred. 50% means average 2× serialization.

**How to measure:**
```bash
nsight compute --metrics smsp::warps_executed_divergent,yield_ratio ./your_kernel
```

Or in code, query the metric after kernel launch:
```cpp
cudaEventRecord(start);
myKernel<<<grid, block>>>(...);
cudaEventRecord(stop);

// Query Branch Efficiency from Nsight Compute profile
```

**Rule of thumb:** If Branch Efficiency is below 80%, investigate the kernel for divergence sources. If below 50%, you're losing significant throughput.

## Reducing Divergence: Techniques

**Technique 1: Sort data to converge branches**
If you know that most threads will take the same path, sort the input data so that similar values are in the same warps. This localizes divergence to fewer warps.

**Technique 2: Replace branches with arithmetic**
```cpp
// Divergent
if (isEven) {
    result = x * 2;
} else {
    result = x + 1;
}

// Convergent (no branches)
int mask = isEven ? 1 : 0;
result = mask * (x * 2) + (!mask) * (x + 1);
```
This trades computation (cheap) for control flow (expensive on GPU). Works when both branches are similar cost.

**Technique 3: Split the kernel**
Launch two kernels: one for threads that take path A, one for path B. This guarantees no intra-warp divergence but adds launch overhead.

**Technique 4: Use warp-level primitives**
```cpp
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

tiled_partition<32> warp = cg::tiled_partition<32>(tg);
if (warp.any(condition)) {
    // Handle case where any thread in warp has condition
}
```
Cooperative groups let you detect and handle divergence at the warp level explicitly.

## Common Misconceptions

**❌ Wrong:** Any conditional causes warp divergence.
**✓ Correct:** Divergence only occurs when threads **in the same warp** take different paths. A warp where all 32 threads enter the if-branch (or all 32 skip it) has zero divergence. The common bounds check `if (idx < n)` only diverges in the last partial warp.

**❌ Wrong:** Divergence in different warps is a problem.
**✓ Correct:** Different warps can take completely different execution paths with zero cost. A warp where all threads take the if-branch and another warp where all threads take the else-branch both execute at 100% efficiency. Divergence is strictly an **intra-warp** phenomenon.

**❌ Wrong:** Independent thread scheduling (Volta+) eliminates divergence overhead.
**✓ Correct:** Volta+ gives each thread its own program counter and enables more flexible scheduling, but **it does not eliminate serialization**. If 16 threads need instruction A and 16 need instruction B, the hardware must still issue both instructions. The penalty is reduced but not zero.

**❌ Wrong:** A kernel with 100% branch efficiency is optimal.
**✓ Correct:** Branch efficiency is just one metric. A kernel can have 100% branch efficiency but be memory-bound, or have poor occupancy, or underutilize Tensor Cores. Branch efficiency must be interpreted alongside other Nsight Compute metrics.

**❌ Wrong:** Predication is always worse than branching.
**✓ Correct:** The compiler often fuses both branches into predicated code when the cost of extra computation is less than the cost of flow control. The SASS shown earlier demonstrates this: computing both paths and selecting with a predicate can be faster than serialization.

## Interview Patterns

**"Why is my kernel slower with this if-statement?"**
Your if-statement causes warp divergence. If threads in the same warp take different paths, the warp serializes both paths. A 2-way divergence (50/50 split) cuts throughput by ~2×. Check Nsight Compute's Branch Efficiency metric — if it's below 80%, you have divergence. Fix by sorting data to converge branches, replacing branches with arithmetic, or splitting into separate kernels.

**"What is the difference between pre-Volta and Volta+ divergence behavior?"**
Pre-Volta (cc < 7.0) has a single program counter per warp, so divergence always serializes the paths (2× slowdown for 2-way). Volta+ (cc >= 7.0) gives each thread its own PC and independent scheduling, enabling more flexible handling. However, it does not eliminate the fundamental SIMT constraint: if threads in the same warp need different instructions, the hardware must still issue them separately. The penalty is reduced but not zero.

**"How can I measure warp divergence in my kernel?"**
Use Nsight Compute. The `Branch Efficiency` metric shows the fraction of non-divergent branches (100% = no divergence). Also check `smsp::warps_executed_divergent` to see how many warps had divergent execution. Run `nsight compute --metrics smsp::branch_efficiency,yield_ratio ./your_kernel`. If branch efficiency is below 80%, investigate the kernel for data-dependent branches.

**"Can I eliminate divergence entirely?"**
For most real kernels, no. But you can minimize it. Techniques: (1) sort data to converge branches, (2) replace branches with arithmetic when both paths are cheap, (3) split kernels to isolate divergent code paths, (4) use cooperative groups for explicit warp-level handling. The goal is not 100% efficiency but maximizing achieved throughput.

## Key Numbers to Memorize

| Property | Value |
|-----|-|
| Divergence penalty (2-way, 50/50 split) | ~2× slowdown |
| Divergence penalty (32-way, all different) | ~32× slowdown |
| Pre-Volta vs Volta+ penalty | Volta+ reduced but non-zero |
| Branch Efficiency threshold (concern) | < 80% |
| Branch Efficiency threshold (severe) | < 50% |
| Bounds check divergence | Only in last partial warp |

**Practical examples:**
- 50/50 split: Branch Efficiency = 50%, throughput = 50% of ideal
- 30/2 split (30 threads path A, 2 path B): Branch Efficiency = ~94%, throughput = ~94% of ideal
- Last warp only: 31 full warps at 100%, 1 warp at 50% → Overall efficiency ~98.5%
