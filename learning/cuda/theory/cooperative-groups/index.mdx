# Cooperative Groups

Cooperative Groups is a CUDA programming model extension (introduced in CUDA 9) that provides explicit synchronization handles for subsets of threads at different granularities. It replaces the rigid, single-level synchronization model of traditional CUDA (`__syncthreads()` only) with a flexible hierarchy: thread blocks, warps, sub-warps, and entire grids.

This model allows programmers to express fine-grained synchronization patterns that were previously impossible or required hacky workarounds, enabling more efficient reductions, scans, and collective operations.

## Why Cooperative Groups Matter for Interviews

Modern high-performance CUDA kernels increasingly rely on sophisticated synchronization patterns:

- **Warp-level reductions** using shuffle instructions instead of shared memory
- **Multi-block algorithms** that require grid-wide synchronization (e.g., multi-stage reductions)
- **Dynamic partitioning** where threads form arbitrary groups at runtime
- **Hopper warpgroup operations** (WMMA, wgmma) that operate on 128 threads

Interviewers at AI infrastructure companies (CoreWeave, Run:ai, Lambda) and large-scale ML teams (OpenAI, Meta AI, NVIDIA) increasingly ask about cooperative groups when discussing kernel optimization. Understanding the abstraction allows you to reason about tradeoffs between different synchronization strategies.

## The Limitation of Traditional CUDA Synchronization

Traditional CUDA provides only two synchronization primitives:

1. **`__syncthreads()`**: Barrier within a thread block (CTA)
2. **Atomics**: Atomic read-modify-write operations (no ordering guarantees)

This creates several problems:

```cpp
// Problem 1: Can't synchronize subsets of threads
__global__ void badPattern(float* data) {
    int tid = threadIdx.x;

    if (tid < 32) {
        // Thread 0-31 do work
        data[tid] = compute(tid);
    } else {
        // Thread 32-63 do different work
        data[tid] = other_compute(tid);
    }

    // This sync includes ALL threads, even those not participating
    __syncthreads(); // Overkill!
}

// Problem 2: Can't express warp-level operations cleanly
__global__ void naiveWarpReduce(float* input, float* output) {
    __shared__ float shared[32];
    int tid = threadIdx.x;

    shared[tid] = input[tid];
    __syncthreads(); // Full block sync for warp-local op!

    // Then use shuffle instructions anyway
    float sum = shared[tid];
    for (int offset = 16; offset > 0; offset >>= 1) {
        sum += __shfl_down_sync(0xffffffff, sum, offset);
    }

    if (tid % 32 == 0) output[tid / 32] = sum;
}
```

Cooperative Groups solves these problems by providing typed abstractions for different thread groupings.

## Thread Block (CTA) Synchronization

The most basic cooperative group is the thread block, equivalent to `__syncthreads()` but composable:

```cpp
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ void myKernel(float* data, int n) {
    // Get the thread block group handle
    cg::thread_block tb = cg::this_thread_block();

    int tid = tb.thread_rank();  // Thread index 0..blockDim.x-1
    int blockSize = tb.size();   // Total threads in block

    // Do some work
    data[tid] = compute(tid);

    // Synchronize all threads in the block
    tb.sync();  // Equivalent to __syncthreads()

    // Now all threads see the updated data
    float sum = 0.0f;
    for (int i = 0; i < 4; i++) {
        int idx = tid + i * blockSize;
        if (idx < n) sum += data[idx];
    }
}
```

Key observations:
- `thread_block` is a handle, not a new thread hierarchy level
- `tb.sync()` is functionally identical to `__syncthreads()`
- The benefit is composability: you can pass `tb` to functions that need block-level sync
- No cooperative launch required — works in normal kernels

## Tiled Partitions: Warp and Sub-Warp Groups

Tiled partitions split a thread block into smaller groups of a specified size:

```cpp
__global__ void tiledPartitionExample(float* input, float* output) {
    cg::thread_block tb = cg::this_thread_block();

    // Split block into warps (32 threads each)
    cg::tiled_partition<32> warp(tb);

    int tid = tb.thread_rank();
    int warpRank = warp.thread_rank();  // 0..31 within this warp
    int warpId = tid / 32;              // Which warp am I in?

    // Perform warp-local reduction
    float local_val = input[tid];
    float sum = local_val;

    // Warp-level shuffle reduction
    for (int offset = warp.size() / 2; offset > 0; offset >>= 1) {
        sum += warp.shfl_down(local_val, offset);
    }

    // First thread in each warp has the result
    if (warpRank == 0) {
        output[warpId] = sum;
    }
}
```

### Common Partition Sizes

| Partition Size | Use Case |
|----------------|----------|
| `tiled_partition<32>` | Warp-level operations (most common) |
| `tiled_partition<16>` | Half-warp operations |
| `tiled_partition<8>` | Sub-warp operations |
| `tiled_partition<64>` | Two-warp operations |

### The `coalesced_threads()` Optimization

After a divergent branch, `coalesced_threads()` returns only the active threads:

```cpp
__global__ void selectiveWork(float* data, int* mask) {
    cg::thread_block tb = cg::this_thread_block();

    // Some threads take the branch, others don't
    if (mask[tb.thread_rank()]) {
        data[tb.thread_rank()] = expensive_compute(tb.thread_rank());
    }

    // Get only the active threads (those that took the branch)
    auto active = cg::coalesced_threads();

    // Sync only active threads, not the full block
    // This is more efficient if the branch is highly skewed
    active.sync();

    // Continue with active threads
}
```

This is more efficient than `tb.sync()` when the branch is highly skewed (only a few threads took the branch), because it avoids synchronizing inactive threads.

## Warp-Level Primitives

Cooperative groups exposes warp-level shuffle operations cleanly:

```cpp
__global__ void warpReduceWithCG(float* input, float* output) {
    cg::thread_block tb = cg::this_thread_block();
    cg::tiled_partition<32> warp(tb);

    float local_val = input[tb.thread_rank()];
    float sum = local_val;

    // Use warp-level shuffle operations
    for (int offset = warp.size() / 2; offset > 0; offset >>= 1) {
        sum += warp.shfl_down(local_val, offset);
    }

    // Other shuffle operations available:
    // warp.shfl_up(val, offset) — shift up
    // warp.shfl_xor(val, offset) — butterfly pattern
    // warp.shfl(val, src_rank) — arbitrary source

    if (warp.thread_rank() == 0) {
        output[tb.thread_rank() / 32] = sum;
    }
}
```

Compared to native shuffle intrinsics (`__shfl_down_sync`), cooperative groups:
- Don't require explicit mask parameters
- Are more composable with other group operations
- Make the scope of the operation explicit

## Grid-Level Synchronization (Advanced)

Grid-level synchronization requires a **cooperative launch** via `cudaLaunchCooperativeKernel`:

```cpp
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ void gridSyncKernel(float* data, int n) {
    cg::grid_group g = cg::this_grid();

    int tid = blockIdx.x * blockDim.x + threadIdx.x;
    int totalThreads = gridDim.x * blockDim.x;

    // Each thread does some work
    data[tid] = compute(tid);

    // Grid-wide synchronization
    // ALL threads in ALL blocks must reach this point
    g.sync();

    // Now all threads can safely read data from any thread
    float sum = 0.0f;
    for (int i = 0; i < totalThreads; i++) {
        sum += data[i];
    }

    if (tid == 0) printf("Sum: %f\n", sum);
}

// Host side
void launchGridSync(float* d_data, int n) {
    int blocks = 4;
    int threads = 256;

    // Must check if device supports cooperative launch
    int maxBlocks = 0;
    cudaDeviceGetAttribute(&maxBlocks, cudaDevAttrMaxCooperativeBlocksPerSM, 0);

    if (blocks > maxBlocks) {
        printf("Warning: cooperative launch may fail\n");
    }

    // Configure for cooperative launch
    cudaLaunchConfig_t config = {
        .gridDim = dim3(blocks),
        .blockDim = dim3(threads),
        .dynamicSmemBytes = 0,
        .stream = 0,
        .coop = true  // Enable cooperative launch
    };

    cudaLaunchCooperativeKernel((void*)gridSyncKernel, config, d_data, n);
    cudaDeviceSynchronize();
}
```

### Critical Requirements for Grid-Level Sync

1. **Cooperative launch required**: Must use `cudaLaunchCooperativeKernel`, not regular `<<<>>>` launch
2. **Device support**: The GPU must support cooperative launch (check with `cudaDeviceGetAttribute`)
3. **Thread limit**: Total thread count must not exceed device limits
4. **No early exits**: ALL threads must reach the barrier — no thread can exit early or the program deadlocks
5. **Single grid only**: Cannot synchronize across multiple concurrent grids

**Common misconception**: Grid sync is like a supercharged `__syncthreads()`. In reality, it has strict requirements that make it much harder to use correctly. Most algorithms avoid grid sync by using multi-pass reductions with atomic operations instead.

## Generic Group Algorithms

One of the most powerful aspects of cooperative groups is that you can write algorithms that work with any group type:

```cpp
// Generic reduction that works for any group type
template<typename ThreadGroup>
__device__ typename ThreadGroup::thread_result_type
reduce_sum(ThreadGroup g, float* workspace, float val) {
    int rank = g.thread_rank();
    int size = g.size();

    // Store this thread's value
    workspace[rank] = val;
    g.sync();

    // Parallel reduction
    for (int stride = size / 2; stride > 0; stride >>= 1) {
        if (rank < stride) {
            workspace[rank] += workspace[rank + stride];
        }
        g.sync();
    }

    // Only rank 0 has the result
    if (rank == 0) {
        return workspace[0];
    }
    return 0;
}

// Usage with any group type
__global__ void demoGenericReduce(float* input, float* output) {
    auto tb = cg::this_thread_block();
    auto warp = cg::tiled_partition<32>(tb);

    extern __shared__ float workspace[];

    // Reduce at block level
    float blockSum = reduce_sum(tb, workspace, input[tb.thread_rank()]);

    // Or at warp level
    float warpSum = reduce_sum(warp, workspace, input[tb.thread_rank()]);
}
```

This genericity is impossible with traditional CUDA primitives because `__syncthreads()` has no handle to pass around.

## Common Misconceptions

❌ **Wrong**: Cooperative groups require cooperative launch.
✓ **Correct**: Only `grid_group` coordination requires cooperative launch. `thread_block` and `tiled_partition` work in normal kernels.

❌ **Wrong**: `__syncthreads()` and `thread_block::sync()` are different operations.
✓ **Correct**: They are equivalent — `thread_block::sync()` is just the cooperative groups abstraction of `__syncthreads()`.

❌ **Wrong**: `coalesced_threads()` works across blocks.
✓ **Correct**: It only works within a block, returning the active threads after divergence.

❌ **Wrong**: Grid sync is commonly used in production kernels.
✓ **Correct**: Grid sync is rare in production because of its strict requirements. Most algorithms use multi-pass reductions with atomics instead.

❌ **Wrong**: Warpgroups (4 warps = 128 threads) are the same as `tiled_partition<128>`.
✓ **Correct**: Warpgroups are a Hopper-specific hardware concept for WMMA operations. `tiled_partition<128>` is a software abstraction that works on all architectures but doesn't have the same hardware guarantees.

## Interview Patterns

Cooperative groups appears in these interview contexts:

1. **Algorithm design**: "How would you implement a parallel reduction that scales across multiple SMs?" Answer: Multi-block reduction using `tiled_partition<32>` for warp-level reduction, then combine block results via a second reduction pass or atomic operations. Grid sync is an alternative but has strict requirements.

2. **Performance tradeoffs**: "When would you use warp shuffles vs. shared memory for a reduction?" Answer: Shuffles are faster for warp-local operations (no shared memory latency, no bank conflicts). Shared memory is necessary for cross-warp or cross-block operations.

3. **Code review**: "What's wrong with this kernel?" (shows kernel with `__syncthreads()` inside a divergent branch). Answer: `__syncthreads()` inside divergence can cause deadlock if some threads skip it. Use `coalesced_threads()` to sync only active threads.

4. **System design**: "Design a kernel that needs to synchronize across all threads in the grid. What options do you have?" Answer: (1) `cudaLaunchCooperativeKernel` with `grid_group::sync()` — strict requirements, (2) Multi-pass reduction with intermediate results in global memory and grid-wide barriers between passes — more flexible but more complex.
