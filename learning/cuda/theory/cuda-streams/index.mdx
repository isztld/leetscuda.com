# CUDA Streams & Async Transfers

CUDA streams enable concurrent execution of kernels and memory transfers — a critical technique for hiding PCIe transfer latency and achieving higher GPU utilization. This article covers asynchronous streams, event-based synchronization, and the cooperative groups programming model for explicit thread coordination.

## Why Streams Matter for Interviews

Without streams, a typical CUDA program serializes:

1. Copy H→D (host-to-device)
2. Run kernel
3. Copy D→H (device-to-host)

Each stage must complete before the next begins. On a system with an H100 connected via PCIe Gen5 x16 (64 GB/s bidirectional), copying 1 GB takes ~15 ms. If your kernel also takes 15 ms, the total is 45 ms — but with async overlap, it could be as low as 20 ms.

Modern GPUs have separate engines for kernel execution, H2D DMA, and D2H DMA. These engines can run **simultaneously** — a kernel can execute while the DMA engine transfers data for the next batch. Interviewers frequently ask you to explain how you'd overlap computation with communication, or to reason about throughput improvements from multi-stream pipelining.

## The Default Stream: A Hidden Bottleneck

Every CUDA operation you issue without specifying a stream goes into the **default stream** (stream 0). Operations in the default stream are synchronizing: they block until all preceding operations on the device complete, and subsequent operations wait for the default stream operation to finish. This makes the default stream safe but eliminates all overlap opportunity.

**Common misconception**: Many developers assume `cudaMemcpy` (the synchronous variant) only blocks the host thread. In reality, it also synchronizes with all GPU work — if there's an in-progress kernel in another stream, `cudaMemcpy` will wait for it to complete before starting the transfer.

**Correct approach**: Always use `cudaMemcpyAsync` with an explicit non-default stream when you need overlap. This requires host memory to be page-locked (pinned), which is a small tradeoff for achieving concurrent execution.

## Creating and Using Streams

```cpp
cudaStream_t stream1, stream2;
cudaStreamCreate(&stream1);
cudaStreamCreate(&stream2);

// Issue work into specific streams
cudaMemcpyAsync(d_a, h_a, bytes, cudaMemcpyHostToDevice, stream1);
kernel<<<grid, block, 0, stream1>>>(d_a, N);
cudaMemcpyAsync(h_out, d_a, bytes, cudaMemcpyDeviceToHost, stream1);

// Destroy when done
cudaStreamDestroy(stream1);
cudaStreamDestroy(stream2);
```

`cudaMemcpyAsync` requires the host buffer to be **pinned memory** (allocated with `cudaMallocHost` or `cudaHostAlloc`) — pageable memory cannot be transferred asynchronously because the OS may swap it out during the transfer, causing data corruption.

```cpp
float* h_pinned;
cudaMallocHost(&h_pinned, bytes); // pinned (page-locked) host memory
```

## Pipelining with Multiple Streams

The classic pipelining pattern divides data into chunks and interleaves H2D, compute, and D2H across multiple streams:

```cpp
const int NUM_STREAMS = 4;
const int chunkSize = N / NUM_STREAMS;
cudaStream_t streams[NUM_STREAMS];
for (int i = 0; i < NUM_STREAMS; i++) cudaStreamCreate(&streams[i]);

for (int i = 0; i < NUM_STREAMS; i++) {
    int offset = i * chunkSize;
    // H2D for chunk i
    cudaMemcpyAsync(d_in + offset, h_in + offset,
                    chunkSize * sizeof(float),
                    cudaMemcpyHostToDevice, streams[i]);
    // Compute on chunk i
    process_kernel<<<gridChunk, block, 0, streams[i]>>>(
        d_in + offset, d_out + offset, chunkSize);
    // D2H for chunk i
    cudaMemcpyAsync(h_out + offset, d_out + offset,
                    chunkSize * sizeof(float),
                    cudaMemcpyDeviceToHost, streams[i]);
}

// Wait for all streams to complete
cudaDeviceSynchronize();
for (int i = 0; i < NUM_STREAMS; i++) cudaStreamDestroy(streams[i]);
```

With enough chunks, the GPU pipeline looks like:

```
Time →
Stream 0: [H2D_0][Kernel_0][D2H_0]
Stream 1:         [H2D_1][Kernel_1][D2H_1]
Stream 2:                 [H2D_2][Kernel_2][D2H_2]
Stream 3:                         [H2D_3][Kernel_3][D2H_3]
```

The DMA engines and compute engines overlap, reducing idle time. A well-tuned multi-stream implementation can achieve 3-4x throughput improvement over sequential single-stream execution, assuming the GPU has sufficient concurrency capacity.

## CUDA Events for Timing and Dependencies

**Events** are synchronization markers that can be inserted into a stream and queried or waited on:

```cpp
cudaEvent_t start, stop;
cudaEventCreate(&start);
cudaEventCreate(&stop);

cudaEventRecord(start, stream1); // record into stream
kernel<<<grid, block, 0, stream1>>>(args...);
cudaEventRecord(stop, stream1);

cudaEventSynchronize(stop); // block host until stop event fires
float ms;
cudaEventElapsedTime(&ms, start, stop);
printf("Kernel time: %.3f ms\n", ms);

cudaEventDestroy(start);
cudaEventDestroy(stop);
```

Events are also used for **cross-stream synchronization**:

```cpp
cudaEvent_t dataReady;
cudaEventCreate(&dataReady);

// Stream 1 produces data
produce_kernel<<<grid, block, 0, stream1>>>(d_data);
cudaEventRecord(dataReady, stream1);

// Stream 2 waits for stream 1's event before consuming
cudaStreamWaitEvent(stream2, dataReady, 0);
consume_kernel<<<grid, block, 0, stream2>>>(d_data);
```

`cudaStreamWaitEvent` inserts a dependency in the stream's hardware queue without blocking the host thread. The GPU scheduler will delay issuing work from stream 2 until the event in stream 1 fires.

## Concurrency Limits and HyperQ

Even with multiple streams, concurrent execution is constrained by hardware resources:

- **Kernel concurrency**: Modern GPUs support up to 32 hardware work queues (HyperQ on Kepler+) and up to 128 concurrent kernels. Small kernels that don't saturate the SM benefit most from concurrent execution.
- **DMA concurrency**: There is typically one H2D DMA engine and one D2H DMA engine, so you can overlap one H2D and one D2H transfer simultaneously.
- **PCIe bandwidth**: All streams share the PCIe bus; issuing many H2D transfers simultaneously does not increase total bandwidth.

**Common misconception**: Operations in different streams always run concurrently. In reality, they may serialize if the GPU is fully occupied or lacks sufficient hardware queues. Use Nsight Systems (`nsys profile`) to verify actual overlap in the GPU timeline.

## Synchronization Options

| Call | Blocks | Scope |
|------|------|------|
| `cudaDeviceSynchronize()` | Host | All operations on all streams |
| `cudaStreamSynchronize(s)` | Host | All operations in stream `s` |
| `cudaEventSynchronize(e)` | Host | Until event `e` fires |
| `cudaStreamWaitEvent(s,e)` | GPU stream | Stream `s` waits for event `e` |
| `__threadfence()` | GPU thread | Memory visibility within device |

## Cooperative Groups: Explicit Thread Coordination

CUDA 9+ introduced **cooperative groups**, a programming model extension that exposes explicit synchronization handles for subsets of threads at different granularities. This addresses a fundamental limitation of traditional CUDA: the only built-in synchronization primitives are `__syncthreads()` (within a thread block) and atomic operations (no barrier semantics).

Cooperative groups provide three levels of thread coordination:

1. **Thread block (CTA)**: The entire cooperative thread array, equivalent to `__syncthreads()`
2. **Tiled partition**: Sub-warp (<32) or warp-level (<16, <8, etc.) groups
3. **Grid group**: The entire grid, requiring cooperative launch

### Thread Block Synchronization

```cpp
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ void myKernel(float* data, int n) {
    cg::thread_block tb = cg::this_thread_block();

    // Do some work
    data[threadIdx.x] = compute(threadIdx.x);

    // Synchronize all threads in the block
    tb.sync();

    // Now all threads see the updated data
    float sum = 0.0f;
    for (int i = 0; i < 4; i++) {
        sum += data[threadIdx.x + i * blockDim.x];
    }
}
```

`tb.sync()` is functionally equivalent to `__syncthreads()` but provides a composable abstraction that can be nested with other cooperative group operations.

### Warp-Level Operations with Tiled Partitions

```cpp
__global__ void warpReduce(float* input, float* output) {
    cg::thread_block tb = cg::this_thread_block();
    cg::tiled_partition<32> warp(tb); // Split block into warps

    float local_val = input[threadIdx.x];
    float sum = local_val;

    // Warp-level reduction using shuffle-like operations
    for (int offset = warp.size() / 2; offset > 0; offset >>= 1) {
        sum += warp.shfl_down(local_val, offset);
    }

    // First thread in each warp writes result
    if (threadIdx.x % 32 == 0) {
        output[blockIdx.x] = sum;
    }
}
```

The `tiled_partition<32>` abstraction gives you warp-level primitives like `shfl_down()`, `shfl_up()`, and `shfl_xor()` that are much more efficient than shared memory for warp-local reductions.

**Common misconception**: Cooperative groups require cooperative launch (`cudaLaunchCooperativeKernel`). In reality, only **grid-level** coordination requires cooperative launch. `thread_block` and `tiled_partition` work in normal kernel launches.

### Coalesced Threads After Divergence

After a divergent branch, you can get only the active threads:

```cpp
__global__ void selectiveWork(float* data, int* mask) {
    cg::thread_block tb = cg::this_thread_block();

    if (mask[threadIdx.x]) {
        data[threadIdx.x] = expensive_compute(threadIdx.x);
    }

    // Get only the active threads (those that took the branch)
    auto active = cg::coalesced_threads();

    // Now operate only on active threads
    active.sync(); // Syncs only active threads, not the full block
}
```

This is more efficient than `tb.sync()` when the branch is highly skewed (only a few threads took the branch).

### Grid-Level Coordination (Advanced)

Grid-level synchronization requires a **cooperative launch**:

```cpp
// Host side
dim3 grid(4), block(256);
cudaLaunchConfig_t config = {
    .gridDim = grid,
    .blockDim = block,
    .dynamicSmemBytes = 0,
    .stream = 0,
    .coop = true // Enable cooperative launch
};

// Must check if device supports it
int maxThreads = 0;
cudaDeviceGetLimit(&maxThreads, cudaLimitMaxL2FetchGranularity); // Check support

cudaLaunchCooperativeKernel(&myKernel, config, args...);
```

Within the kernel:

```cpp
__global__ void gridCoopKernel(float* data) {
    cg::grid_group g = cg::this_grid();

    // All threads in the grid can now synchronize
    g.sync();

    // Proceed with grid-wide computation
}
```

**Common misconception**: Grid-level sync is like a supercharged `__syncthreads()`. In reality, it has strict requirements: the device must support cooperative launch, the total thread count must be within limits, and **all** threads in the grid must reach the barrier (no early exits).

## Common Misconceptions About Streams

❌ **Wrong**: Operations in different streams always run concurrently.
✓ **Correct**: They may serialize if the GPU is fully occupied or lacks sufficient hardware queues. HyperQ (Kepler+) allows up to 32 concurrent work queues, but actual concurrency depends on resource availability.

❌ **Wrong**: The default stream is the same as a user-created stream.
✓ **Correct**: The default stream (stream 0) synchronizes with all other streams. Any work in the default stream blocks until all other streams complete, and all other streams wait for the default stream.

❌ **Wrong**: `cudaMemcpy` only synchronizes the host thread.
✓ **Correct**: Synchronous `cudaMemcpy` blocks the host AND synchronizes with all GPU operations. Use `cudaMemcpyAsync` with explicit streams for overlap.

## Common Misconceptions About Cooperative Groups

❌ **Wrong**: Cooperative groups require cooperative launch.
✓ **Correct**: Only `grid_group` coordination requires cooperative launch. `thread_block` and `tiled_partition` work in normal kernels.

❌ **Wrong**: `__syncthreads()` and `thread_block::sync()` are different operations.
✓ **Correct**: They are equivalent — `thread_block::sync()` is just the cooperative groups abstraction of `__syncthreads()`.

❌ **Wrong**: `coalesced_threads()` works across blocks.
✓ **Correct**: It only works within a block, returning the active threads after divergence.

## Interview Patterns

CUDA streams and cooperative groups appear in these interview contexts:

1. **System design**: "Design a real-time inference pipeline that processes video frames. How do you overlap preprocessing, model inference, and post-processing?" Answer: Multi-stream pipeline with pinned memory buffers and event-based dependencies.

2. **Performance debugging**: "Your multi-stream kernel shows no speedup. What could be wrong?" Answer: Check for (1) default stream usage causing serialization, (2) insufficient chunk size to saturate the GPU, (3) pageable memory preventing async transfers, (4) kernel occupancy too low for true concurrency.

3. **Algorithm design**: "Implement a parallel reduction that scales across multiple SMs." Answer: Use multi-block reduction with `tiled_partition<32>` for warp-level reduction, then combine block results via atomic operations or a second reduction pass.

4. **Tradeoff analysis**: "When would you use cooperative groups vs. traditional shared memory + `__syncthreads()`?" Answer: Cooperative groups provides better composability for nested synchronizations, warp-level primitives for efficient reductions, and grid-level coordination for algorithms requiring global barriers (though the latter has strict launch requirements).
