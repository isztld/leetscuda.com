# GEMM Fundamentals & Tiling

General Matrix Multiply (GEMM) is the computational kernel that powers modern machine learning. Every transformer inference, every training step, every recommendation system ultimately reduces to matrix multiplications. Understanding GEMM from first principles—its arithmetic intensity, tiling strategies, and hardware utilization—is essential for any engineer working on AI infrastructure.

## Why this matters for interviews

GEMM questions are ubiquitous in ML infrastructure interviews because they test multiple dimensions of knowledge simultaneously:

- **Algorithmic thinking**: Can you derive the O(N) arithmetic intensity scaling from first principles?
- **Systems intuition**: Do you understand why naive implementations are memory-bound and how tiling fixes this?
- **Hardware awareness**: Can you reason about the memory hierarchy from registers to HBM?
- **Practical judgment**: When should you use cuBLAS vs. custom kernels vs. Tensor Cores?

Expect GEMM questions at companies building LLM serving systems (vLLM, TGI), AI training frameworks (PyTorch, JAX), or custom inference accelerators.

## The GEMM computation: C = αA×B + βC

The BLAS GEMM operation computes:

```
C = αA×B + βC
```

where:
- A is M×K
- B is K×N
- C is M×N
- α and β are scalar multipliers

### FLOPs count and arithmetic intensity

For a single output element C[i,j], we perform K multiplications and K-1 additions, totaling 2K-1 FLOPs. For the full M×N output matrix, we need:

**FLOPs = 2MNK** (one multiply + one accumulate per element)

For square matrices (M=N=K), this becomes **2N³ FLOPs**.

The memory traffic is straightforward: we read matrix A (MN elements), matrix B (KN elements), and write matrix C (MN elements). For square matrices, that's **3N² elements** or **12N² bytes** for single-precision float.

This gives us arithmetic intensity:

**Arithmetic Intensity = 2N³ FLOPs / 12N² bytes = N/6 FLOPs/byte**

For general matrices: **Arithmetic Intensity = 2MNK / (4(MK + KN + MN)) bytes**

This O(N) scaling is the key insight: as matrices grow, arithmetic intensity grows linearly. Small GEMM (N<256) remains memory-bound even on modern hardware. Large GEMM (N>2000) becomes compute-bound.

### Ridge point calculation

A kernel becomes compute-bound when its arithmetic intensity exceeds the hardware's ridge point:

**Ridge Point = Peak Compute (FLOPs/s) / Peak Memory Bandwidth (bytes/s)**

For H100 with BF16 Tensor Cores:
- Peak compute: 989 TFLOPS
- Memory bandwidth: 3.35 TB/s = 3,350,000 GB/s = 3,350,000,000,000 bytes/s
- Ridge point: 989 / 3,350 ≈ 295 FLOPs/byte

To reach compute-bound territory with square GEMM: N/6 ≥ 295, so **N ≥ 1,770**.

For A100 with BF16:
- Peak compute: 312 TFLOPS
- Memory bandwidth: 2.0 TB/s
- Ridge point: 312 / 2,000 ≈ 156 FLOPs/byte
- Square GEMM becomes compute-bound at **N ≥ 936**.

| System (Compute / Memory) | Arithmetic Bandwidth (TFLOPs/s) | Memory Bandwidth (TB/s) | Ridge Point (FLOPs/byte) |
|---------------------------|--------------------------------|------------------------|-------------------------|
| A100 80GB SXM BF16 TC | 312 | 2.0 | 156 |
| H100 SXM BF16 TC | 989 | 3.35 | 295 |
| H100 SXM FP8 TC | 1,979 | 3.35 | 592 |
| B200 BF16 TC | 2,250 | 8.0 | 281 |
| B200 FP8 TC | 4,500 | 8.0 | 562 |
| B200 FP4 TC | 9,000 | 8.0 | 1,125 |

## Three-level tiling hierarchy

Naive GEMM implementation is straightforward: each thread computes one output element C[i,j] by loading row i of A and column j of B, then performing a dot product. This approach streams through all of A and B for each output element—terrible memory efficiency.

High-performance GEMM uses a **three-level tiling hierarchy** that maps to the GPU memory structure:

### 1. Grid-level (Thread Block) Tile

Each thread block computes a BM×BN output tile by streaming over K in BK-sized steps. For H100, typical values are **BM=128, BN=128, BK=32**.

Memory traffic per block per K-step:
- Load A tile: 128×32×4B = 16KB
- Load B tile: 32×128×4B = 16KB
- Total: 32KB per K-step

With K=4096 and BK=32, we have 128 K-steps, loading **4MB total per matrix**. Compare this to the naive approach which would reload A and B 128×128=16,384 times!

The block tile lives in **shared memory** (L1 cache on Hopper), which has sub-microsecond latency compared to ~100ns for HBM.

### 2. Warp-level Tile

Within each thread block, each warp computes a WM×WN subtile. Typical H100 configuration: **WM=64, WN=64**.

With a 128×128 block tile and 64×64 warp tile, each block has 4 warps. Each warp uses **register blocking**: threads accumulate a TM×TN fragment, typically **8×8 per thread**.

With 32 threads per warp and 8×8 = 64 elements per thread, each warp accumulates 2,048 elements in registers. This maps cleanly to the warp-level matrix multiply operations.

### 3. Thread-level Tile

Each thread computes a TM×TN fragment using register accumulation. The thread-level tile size determines how many accumulators each thread maintains.

With **TM=8, TN=8**, each thread performs 8×8 = 64 multiply-adds per K-step within its assigned tile. This is small enough to fit comfortably in registers while providing enough parallelism to hide latency.

### Double buffering and async copies

On Hopper, the tiling hierarchy extends with **asynchronous global-to-shared copies**. Instead of blocking while loading the next K-tile, we:

1. Load K-tile i+1 into alternate shared memory buffer using async copy (`cp.async`)
2. Commit and fence the async operations
3. Compute on K-tile i while the hardware prefetches K-tile i+1
4. Swap buffers and repeat

This effectively hides global memory latency. H100's TMA (Tensor Memory Accelerator) automates this process further with dedicated hardware engines.

## CUDA 12.6 code example: Tiled SGEMM

Here's a minimal tiled SGEMM kernel demonstrating the three-level hierarchy:

```cpp
// Kernel configuration: BM=128, BN=128, BK=32, WM=64, WN=64, TM=8, TN=8
constexpr int BM = 128;
constexpr int BN = 128;
constexpr int BK = 32;
constexpr int WM = 64;
constexpr int WN = 64;
constexpr int TM = 8;
constexpr int TN = 8;

__global__ void sgemm_tiled(const float* A, const float* B, float* C,
                            int M, int N, int K) {
    // Shared memory tiles for A and B
    __shared__ float As[BM][BK];
    __shared__ float Bs[BK][BN];

    // Thread block computes C[tm, tn] output tile
    const int tm = blockIdx.y * BM;
    const int tn = blockIdx.x * BN;

    // Each thread accumulates a TM×TN fragment
    float accum[TM][TN] = {0.0f};

    // Thread coordinates within block
    const int tx = threadIdx.x;
    const int ty = threadIdx.y;

    // Iterate over K in BK-sized steps
    for (int k = 0; k < K; k += BK) {
        // Load A tile: each thread loads multiple elements coalesced
        #pragma unroll
        for (int i = 0; i < TM; i++) {
            int row = ty * TM + i;
            #pragma unroll
            for (int j = 0; j < BK; j += 4) {
                if (tm + row < M && k + j < K) {
                    As[row][j] = A[(tm + row) * K + (k + j)];
                    As[row][j + 1] = A[(tm + row) * K + (k + j + 1)];
                    As[row][j + 2] = A[(tm + row) * K + (k + j + 2)];
                    As[row][j + 3] = A[(tm + row) * K + (k + j + 3)];
                }
            }
        }

        // Load B tile similarly
        #pragma unroll
        for (int i = 0; i < BK; i++) {
            #pragma unroll
            for (int j = 0; j < TN; j += 4) {
                int col = tx * TN + j;
                if (k + i < K && tn + col < N) {
                    Bs[i][col] = B[(k + i) * N + (tn + col)];
                    Bs[i][col + 1] = B[(k + i) * N + (tn + col + 1)];
                    Bs[i][col + 2] = B[(k + i) * N + (tn + col + 2)];
                    Bs[i][col + 3] = B[(k + i) * N + (tn + col + 3)];
                }
            }
        }

        __syncthreads();

        // Compute GEMM on this K-tile
        #pragma unroll
        for (int kk = 0; kk < BK; kk++) {
            #pragma unroll
            for (int i = 0; i < TM; i++) {
                #pragma unroll
                for (int j = 0; j < TN; j++) {
                    float a = As[ty * TM + i][kk];
                    float b = Bs[kk][tx * TN + j];
                    accum[i][j] += a * b;
                }
            }
        }

        __syncthreads();
    }

    // Write result back to global memory
    #pragma unroll
    for (int i = 0; i < TM; i++) {
        #pragma unroll
        for (int j = 0; j < TN; j++) {
            int row = tm + ty * TM + i;
            int col = tn + tx * TN + j;
            if (row < M && col < N) {
                C[row * N + col] = accum[i][j];
            }
        }
    }
}
```

Key observations:
- **Vectorized loads**: `float4` loads (4 consecutive floats) improve memory bandwidth
- **Register tiling**: TM×TN = 8×8 = 64 accumulators per thread
- **Synch barriers**: `__syncthreads()` ensures all shared memory loads complete before computation

## Common misconceptions

### GEMM is always compute-bound

**Wrong**: GEMM is inherently compute-bound because it does O(N³) work.

**Correct**: Only large GEMM is compute-bound. Small GEMM (N<256 on H100) has arithmetic intensity below the ridge point and remains memory-bound. This is why inference with small batch sizes is memory-bound despite GEMM being "compute-heavy" in the asymptotic sense.

### Memory-bound GEMM cannot be improved

**Wrong**: If GEMM is memory-bound, there's nothing to optimize.

**Correct**: You can increase effective arithmetic intensity by:
- **Fusing operations**: Combine GEMM with bias add + ReLU into a single kernel
- **Batching**: Run multiple small GEMMs together to amortize memory overhead
- **Quantization**: Reduce memory traffic by using FP16/BF16/FP8 (also enables Tensor Cores)

### cuBLAS automatically uses Tensor Cores

**Wrong**: cuBLAS always maximizes performance by using Tensor Cores.

**Correct**: cuBLAS uses Tensor Cores only when:
- Data type is FP16, BF16, or TF32 (not FP32 native)
- Matrix dimensions align with Tensor Core requirements (multiples of 16)
- `CUBLAS_COMPUTE_16F` or `CUBLAS_TENSOR_OP_MATH` mode is set

For FP32 workloads on A100/H100, cuBLAS may still use CUDA Cores unless explicitly configured for TF32.

### Larger tiles always perform better

**Wrong**: Bigger BM, BN, BK values always give more overlap and better performance.

**Correct**: Tile sizes are constrained by:
- **Shared memory capacity**: 128×128×BK=32 requires 64KB per matrix, exceeding some SMs
- **Occupancy**: Larger tiles reduce register pressure but may reduce active warps
- **Memory alignment**: Vectorized loads require 16-byte alignment

The optimal tile size varies by GPU architecture: H100 benefits from larger tiles than A100 due to improved TMA throughput.

### One should implement GEMM from scratch

**Wrong**: Custom GEMM is always faster than library calls.

**Correct**: cuBLAS achieves ~95% of Tensor Core peak for large square GEMM. Custom kernels only win when:
- You need custom epilogues (activation fusion, quantization)
- Matrix dimensions are unusual (very narrow or very tall)
- You're building a library like CUTLASS that targets new architectures

## Interview patterns

### Whiteboard: Derive arithmetic intensity

**Question**: "What's the arithmetic intensity of SGEMM for square matrices? At what size does it become compute-bound on H100?"

**Expected answer**:
- FLOPs = 2N³, bytes = 12N²
- Arithmetic intensity = N/6 FLOPs/byte
- H100 ridge point = 989 TFLOPS / 3.35 TB/s = 295 FLOPs/byte
- Compute-bound when N/6 ≥ 295, so N ≥ 1,770

### Whiteboard: Design a tiling strategy

**Question**: "Design a tiling strategy for GEMM on H100. What tile sizes would you choose and why?"

**Expected answer**:
- **Block tile (BM×BN×BK)**: 128×128×32 fits in 64KB+64KB shared memory, provides good coalescing
- **Warp tile (WM×WN)**: 64×64 divides evenly into block tile, maps to warp scheduling
- **Thread tile (TM×TN)**: 8×8 gives 64 accumulators per thread, fits in register file
- Double buffering with async copies to hide global memory latency

### Coding: Implement naive GEMM, then optimize

**Question**: "Write a naive GEMM kernel, then optimize it with shared memory tiling."

**Progression**:
1. Naive: Each thread computes one output element, loads A and B from global memory
2. Tiled: Block loads BM×BK and BK×BN tiles into shared memory, computes BM×BN output tile
3. Optimized: Add vectorized loads, register blocking, double buffering

**Success criteria**:
- Naive version: ~50-100 GFLOPS (memory-bound)
- Tiled version: ~500-700 GFLOPS (70%+ of cuBLAS)
- Fully optimized: ~800+ GFLOPS (85%+ of cuBLAS)
