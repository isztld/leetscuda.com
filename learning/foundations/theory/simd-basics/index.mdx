# SIMD & Vectorisation

## Why this matters for interviews

SIMD is foundational for CPU performance engineering and provides the conceptual model for understanding GPU warp execution. Interviewers ask: "Why doesn't this loop auto-vectorize?" or "How would you implement a dot product using AVX2?" Understanding SIMD masking, alignment requirements, and the FMA throughput model is also essential for reading performance analysis of ML workloads on CPU inference engines (llama.cpp, ONNX Runtime).

SIMD (Single Instruction, Multiple Data) is a CPU architecture feature that applies one instruction to multiple data elements simultaneously. Modern CPUs can process 8 float32 values, 16 int16 values, or 4 float64 values with a single instruction via AVX2. Understanding SIMD is essential for writing high-performance CPU code and for understanding why GPU programming is designed the way it is — CUDA's warp model is essentially SIMD taken to its logical extreme.

## SIMD Registers and Instruction Sets

x86 CPUs have a hierarchy of SIMD instruction sets:

| ISA | Register width | Float32 lanes | Year |
|-----|---------------|--------------|------|
| SSE2 | 128-bit (XMM) | 4 | 2001 |
| AVX | 256-bit (YMM) | 8 | 2011 |
| AVX2 | 256-bit (YMM) | 8 | 2013 |
| AVX-512 | 512-bit (ZMM) | 16 | 2017 |

AVX2 is the baseline for modern AI/ML CPU code. AVX-512 doubles the lane count but is not universally supported (absent from AMD Zen 3, some Intel consumer chips).

A 256-bit YMM register holds 8 × 32-bit floats. An AVX2 addition operates on all 8 simultaneously:

```
ymm0: [a0, a1, a2, a3, a4, a5, a6, a7]
ymm1: [b0, b1, b2, b3, b4, b5, b6, b7]
       ↓
vaddps ymm2, ymm0, ymm1   ; ymm2 = [a0+b0, a1+b1, ..., a7+b7]
```

## Auto-Vectorisation

The simplest path to SIMD is letting the compiler do it. Modern compilers (GCC, Clang, MSVC) automatically vectorise simple loops when:
- The loop body is simple (no complex control flow)
- Loop iterations are independent (no loop-carried dependencies)
- The compiler can prove aliasing does not occur (pointers don't overlap)

```cpp
// This loop auto-vectorises with -O2 -mavx2
void add_arrays(float* __restrict__ c,
                const float* __restrict__ a,
                const float* __restrict__ b,
                int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];
    }
}
```

`__restrict__` tells the compiler that `c`, `a`, and `b` do not alias — without this, the compiler must assume writes to `c` may affect reads from `a` or `b`, preventing vectorisation.

Check whether a loop was vectorised with:
```bash
# GCC
g++ -O2 -mavx2 -fopt-info-vec myfile.cpp
# Clang
clang++ -O2 -mavx2 -Rpass=loop-vectorize myfile.cpp
# Also: inspect assembly with gcc -S or objdump -d
```

## Intel Intrinsics (Manual SIMD)

When auto-vectorisation fails or you need precise control, use **intrinsics** — C functions that map 1:1 to SIMD instructions:

```cpp
#include <immintrin.h>   // AVX/AVX2 intrinsics

void dot_product_avx2(const float* a, const float* b, int n, float* result) {
    __m256 sum = _mm256_setzero_ps();          // ymm: [0,0,0,0,0,0,0,0]

    int i = 0;
    for (; i <= n - 8; i += 8) {
        __m256 va = _mm256_loadu_ps(a + i);    // load 8 floats unaligned
        __m256 vb = _mm256_loadu_ps(b + i);
        sum = _mm256_fmadd_ps(va, vb, sum);    // sum += va * vb (fused multiply-add)
    }

    // Horizontal sum: reduce 8 lanes to 1 scalar
    __m128 lo = _mm256_castps256_ps128(sum);
    __m128 hi = _mm256_extractf128_ps(sum, 1);
    __m128 s  = _mm_add_ps(lo, hi);
    s = _mm_hadd_ps(s, s);
    s = _mm_hadd_ps(s, s);
    *result = _mm_cvtss_f32(s);

    // Handle remaining elements (tail loop)
    for (; i < n; i++) *result += a[i] * b[i];
}
```

Intrinsic naming convention: `_mm256_<op>_<type>` where `ps` = packed single (float32), `pd` = packed double (float64), `epi32` = packed 32-bit integer.

## FMA (Fused Multiply-Add)

FMA computes `a * b + c` in a single instruction with one rounding step instead of two. This:
- Increases arithmetic density (2 FLOPs per instruction instead of 1)
- Improves numerical accuracy (single rounding)
- Enables reaching peak FLOP/s: peak_flops = cores × frequency × FMA_units × 2 × SIMD_width

For AVX2 FMA on a single core at 3 GHz: `1 core × 3 GHz × 2 FMA units × 2 FLOPs/FMA × 8 lanes = 96 GFLOP/s`. This is the "roofline" compute peak per core.

## SIMD and Control Flow

SIMD cannot natively branch per lane. Conditional operations use **masking**:

```cpp
// Scalar: if (a[i] > 0) c[i] = a[i]; else c[i] = 0;

// SIMD equivalent using blend
__m256 va    = _mm256_loadu_ps(a + i);
__m256 zero  = _mm256_setzero_ps();
__m256 mask  = _mm256_cmp_ps(va, zero, _CMP_GT_OQ);  // mask where a > 0
__m256 vc    = _mm256_blendv_ps(zero, va, mask);      // select va or 0 per lane
_mm256_storeu_ps(c + i, vc);
```

This is exactly analogous to CUDA warp divergence: all lanes execute both branches, and the mask selects the result. Avoid complex branching in hot SIMD loops.

## Memory Alignment

SIMD loads and stores are fastest when the address is aligned to the SIMD register width:
- 128-bit (SSE): 16-byte aligned
- 256-bit (AVX): 32-byte aligned
- 512-bit (AVX-512): 64-byte aligned

Use `_mm256_load_ps` for 32-byte aligned data (faults if misaligned) and `_mm256_loadu_ps` for unaligned data (slightly slower). Allocate aligned memory with:

```cpp
float* buf = (float*)aligned_alloc(32, n * sizeof(float));
// or
alignas(32) float buf[1024];
```

## Practical Impact

AVX2 provides up to 8× throughput for float32 operations vs scalar code. Real-world speedup is often 4–6× due to memory bandwidth limits, loop overhead, and imperfect vectorisation. Combined with multithreading (OpenMP, std::thread), AVX2 + 96 cores can deliver ~10 TFLOP/s on a dual-socket server — within 2–5× of a data-centre GPU for specific workloads.

Understanding SIMD helps you read profiler output, understand why loops don't vectorise, and appreciate the design trade-offs behind GPU warp execution.

## Common misconceptions

❌ **Wrong**: "Auto-vectorization always works for simple loops"

✓ **Correct**: Auto-vectorization requires the compiler to prove several conditions: no pointer aliasing (use `__restrict__`), no loop-carried dependencies, and no complex control flow. Without `__restrict__`, the compiler must assume `a`, `b`, and `c` in `c[i] = a[i] + b[i]` could overlap — making vectorization unsafe. Always check with `-fopt-info-vec` (GCC) or `-Rpass=loop-vectorize` (Clang) to confirm vectorization happened.

❌ **Wrong**: "Unaligned SIMD loads (`_mm256_loadu_ps`) are significantly slower than aligned loads"

✓ **Correct**: On modern Intel CPUs (Haswell and later) and AMD Zen 2+, **unaligned loads have the same throughput as aligned loads** when the data happens to be aligned. The performance penalty only occurs on cache-line boundary crossings (when the 32 bytes span two 64-byte cache lines). Use `_mm256_loadu_ps` by default; only switch to aligned loads if you've verified the data is always 32-byte aligned.

❌ **Wrong**: "SIMD and GPU warp execution are fundamentally different"

✓ **Correct**: They are architecturally analogous. A GPU warp (32 threads) is SIMD with 32-wide lanes. Both execute a single instruction across multiple data elements simultaneously. Both handle conditional branches via masking — inactive lanes compute silently and their results are discarded. Understanding SIMD masking (blend operations) directly transfers to understanding warp divergence and predicated GPU instructions.

## Interview patterns

**Question 1**: "Why might a simple elementwise loop fail to auto-vectorize?"

**Answer**: Common reasons: (1) pointer aliasing — if `a`, `b`, `c` could overlap in memory, add `__restrict__`; (2) loop-carried dependency — if `a[i]` depends on `a[i-1]`, vectorization is unsafe; (3) function calls — most function calls prevent vectorization (use `__attribute__((const))` or inline); (4) non-unit stride — `a[i*stride]` may not vectorize. Check with compiler flags and fix one issue at a time.

**Question 2**: "What is the peak FLOP/s for an AVX2 FMA core at 3 GHz?"

**Answer**: `1 core × 3 GHz × 2 FMA units × 2 FLOPs/FMA × 8 float32 lanes = 96 GFLOP/s`. The 2 FMA units (on most Intel designs) allow two FMAs to issue per cycle. Each FMA computes `a*b+c` (2 FLOPs). With 8 lanes (AVX2 256-bit / 32-bit per float), total is 96 GFLOP/s per core. A 96-core dual-socket server can theoretically reach ~9 TFLOP/s FP32.

**Question 3**: "How does SIMD handle a conditional `if (a[i] > 0) c[i] = a[i]; else c[i] = 0`?"

**Answer**: SIMD cannot branch per lane. Use a **blend**: (1) compute comparison `mask = _mm256_cmp_ps(va, zero, _CMP_GT_OQ)` — produces a bitmask of which lanes satisfy `a[i] > 0`; (2) blend `vc = _mm256_blendv_ps(zero, va, mask)` — selects `va` where mask is 1, `zero` elsewhere. All lanes compute both "branches" and the mask selects the result. This is identical to GPU predicated execution.

## Key Concepts

- **SIMD**: one instruction operates on multiple data lanes simultaneously; AVX2 provides 8 float32 lanes per instruction.
- **Auto-vectorisation**: compiler transforms scalar loops to SIMD automatically; requires independent iterations and `__restrict__`.
- **Intrinsics**: C functions that map to SIMD instructions; use when auto-vectorisation fails or precision control is needed.
- **FMA**: fused multiply-add; computes `a*b+c` in one instruction with one rounding; doubles FLOP/s per instruction.
- **Masking**: conditional SIMD via blend operations; all lanes compute both branches, mask selects result.
- **Alignment**: AVX2 loads are fastest when addresses are 32-byte aligned.
