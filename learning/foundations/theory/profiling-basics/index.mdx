# Profiling GPU Code

## Why this matters for interviews

Profiling is the most critical practical skill for GPU performance engineering. Interviewers ask: "Your kernel is 5× slower than expected — how do you find the bottleneck?" Without a systematic profiling workflow (nsys → ncu → roofline → warp stalls), you're guessing. Understanding how to interpret roofline position and warp stall breakdowns is expected at senior and above.

Writing a correct CUDA kernel is the first step. Making it fast requires measurement. NVIDIA provides two primary profiling tools: **Nsight Systems** for system-level timeline analysis and **Nsight Compute** for kernel-level roofline and micro-architectural analysis. Using them together reveals whether your kernel is memory-bandwidth-bound, compute-bound, or suffering from latency penalties, warp stalls, or occupancy problems.

## The Two Tools

| Tool | Focus | Output |
|------|-------|--------|
| **Nsight Systems** (`nsys`) | System timeline: CPU, GPU, memory transfers, API calls | Timeline view (`.nsys-rep`) |
| **Nsight Compute** (`ncu`) | Per-kernel micro-architecture: roofline, memory throughput, warp stalls | Metric tables, roofline chart |

**Always start with Nsight Systems** to identify which kernel is the bottleneck and to see if CPU or PCIe transfer time dominates. Then use Nsight Compute on the specific bottleneck kernel.

## Nsight Systems — System Timeline

```bash
# Profile an entire application
nsys profile --trace=cuda,nvtx,osrt \
             --output=profile.nsys-rep \
             ./my_cuda_app

# Open in GUI
nsys-ui profile.nsys-rep

# Or generate a text report
nsys stats profile.nsys-rep
```

The timeline view shows:
- **CUDA API calls** on the CPU row (cudaMalloc, cudaMemcpy, kernel launches)
- **GPU kernels** with duration on the GPU compute row
- **Memory transfers** (H2D, D2H, D2D) on the DMA row
- **NVTX ranges** (custom annotations) for correlation with application logic

**NVTX annotations** mark regions of interest:

```cpp
#include <nvtx3/nvToolsExt.h>

nvtxRangePush("matrix_multiply");
matrix_multiply<<<grid, block>>>(A, B, C, N);
cudaDeviceSynchronize();
nvtxRangePop();
```

NVTX ranges appear colour-coded on the timeline, making it easy to see which application phase corresponds to which GPU activity.

### What to Look For in Nsight Systems

1. **Transfer gaps**: are there idle periods on the GPU while the CPU transfers data? Consider overlapping with streams.
2. **Serial kernels**: are multiple kernels executing sequentially when they could overlap in different streams?
3. **CPU bottleneck**: is the GPU idle because the CPU is preparing data or running Python? Profile the CPU row.
4. **Long cudaMalloc calls**: repeated `cudaMalloc` in the hot path adds latency; pre-allocate or use a memory pool.

## Nsight Compute — Kernel Deep Dive

```bash
# Profile a specific kernel with default metrics
ncu --target-processes all \
    --kernel-name my_kernel \
    --output profile.ncu-rep \
    ./my_cuda_app

# Open in GUI
ncu-ui profile.ncu-rep

# Or print to console (reduced metric set)
ncu --metrics sm__throughput.avg.pct_of_peak_sustained_elapsed,\
l1tex__t_bytes_pipe_lsu_mem_global_op_ld.sum.per_second \
    ./my_cuda_app
```

Nsight Compute replays the kernel **multiple times** (once per metric collection pass) which can make profiling slow for long-running kernels. Use `--kernel-name` and `--launch-skip` to narrow to the kernel you care about.

### Roofline Analysis

The **roofline chart** in Nsight Compute plots your kernel's achieved FLOP/s against its arithmetic intensity (FLOP/byte). Two roof lines bound performance:

- **Compute roof**: peak FLOP/s of the GPU (e.g., 989 TFLOPS FP16/BF16 on H100, 312 TFLOPS on A100)
- **Memory bandwidth roof**: peak_FLOP/s = peak_BW × arithmetic_intensity (e.g., 3.35 TB/s × intensity)

If your kernel sits below the memory bandwidth roof, it is **memory-bound** — increasing compute (more FMAs) won't help; you need to reduce bytes read or increase data reuse. If it sits below the compute roof, it is **compute-bound** — increase arithmetic intensity or reduce wasted arithmetic.

Most LLM decode kernels at batch size 1 are heavily memory-bound (arithmetic intensity < 1 FLOP/byte).

### Warp Stall Analysis

Warp stalls occur when a warp cannot issue an instruction. Common stall reasons:

| Stall reason | Cause | Fix |
|-------------|-------|-----|
| `Long Scoreboard` | Waiting for global memory load | Increase occupancy; prefetch; cache hits |
| `Short Scoreboard` | Waiting for shared memory or L1 | Reduce bank conflicts; pipeline loads |
| `No Instruction` | Warp has no ready instruction | Instruction cache miss; reduce register pressure |
| `Barrier` | Waiting at `__syncthreads()` | Minimise unnecessary barriers |
| `Memory Throttle` | Memory pipeline saturated | Reduce memory transactions |

```bash
# Collect warp stall metrics
ncu --metrics \
    smsp__average_warp_latency_due_to_long_scoreboard.ratio,\
    smsp__average_warp_latency_due_to_barrier.ratio \
    ./my_cuda_app
```

### Memory Throughput

Nsight Compute shows achieved vs peak memory bandwidth for global, shared, and L1/L2:

- **Global memory load throughput**: should approach HBM peak (3.35 TB/s for H100) for bandwidth-bound kernels
- **L1 hit rate**: high hit rate means data is reused within an SM; low hit rate means every access goes to L2 or HBM
- **Shared memory bank conflicts**: should be near 0 for optimised kernels

### Occupancy

Nsight Compute reports:
- **Theoretical occupancy**: maximum possible given kernel resources (registers, shared mem)
- **Achieved occupancy**: actual average warps active / max warps, measured during execution

Low occupancy (< 25%) is often a red flag. Check:
- Register usage with `nvcc -Xptxas -v` — high register counts limit warps per SM
- Shared memory allocation — large dynamic shared memory allocations limit block count

## Workflow Summary

```
1. nsys profile → find the slow kernel
2. ncu on that kernel → roofline: compute-bound or memory-bound?
3. If memory-bound:
   - Check global memory efficiency (coalescing)
   - Check L1/L2 hit rates (data reuse)
   - Check shared memory usage (tiling opportunities)
4. If compute-bound:
   - Check warp stalls (instruction-level bottlenecks)
   - Check Tensor Core utilization (use WMMA/cuBLAS for GEMM)
5. Iterate: change kernel, re-profile, verify improvement
```

## Common misconceptions

❌ **Wrong**: "Nsight Systems and Nsight Compute are interchangeable"

✓ **Correct**: They serve distinct purposes. **Nsight Systems** (`nsys`) profiles the whole application timeline — it shows which kernel is slow, whether the CPU is the bottleneck, and whether H2D/D2H transfers overlap with compute. **Nsight Compute** (`ncu`) dives into a single kernel's microarchitecture. Always start with `nsys` to identify the bottleneck, then use `ncu` on that specific kernel.

❌ **Wrong**: "High SM throughput means the kernel is well-optimized"

✓ **Correct**: A memory-bound kernel can show high SM activity while being far from its bandwidth ceiling. Look at the **roofline position**: if the kernel sits below the memory bandwidth roof, it's memory-bound regardless of SM utilization. The relevant metric is `l1tex__data_pipe_lsu_wavefronts_mem_global_op_ld.rate` vs peak HBM bandwidth, not just SM throughput.

❌ **Wrong**: "Nsight Compute gives accurate wall-clock timing"

✓ **Correct**: Nsight Compute introduces **3–10× replay overhead** by replaying the kernel multiple times to collect different hardware counter groups. Use **CUDA events** (`cudaEventRecord`, `cudaEventElapsedTime`) for accurate wall-clock timing, and use `ncu` only for identifying the nature of the bottleneck (memory vs compute) and relative comparisons between kernel variants.

## Interview patterns

**Question 1**: "Your matrix multiply is slower than cuBLAS. How do you debug it?"

**Answer**: First, `nsys profile` to confirm the kernel is the bottleneck (not data transfers). Then `ncu --set full` on the kernel: check roofline position (memory-bound or compute-bound?), SM throughput vs peak, warp stall breakdown (Long Scoreboard = memory-bound), and occupancy. For GEMM, common issues are: not using Tensor Cores, misaligned tile sizes, bank conflicts in shared memory staging, or insufficient occupancy for latency hiding.

**Question 2**: "Global Load Efficiency is 50%. What does this mean and how do you fix it?"

**Answer**: 50% efficiency means you're reading 2× more data than needed due to uncoalesced accesses. Each warp (32 threads) is accessing non-contiguous addresses, causing 2 memory transactions where 1 would suffice. Fix: ensure adjacent threads access adjacent memory addresses (stride = 1 in the innermost dimension). Check your thread index mapping — if threads access `input[row * N + col]` where `col` varies per thread, this is coalesced. If `row` varies per thread, this is strided and uncoalesced.

**Question 3**: "How do you reduce Long Scoreboard stall cycles?"

**Answer**: Long Scoreboard stalls mean warps are waiting for global memory loads. Strategies: (1) increase occupancy — more warps per SM means more warps available to issue while others wait; (2) use async prefetch (`cp.async`) to issue loads early and overlap with computation; (3) use shared memory tiling to load once and reuse many times, reducing global memory traffic; (4) vectorize loads (use `float4` instead of `float`) to reduce the number of load instructions.

## Key Concepts

- **Nsight Systems**: system-level timeline; reveals CPU/GPU overlap, transfer bottlenecks, kernel sequence.
- **Nsight Compute**: kernel-level deep dive; roofline, warp stalls, memory throughput, occupancy.
- **Roofline**: plots achieved FLOP/s vs arithmetic intensity to determine if a kernel is compute- or memory-bound.
- **Warp stall**: a warp cannot issue an instruction; `Long Scoreboard` (global memory miss) is the most common.
- **NVTX**: custom profiling annotations that appear on the Nsight Systems timeline.
- **Occupancy**: ratio of active warps to max warps per SM; limited by registers and shared memory.
