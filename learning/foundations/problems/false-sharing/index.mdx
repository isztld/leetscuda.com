## Background

Modern CPUs cache data in **cache lines** (typically 64 bytes). When two threads on different cores write to *different variables that share the same cache line*, the hardware's cache coherence protocol (MESI) forces that cache line to bounce between the cores' L1 caches on every write — even though the threads are not sharing data logically. This is **false sharing**, and it can degrade multi-core throughput by 10–100×.

The canonical example is an array of counters where each thread increments `counters[thread_id]`. If `sizeof(counter) < 64` and threads are adjacent in the array, their counters share a cache line. Every increment by thread 0 invalidates the cache line in thread 1's L1, and vice versa.

The fix is **padding**: ensure each thread's data occupies at least one full cache line. This is done by either:
- Aligning and sizing the struct to 64 bytes (`alignas(64)`)
- Interleaving padding bytes manually

## What to Implement

Implement two versions of a parallel accumulator:

```cpp
// Version A — exhibits false sharing (naive layout)
struct NaiveCounters {
    long long values[MAX_THREADS];
};

// Version B — eliminates false sharing (padded layout)
struct alignas(64) PaddedCounter {
    long long value;
    char pad[64 - sizeof(long long)];
};
struct PaddedCounters {
    PaddedCounter slots[MAX_THREADS];
};
```

Then implement the benchmark:

```cpp
// Returns elapsed milliseconds for T threads each incrementing their slot N times
double benchmark_naive(int T, long long N);
double benchmark_padded(int T, long long N);
```

Use `std::thread` and `std::chrono::high_resolution_clock` for timing. Each thread increments its own slot in a tight loop `N` times. The function returns total wall-clock time in milliseconds.

Finally, implement `speedup(int T, long long N)` that returns `benchmark_naive / benchmark_padded`.

## Constraints

- `1 ≤ T ≤ 64` (number of threads)
- `1 ≤ N ≤ 10^8` per thread
- `sizeof(PaddedCounter)` must equal 64
- The padded version must be at least 2× faster than naive for T≥4 and N=10^7

## Example

```
T=8 threads, N=10^7 increments each:
benchmark_naive(8, 1e7)  ≈ 850 ms   (false sharing thrashing)
benchmark_padded(8, 1e7) ≈  40 ms   (each thread in its own cache line)
speedup ≈ 21×
```

double benchmark_naive(int T, long long N) {
    NaiveCounters c;
    std::memset(&c, 0, sizeof(c));
    std::vector<std::thread> threads;
    auto t0 = std::chrono::high_resolution_clock::now();
    for (int tid = 0; tid < T; ++tid)
        threads.emplace_back([&c, tid, N]{ for (long long i = 0; i < N; ++i) c.values[tid]++; });
    for (auto& t : threads) t.join();
    auto t1 = std::chrono::high_resolution_clock::now();
    return std::chrono::duration<double, std::milli>(t1 - t0).count();
}

double benchmark_padded(int T, long long N) {

    std::memset(&c, 0, sizeof(c));

    for (auto& t : threads) t.join();
    auto t1 = std::chrono::high_resolution_clock::now();
    return std::chrono::duration<double, std::milli>(t1 - t0).count();
}

## Approach

The benchmark launches T threads, each incrementing its own counter N times, and measures total wall-clock duration. The only difference between the two versions is the data layout.

```cpp
double benchmark_naive(int T, long long N) {
    NaiveCounters c = {};
    std::vector<std::thread> threads;
    auto t0 = std::chrono::high_resolution_clock::now();
    for (int tid = 0; tid < T; ++tid)
        threads.emplace_back([&c, tid, N]{
            for (long long i = 0; i < N; ++i) c.values[tid]++;
        });
    for (auto& t : threads) t.join();
    auto t1 = std::chrono::high_resolution_clock::now();
    return std::chrono::duration<double, std::milli>(t1 - t0).count();
}
```

The padded version is identical except `c.slots[tid].value`. The `static_assert` verifies the layout is correct at compile time.

## Complexity

- **Time**: O(T × N) — each thread does N increments
- **Space**: O(T × 64) bytes for padded, O(T × 8) bytes for naive

## Key Insight

In the naive layout, 8 `long long` counters fit in one 64-byte cache line. When thread 0 writes `values[0]`, it invalidates the entire cache line in all other cores' L1 caches — even though threads 1–7 are writing different values. With padding, each counter occupies a full cache line, so writes never interfere.

## Common Mistakes

- **Wrong pad size**: `char pad[64 - sizeof(long long)]` = 56 bytes on 64-bit systems. Verify with `sizeof(PaddedCounter) == 64`.
- **Capturing tid by reference in the lambda**: Use `[&c, tid, N]` (tid by value). Capturing by reference means all threads share the same tid variable and may all read its final value.
- **Inconsistent timing scope**: Start the timer after thread creation is complete, or include thread creation in both benchmarks to keep the comparison fair.
