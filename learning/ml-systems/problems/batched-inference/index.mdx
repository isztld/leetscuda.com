## Background

LLM serving systems like vLLM and TensorRT-LLM use **continuous batching** (also called in-flight batching) to maximise GPU utilisation. Classic static batching waits until `B` requests arrive, processes them as a fixed batch, and only starts the next batch when all `B` requests finish. This leaves the GPU idle whenever requests have different output lengths.

Continuous batching solves this by treating each generation step independently. At every step, the scheduler selects a batch of active requests, runs one forward pass, and immediately replaces any finished requests with waiting ones. The batch size stays near-constant, the GPU stays busy, and P50/P99 latency drops significantly.

## What to Implement

Implement `ContinuousBatchScheduler` in C++:

```cpp
class ContinuousBatchScheduler {
public:
    ContinuousBatchScheduler(int max_batch_size, int max_seq_len);
    void add_request(int req_id, int prompt_len, int max_new_tokens);
    // Returns the batch of req_ids selected for the next step.
    // Preempts (moves to waiting) any request that would exceed max_seq_len.
    std::vector<int> schedule();
    // Mark a request as having generated one token (increments its token count).
    void step(int req_id);
    // Returns true if the request has finished (token count == max_new_tokens).
    bool is_done(int req_id) const;
    // Number of requests still running or waiting.
    int pending_count() const;
};
```

Scheduling policy:
1. Maintain a **running set** (at most `max_batch_size` requests) and a **waiting queue** (FIFO)
2. `schedule()` fills the running set from the waiting queue until full, then returns the running set
3. Any request in the running set whose `prompt_len + tokens_generated` would exceed `max_seq_len` is preempted back to the waiting queue
4. `step()` increments the token counter; if a request finishes, remove it from the running set

## Constraints

- `1 ≤ max_batch_size ≤ 64`
- `1 ≤ max_seq_len ≤ 8192`
- Request IDs are unique non-negative integers
- `schedule()` is always called before `step()` for each time step

## Example

```
scheduler(max_batch=2, max_seq=10)
add(req=0, prompt=3, max_new=4)   // finishes at step 4
add(req=1, prompt=5, max_new=3)   // finishes at step 3
add(req=2, prompt=1, max_new=2)   // waiting initially

schedule() → [0, 1]   // batch full
step(0); step(1)
schedule() → [0, 1]   // req 1 finishes after 3 more steps, req 2 fills in
```

---solution---
void add_request(int req_id, int prompt_len, int max_new_tokens) {
    waiting_.push({req_id, prompt_len, max_new_tokens, 0});
}

std::vector<int> schedule() {
    // Preempt overflowing requests
    auto it = running_.begin();
    while (it != running_.end()) {
        if (it->prompt_len + it->tokens_generated >= max_seq_len_) {
            waiting_.push(*it);
            it = running_.erase(it);
        } else {
            ++it;
        }
    }
    // Fill slots from waiting queue
    while ((int)running_.size() < max_batch_size_ && !waiting_.empty()) {
        running_.push_back(waiting_.front());
        waiting_.pop();
    }
    std::vector<int> batch;
    for (auto& r : running_) batch.push_back(r.req_id);
    return batch;
}

void step(int req_id) {
    for (auto it = running_.begin(); it != running_.end(); ++it) {
        if (it->req_id == req_id) {
            it->tokens_generated++;
            if (it->tokens_generated >= it->max_new_tokens)
                running_.erase(it);
            return;
        }
    }
}

    int num_adds;
    scanf("%d", &num_adds);

    ContinuousBatchScheduler sched(max_batch, max_seq);

    for (int i = 0; i < num_adds; i++) {
        int req_id, prompt_len, max_new;
        scanf("%d %d %d", &req_id, &prompt_len, &max_new);
        sched.add_request(req_id, prompt_len, max_new);
    }

    int steps = 0;
    while (sched.pending_count() > 0 && steps < 1000) {
        auto batch = sched.schedule();
        if (batch.empty()) break;
        for (int req_id : batch) sched.step(req_id);
        steps++;
    }

    printf("%d\n", steps);
    return 0;
}

---editorial---
## Approach

Continuous batching maintains two data structures: a **running set** (vector of active requests, bounded by `max_batch_size`) and a **waiting queue** (FIFO of pending requests).

**`schedule()`**:
1. Scan the running set and preempt any request whose `prompt_len + tokens_generated >= max_seq_len` — move it back to the front of the waiting queue
2. Fill empty slots from the waiting queue until running size == max_batch_size or queue is empty
3. Return the vector of req_ids in the running set

**`step(req_id)`**: find the request in running_, increment its token count, and remove it if done.

## Complexity

- **schedule()**: O(B + W) where B = max_batch_size, W = waiting queue size (both small in practice)
- **Space**: O(B + W)

## Key Insight

The difference from static batching is that `schedule()` is called at every generation step, not once per batch. Finished requests are replaced immediately, so the batch stays full and GPU utilisation stays high even when requests have different lengths.

## Common Mistakes

- **Checking `tokens_generated == max_new_tokens` instead of `>=`**: Use `>=` to avoid missing the termination condition if tokens_generated somehow overshoots.
- **Preempting after filling instead of before**: Preemption should happen first so that freed slots are available for new requests in the same `schedule()` call.
- **Not handling the waiting queue FIFO property**: Requests added earlier should be scheduled earlier. `std::queue` gives FIFO naturally.
