# Memory-Bound vs Compute-Bound Analysis

Understanding whether your kernel is memory-bound or compute-bound is the single most important diagnostic for performance optimization. This distinction determines which optimization strategies will actually move the needle—and which will waste your time.

## Why this matters for interviews

Principal-level interviews for AI infrastructure roles routinely include performance optimization questions. Interviewers expect candidates to:

1. **Quickly diagnose the bottleneck**: Given a kernel or performance problem, identify whether it is memory-bound or compute-bound using arithmetic intensity and the roofline model.
2. **Apply the right optimization**: Propose targeted fixes that address the actual bottleneck, not symptoms.
3. **Quantify the improvement**: Estimate achievable bandwidth utilization or FLOPS utilization before writing code.

The difference between a junior and senior engineer is often this: the junior engineer optimizes everything; the senior engineer optimizes only the bottleneck.

## The fundamental distinction

A kernel is **memory-bound** if its runtime scales primarily with memory access time. Specifically, runtime ≈ (bytes_transferred) / (memory_bandwidth). The limiting factor is the bandwidth between GPU RAM and the SMs' local caches—typically HBM or HBM2e on modern data center GPUs.

A kernel is **compute-bound** if its runtime scales primarily with the number of arithmetic operations. Specifically, runtime ≈ (FLOPs) / (arithmetic_bandwidth). The limiting factor is the throughput of CUDA Cores or Tensor Cores.

This distinction comes directly from the roofline model. The ridge point—the arithmetic intensity where the memory roof meets the compute roof—separates the two regimes:

| System (Compute / Memory) | Arithmetic Bandwidth (TFLOPs/s) | Memory Bandwidth (TB/s) | Ridge Point (FLOPs/byte) |
|---------------------------|--------------------------------|------------------------|--------------------------|
| A100 80GB SXM BF16 TC / HBM2e | 312 | 2.0 | 156 |
| H100 SXM BF16 TC / HBM3 | 989 | 3.35 | 295 |
| B200 BF16 TC / HBM3e | 2250 | 8.0 | 281 |
| H100 SXM FP8 TC / HBM3 | 1979 | 3.35 | 592 |
| B200 FP8 TC / HBM3e | 4500 | 8.0 | 562 |
| B200 FP4 TC / HBM3e | 9000 | 8.0 | 1125 |

Data from [arithmetic-intensity.md](/gpu-glossary/perf/arithmetic-intensity)

If your kernel's arithmetic intensity is **below** the ridge point, you are memory-bound. If it is **above**, you are compute-bound.

## Arithmetic intensity: the key metric

Arithmetic intensity is the number of floating-point operations performed per byte of memory transferred between DRAM and the compute units:

```
arithmetic_intensity = FLOPs / (bytes_read + bytes_written)
```

Common kernels and their arithmetic intensities:

| Kernel | FLOPs | Bytes Moved | Arithmetic Intensity | Scaling |
|--------|-------|-------------|---------------------|---------|
| SAXPY (y = ax + y) | 2N | 12N | 1/6 ≈ 0.167 | O(1) |
| SGEMM (C = A @ B + C) | 2N³ | 16N² | N/8 | O(N) |
| Single-Precision FFT | 5/2 N log(N) | 16N | (5/32) log(N) | O(log N) |

Source: [arithmetic-intensity.md](/gpu-glossary/perf/arithmetic-intensity)

The scaling behavior is critical. SAXPY has O(1) arithmetic intensity—it stays memory-bound regardless of problem size. SGEMM has O(N) arithmetic intensity—small matrices are memory-bound, but large matrices (N > 1770 on H100) become compute-bound.

### Real-world example: LLM inference

Large language model inference during the decode/generation stage is typically memory-bound. Consider a 500B parameter model at 16-bit precision (1 TB total). On a GPU with 10 TB/s memory bandwidth, you can load weights once every 100 ms. This sets a hard lower bound on intertoken latency—no amount of compute optimization helps until you increase batch size to raise arithmetic intensity.

During prefill/prompt processing, the same model may become compute-bound because weights are loaded once and reused across many tokens, increasing arithmetic intensity.

Source: [memory-bound.md](/gpu-glossary/perf/memory-bound), [compute-bound.md](/gpu-glossary/perf/compute-bound)

## Memory-bound optimization strategies

When your kernel is memory-bound, optimize these levers:

### 1. Coalesced memory access

Memory coalescing services multiple logical reads in a single physical DRAM burst. For a warp of 32 threads, perfectly coalesced accesses transfer 128 bytes in one transaction—enough for each thread to load one 32-bit float.

Non-coalesced access patterns (strided, irregular) force multiple physical transactions per warp, wasting bandwidth. A strided read kernel with stride=2 cuts throughput by 50%; stride=4 cuts it by 75%.

Example of coalesced vs strided access:

```cpp
// Coalesced: consecutive threads read consecutive elements
__global__ void coalesced_read(const float* __restrict__ in, float* out, size_t N) {
    size_t t = blockIdx.x * blockDim.x + threadIdx.x;
    if (t < N) out[t] = in[t];
}

// Strided: consecutive threads read far-apart elements (BAD)
__global__ void strided_read(const float* __restrict__ in, float* out, size_t N, int stride) {
    size_t t = blockIdx.x * blockDim.x + threadIdx.x;
    for (size_t j = t * stride; j < N; j += blockDim.x * gridDim.x * stride) {
        out[t] = in[j];  // Addresses differ by (stride * 4) bytes across warp
    }
}
```

Source: [memory-coalescing.md](/gpu-glossary/perf/memory-coalescing)

### 2. Shared memory tiling and reuse

Load data into shared memory once, reuse it multiple times. This eliminates redundant global memory reads. In matrix multiplication, each tile of input matrices is loaded into shared memory and then used for multiple output elements.

### 3. Reducing redundant loads

Cache frequently accessed data in registers or shared memory. Avoid reloading the same values from global memory across thread blocks or warps when the data fits in faster storage.

### 4. Operation fusion

Fusing multiple memory-bound kernels eliminates intermediate reads and writes. This is the most impactful optimization for LLM inference, where attention, softmax, and layer norm kernels are all memory-bound individually but can be fused into a single kernel with higher arithmetic intensity.

### 5. Quantization

Reducing data precision from FP16 to FP8 or INT4 halves or quarters memory traffic. For memory-bound workloads, this directly increases effective bandwidth and throughput.

### Bandwidth utilization target

Aim for **>80% bandwidth utilization** for memory-bound kernels. Compute it as:

```
bandwidth_utilization = achieved_bandwidth / peak_bandwidth
```

Nsight Compute reports this directly in the Memory Workload Analysis section.

## Compute-bound optimization strategies

When your kernel is compute-bound, optimize these levers:

### 1. Increase instruction-level parallelism (ILP)

Unroll loops and issue multiple independent instructions per thread to keep the scheduler busy. Modern SMs can issue multiple instructions per clock if there is enough independent work.

### 2. Reduce register pressure

High register usage limits occupancy, which in turn limits the number of warps available to hide latency. Use `__launch_bounds__` or register hints to control register allocation.

### 3. Use Tensor Cores

Tensor Cores deliver dramatically higher arithmetic bandwidth for matrix operations. On B200, FP4 Tensor Core arithmetic bandwidth is 9000 TFLOPS—45× higher than the H100 FP32 CUDA Core bandwidth of 200 TFLOPS. Switch to mixed-precision tensor operations whenever the algorithm allows.

### 4. Register blocking

Structure computation to maximize register reuse. In SGEMM, each thread computes a small tile of the output using only registers for intermediate values, minimizing spills to shared or global memory.

### 5. Dual-issue and warp-level primitives

Some operations can be dual-issued on modern SMs. Use warp-level primitives (__shfl_sync, __activemask) to avoid shared memory synchronization overhead.

## Identifying your bottleneck in practice

### Using Nsight Compute

Run Nsight Compute with full metrics:

```bash
ncu --set full --export report.ncu-rep -o report ./your_program
```

Key sections to examine:

1. **Roofline Analysis**: Shows your kernel's position on the plot. Left of ridge = memory-bound, right of ridge = compute-bound.
2. **Memory Workload Analysis**: Reports L1/global load/store bytes and achieved memory throughput.
3. **Warp State Statistics**: Shows stall reasons. Long scoreboard stalls suggest memory-bound; high issue queue occupancy suggests compute-bound.
4. **Compute Workload Analysis**: Shows instruction mix and pipe utilization. High ADU (arithmetic dispatch unit) utilization suggests compute-bound.

### Quick heuristics

- **Vector operations** (add, multiply, ReLU): Almost always memory-bound. They do O(1) FLOPs per element but O(1) memory accesses per element.
- **Matrix multiplication**: Memory-bound for small N, compute-bound for large N. The crossover point depends on your hardware's ridge point.
- **Attention mechanism**: Memory-bound for long sequences due to O(1/seq_len) arithmetic intensity. This motivates FlashAttention's tiling strategy to increase effective arithmetic intensity.

## Common misconceptions

❌ **A kernel can be both memory-bound and compute-bound simultaneously**

✓ **Reality**: A kernel hits only one roof at a time. It is either limited by memory bandwidth (left of ridge) or arithmetic bandwidth (right of ridge). The two roofs intersect at exactly one point—the ridge point. There is no "both" regime.

❌ **Kernel fusion always helps**

✓ **Reality**: Fusion can increase register pressure and reduce occupancy if not done carefully. Fusing too many operations may push a kernel over the register limit, reducing the number of concurrent warps and eliminating latency hiding. Always profile before and after fusion.

❌ **Fixing memory access patterns always speeds up a compute-bound kernel**

✓ **Reality**: Memory optimizations have no impact when compute is the bottleneck. If your kernel is already to the right of the ridge point, making memory access coalesced won't help—you need to increase arithmetic bandwidth (Tensor Cores) or reduce FLOPs (algorithmic improvement).

## Interview patterns

Expect these question types:

1. **"Here's a kernel. Is it memory-bound or compute-bound? How do you know?"**

   Calculate arithmetic intensity. Compare to ridge point. Cite specific formulas.

2. **"This kernel is slow. How would you profile it and what would you look for?"**

   Mention Nsight Compute, roofline position, memory throughput, stall reasons. Distinguish between Systems (concurrency) and Compute (per-kernel) profiling.

3. **"We need to speed up this LLM inference pipeline. Where do we start?"**

   Identify the bottleneck stage (decode is memory-bound, prefill is often compute-bound). Propose fusion for memory-bound stages, Tensor Cores for compute-bound stages.

4. **"Why is FlashAttention faster than standard attention?"**

   Standard attention writes O(N²) intermediates to HBM. FlashAttention tiles attention computation to fit in SRAM, eliminating the HBM write. This increases arithmetic intensity, moving the kernel closer to the compute roof.
