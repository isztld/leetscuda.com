# Shared Memory & Bank Conflicts

## Why this matters for interviews

Shared memory is the primary tool for crossing the gap between a kernel that saturates 10% of peak memory bandwidth and one that saturates 80%. Every tiling optimization — tiled GEMM, optimized reductions, fused attention — relies on it. Principal-level interviews ask "why is tiled matrix multiply faster than naïve?" and expect you to trace through the CGMA ratio improvement, the `__syncthreads()` requirement, the bank layout, and the padding fix, without hesitation. Bank conflict questions are among the most commonly tested numerical reasoning problems in CUDA interviews: given an access pattern, compute the conflict degree and the throughput penalty.

## What Shared Memory Is

Shared memory is a software-managed SRAM scratchpad that lives inside the Streaming Multiprocessor (SM), physically co-located with the L1 data cache. The key distinctions from every other memory tier:

- **Scope:** Shared between all threads in a *thread block*. Different blocks executing on different SMs each have independent, isolated copies. Threads from separate blocks cannot communicate through shared memory — not even if they run on the same SM.
- **Lifetime:** The block's lifetime. Data is lost when the block retires.
- **Latency:** ~4 cycles on a cache hit (no conflict), vs ~400 cycles for an HBM miss. L2 hits are ~200 cycles. Shared memory is one to two orders of magnitude faster than global memory.
- **Programmer-managed:** The hardware does not cache global memory loads into shared memory automatically. You explicitly load from global, use the data, and explicitly write results back.

**How to declare shared memory:**

```cuda
// Static allocation (at compile time)
__global__ void kernel(float* A, float* B, float* C) {
    __shared__ float tile_A[32][32];  // 32×32 floats = 4 KB
    __shared__ float tile_B[32][32];

    int tx = threadIdx.x, ty = threadIdx.y;
    // Load from global into shared
    tile_A[ty][tx] = A[blockIdx.y * 32 * K + ty * K + blockIdx.x * 32 + tx];
    tile_B[ty][tx] = B[(blockIdx.x * 32 + ty) * N + blockIdx.y * 32 + tx];
    __syncthreads();  // Barrier: all threads in block must reach here before any proceed
    // ... use tile_A, tile_B ...
    __syncthreads();  // Barrier before overwriting with next tile
}

// Dynamic allocation (size set at kernel launch)
extern __shared__ float smem[];  // no size in declaration
// launch: kernel<<<grid, block, shared_bytes_per_block>>>(...)
```

Static allocations are limited to 48 KB per block to maintain architectural compatibility across all devices. Dynamic allocation requires opting in above 48 KB:

```cuda
cudaFuncSetAttribute(kernel,
    cudaFuncAttributeMaxDynamicSharedMemorySize,
    163 * 1024);  // Up to 163 KB on A100 (164 KB - 1 KB reserved by CUDA)
```

## The Physical Layout: 32 Banks

Shared memory is physically organized into **32 banks**, each 4 bytes wide. This interleaved layout allows 32 independent simultaneous accesses — one per thread in a warp — at full bandwidth. The bank for a given 4-byte word is:

```
bank = (byte_address / 4) % 32
```

In other words, address 0 is in bank 0, address 4 is in bank 1, ..., address 124 is in bank 31, address 128 is back in bank 0. **Addresses that differ by 128 bytes map to the same bank.**

For a typical `float` array in shared memory:

```
__shared__ float data[64];
// data[0]  → bank 0  (byte 0)
// data[1]  → bank 1  (byte 4)
// ...
// data[31] → bank 31 (byte 124)
// data[32] → bank 0  (byte 128)  ← same bank as data[0]
// data[33] → bank 1  (byte 132)  ← same bank as data[1]
```

## The Bank Conflict Mechanism

A **bank conflict** occurs when two or more threads in the same warp access *different addresses that map to the same bank*. The hardware serializes these accesses, reducing throughput by the conflict degree.

**Conflict-free access (sequential elements):**

```cuda
float val = smem[threadIdx.x];
// Thread 0 → bank 0, Thread 1 → bank 1, ..., Thread 31 → bank 31
// All 32 threads hit different banks → 1 transaction
```

**32-way conflict (column access in row-major array):**

```cuda
__shared__ float tile[32][32];
// Thread t reads column c from row t: tile[t][0]
// tile[0][0] → bank 0  (byte 0)
// tile[1][0] → bank 0  (byte 128)   ← 32 bytes/row × 4 bytes/float = 128 bytes apart
// tile[2][0] → bank 0  (byte 256)
// ... all 32 threads hit bank 0 → 32 serialized transactions
```

A 32-way conflict means the 32 accesses that could complete in 1 cycle now require 32 cycles — a 32× throughput reduction. Nsight Compute reports this as **Shared Memory Bank Conflicts** (counter `l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_ld`); the ideal value is 0.

**2-way conflict (stride-2 access):**

```cuda
float val = smem[threadIdx.x * 2];
// Thread 0 → bank 0 (byte 0), Thread 1 → bank 2 (byte 8), ..., Thread 15 → bank 30
// Thread 16 → bank 0 (byte 128), Thread 17 → bank 2 (byte 136), ...
// Bank 0 is accessed by threads 0 and 16 → 2-way conflict
// Throughput halved
```

The general rule: stride `s` (in 32-bit words) causes a conflict degree of `gcd(s, 32)` when `s` divides 32, and is conflict-free when `gcd(s, 32) = 1`. Odd strides are always conflict-free.

## The Padding Fix

The canonical fix for column-access bank conflicts in 2D tiles is to **pad the inner dimension by 1 element**:

```cuda
// Conflicted: all column accesses hit bank 0
__shared__ float tile[32][32];     // 32 × 4 = 128 bytes per row → 128B period = bank repeat

// Conflict-free: column access shifts bank by 1 per row
__shared__ float tile[32][33];     // 33 × 4 = 132 bytes per row → no power-of-2 repeat
// tile[0][0] → bank 0  (byte 0)
// tile[1][0] → bank 1  (byte 132 → (132/4)%32 = 33%32 = 1)
// tile[2][0] → bank 2  (byte 264 → 66%32 = 2)
// ... tile[t][0] → bank t → all 32 threads hit different banks
```

The memory overhead of padding is one `float` per row — 4 bytes per row out of 128. For a 32×32 tile this adds 32 × 4 = 128 bytes out of 4096, a 3.1% increase. Negligible.

The same padding technique applies to any tile width that is a multiple of 32. If your tile is 16 wide, no padding is needed — each row is 16 × 4 = 64 bytes, and a column access hits banks 0, 2, 4, ..., 30, 0, 2, ... (2-way conflict). Padding to 17 makes the row 68 bytes, converting it to conflict-free.

## Tiling Pattern: Load–Sync–Compute–Sync

The canonical shared memory usage pattern for tiled GEMM (and any blocked algorithm) is:

```cuda
__global__ void tiledGEMM(const float* A, const float* B, float* C,
                           int M, int N, int K) {
    constexpr int TILE = 32;
    __shared__ float As[TILE][TILE];
    __shared__ float Bs[TILE][TILE + 1];  // padded to avoid bank conflicts

    int row = blockIdx.y * TILE + threadIdx.y;
    int col = blockIdx.x * TILE + threadIdx.x;
    float sum = 0.0f;

    for (int t = 0; t < K / TILE; ++t) {
        // Phase 1: coalesced load from global → shared
        As[threadIdx.y][threadIdx.x] = A[row * K + t * TILE + threadIdx.x];
        Bs[threadIdx.y][threadIdx.x] = B[(t * TILE + threadIdx.y) * N + col];
        __syncthreads();  // Barrier 1: ensure all loads complete before computation

        // Phase 2: compute using shared memory (no global traffic)
        for (int k = 0; k < TILE; ++k)
            sum += As[threadIdx.y][k] * Bs[k][threadIdx.x];
        __syncthreads();  // Barrier 2: ensure all threads done before next tile loads
    }
    if (row < M && col < N) C[row * N + col] = sum;
}
```

**Why the second `__syncthreads()` is mandatory:** without it, a fast thread could begin loading the next tile into `As`/`Bs` before a slow thread finishes reading the current tile, corrupting the data for the slow thread's computation.

**CGMA analysis:** Each thread reads 2×TILE floats (one row of A, one column of B) from global memory and performs 2×TILE FLOPs. CGMA = (2×TILE FLOPs) / (2×TILE bytes/4) = TILE FLOPs/float — exactly TILE. At TILE=32 that's 32 FLOPs per byte, vs 1 FLOPs/byte for naïve GEMM. On H100, the ridge point is ~295 FLOPs/byte, so TILE=32 still leaves you memory-bound, but it's 32× better than naïve.

## Shared Memory Capacity and Configuration

The L1 data cache and shared memory share the same physical SRAM pool per SM. The programmer controls the split via `cudaFuncSetAttribute`:

| GPU | Architecture | Combined pool | Max shared/SM | Max shared/block |
|-----|-------------|---------------|---------------|-----------------|
| H100 SXM5 | Hopper (cc 9.0) | 256 KB | 228 KB | 227 KB |
| A100 SXM4 | Ampere (cc 8.0) | 192 KB | 164 KB | 163 KB |
| RTX 3090 | Ampere (cc 8.6) | 128 KB | 100 KB | 99 KB |
| V100 SXM2 | Volta (cc 7.0) | 128 KB | 96 KB | 96 KB |

*Sources: Hopper Tuning Guide §1.4.2.4; Ampere Tuning Guide §1.4.2.3*

CUDA reserves 1 KB per thread block from the shared memory pool, leaving 227 KB usable per block on H100 and 163 KB on A100. Static `__shared__` declarations are capped at 48 KB without opt-in, regardless of hardware capacity. Exceeding 48 KB requires dynamic shared memory + `cudaFuncSetAttribute`.

Larger shared memory carveouts reduce the L1 cache available for global memory accesses. For kernels where both matter, profile with different carveout settings to find the optimum.

## Common misconceptions

❌ **Wrong**: Shared memory is automatically part of the L1 cache — data loaded to global memory is transparently available in shared memory.

✓ **Correct**: Shared memory and L1 cache share the same physical SRAM array but are accessed through distinct address spaces. The L1 caches global memory loads automatically (hardware-managed). Shared memory requires explicit programmer loads (`smem[i] = global[i]`). Data you write into shared memory is never automatically present in L1 or vice versa.

❌ **Wrong**: A bank conflict occurs when two threads read the *same address* in the same bank.

✓ **Correct**: Two threads reading the *same address* in the same bank is a **broadcast** — the hardware multicasts the value to all requesting threads in a single transaction. No conflict. A conflict requires *different addresses* that happen to map to the same bank. The hardware cannot serve them simultaneously because a bank has only one read port; it must serialize.

❌ **Wrong**: Shared memory is global to all blocks — any block can read data another block wrote.

✓ **Correct**: Each block has its own independent shared memory allocation. Two blocks cannot communicate through shared memory, even if they reside on the same SM. For inter-block communication, use global memory (with atomics or careful staging) or, on Hopper, Distributed Shared Memory across a thread block cluster.

❌ **Wrong**: Bank conflicts only occur when accessing columns — row access is always conflict-free.

✓ **Correct**: Conflicts arise from any access pattern where stride (in 32-bit words) is a multiple of 32. Column access in a 32-wide float array has stride 32 — a 32-way conflict. Stride-16 causes a 16-way conflict. Stride-2 causes a 2-way conflict. Only odd strides and stride-1 (sequential) are always conflict-free. A diagonal access pattern on a square array can also cause conflicts.

❌ **Wrong**: Padding shared memory arrays wastes significant memory and reduces occupancy.

✓ **Correct**: Padding a 32×32 float tile by one element per row adds 32 × 4 = 128 bytes, increasing the allocation from 4096 to 4224 bytes — a 3.1% increase. For a kernel where shared memory limits occupancy, this change is almost never the binding constraint. The throughput improvement from eliminating a 32-way conflict (32× serialization) always outweighs a 3% occupancy reduction.

❌ **Wrong**: You only need one `__syncthreads()` per tile in a tiled GEMM — after the loads.

✓ **Correct**: Two barriers are required: one after loading to ensure all threads see complete tiles before computation starts, and one after computation to ensure all threads finish reading the current tile before any thread begins loading the next. Omitting the second barrier introduces a race between the fast threads of the next iteration and the slow threads of the current one.

## Interview patterns

**Question**: "Why is tiled GEMM faster than naïve GEMM?"

**Answer**: In naïve GEMM, each thread independently reads its entire row of A and column of B from global memory. If M=N=K=1024, each thread reads 2×1024 floats = 8 KB from HBM. Total global reads: 1024² × 8 KB = 8 GB, when the data is only 12 MB. Tiled GEMM loads each element of A and B into shared memory *once per tile phase* and reuses it across all 32 (or 16, or 64) threads sharing that tile. CGMA goes from 1 to TILE, reducing HBM traffic by TILE×. The practical limit is that for H100 you need TILE ≥ 295 to reach the compute-bound regime; production CUTLASS kernels use multi-stage pipelining with tiles of 128-256 to approach this.

**Question**: "Given `__shared__ float tile[32][32]`, what is the bank conflict pattern when thread `t` reads `tile[t][0]`?"

**Answer**: Row `t`, column 0 is at byte offset `t * 32 * 4 = t * 128`. Bank = `(t * 128 / 4) % 32 = (t * 32) % 32 = 0` for all `t`. All 32 threads hit bank 0, different addresses. Result: 32-way bank conflict, 32× throughput penalty. Fix: `__shared__ float tile[32][33]`. Now `tile[t][0]` is at byte `t * 33 * 4 = t * 132`. Bank = `(t * 132 / 4) % 32 = (t * 33) % 32`. Since 33 % 32 = 1, bank = `t % 32`. Thread `t` hits bank `t` — 32 distinct banks, zero conflict.

**Question**: "How much shared memory can a single thread block use on an H100?"

**Answer**: 227 KB. The H100 has a 256 KB combined L1/shared pool per SM with a maximum shared carveout of 228 KB. CUDA reserves 1 KB per thread block, leaving 227 KB usable. This requires dynamic shared memory with `cudaFuncSetAttribute` setting `cudaFuncAttributeMaxDynamicSharedMemorySize` to the desired value; static declarations cannot exceed 48 KB without explicit opt-in.

**Question**: "What does `__syncthreads()` actually do and what happens if you forget it?"

**Answer**: `__syncthreads()` inserts a block-level barrier: no thread in the block proceeds past it until all threads have reached it. Under the hood, it flushes the shared memory write pipeline and ensures visibility. If you omit it after loading tiles, threads running ahead in the warp scheduler may begin computing with partially-written shared memory (data from a previous tile or garbage). The resulting error is a data race: non-deterministic wrong answers with no crash or CUDA error code. These bugs are silent and hard to find without `cuda-memcheck --check-shared-mem-race`.
