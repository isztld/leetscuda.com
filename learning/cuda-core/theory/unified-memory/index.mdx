# Unified Memory

## Why this matters for interviews

Unified Memory questions test whether you understand GPU memory management beyond the basic `cudaMalloc` + `cudaMemcpy` model. At companies running large-scale distributed training (Meta, Microsoft, Google) or building GPU infrastructure (NVIDIA, AWS), expect "when should you use `cudaMallocManaged` versus explicit transfers?", "what happens on a page fault?", and "how does Unified Memory behave on NVLink vs PCIe systems?" These are practical questions — Unified Memory is the default allocation model in many production codebases, but misusing it can destroy performance. This article gives you the precise mental model to answer them.

## Unified Memory Fundamentals

**Unified Memory** (introduced with compute capability 6.0+, Pascal architecture) provides a single pointer valid on both CPU and GPU. The CUDA runtime automatically migrates pages between host and device on demand, using GPU page faults to trigger migration on Linux with Pascal+.

**The key abstraction:** `cudaMallocManaged()` replaces the separate `cudaMalloc` + `cudaMemcpy` workflow. You allocate once, access from both host and device, and the runtime handles the movement.

```cpp
// Traditional approach
float *d_data, *h_data;
cudaMalloc(&d_data, N * sizeof(float));
cudaMemcpy(d_data, h_data, N * sizeof(float), cudaMemcpyHostToDevice);
// ... kernel ...
cudaMemcpy(h_data, d_data, N * sizeof(float), cudaMemcpyDeviceToHost);
cudaFree(d_data);

// Unified Memory approach
float *data;
cudaMallocManaged(&data, N * sizeof(float));
// ... initialize on host ...
// ... kernel uses data directly ...
// ... read results on host ...
cudaFree(data);  // Syncs automatically
```

## The Page Fault Mechanism

The critical detail: **Unified Memory is not magical — data still moves**. When a GPU first accesses a page (4KB) that is not resident on the GPU, it triggers a page fault. The CUDA runtime's **Migration Engine** then copies that page from host to device memory.

**The page fault penalty: ~10,000 cycles per page (4KB) on first access.** This is the cost of:
1. GPU trapping on the fault
2. Runtime detecting the fault
3. Copying the page from host to device over PCIe (or NVLink)
4. Updating page tables
5. Resuming the GPU

For a 1GB working set accessed sequentially, that's roughly 262,144 page faults × 10,000 cycles = 2.6 billion cycles of overhead. At a 1.5 GHz GPU clock, that's ~1.7 seconds of pure fault-handling latency — often more than the actual computation.

## Prefetching: Eliminating Page Fault Overhead

The solution: **`cudaMemPrefetchAsync()`** pre-migrates pages before they are accessed, eliminating the fault overhead.

```cpp
float *data;
cudaMallocManaged(&data, N * sizeof(float));

// Initialize on host
for (int i = 0; i < N; i++) {
    data[i] = i * 1.0f;
}

// Prefetch to GPU before kernel launch (eliminates page faults)
cudaMemPrefetchAsync(data, N * sizeof(float), device_id, 0);
cudaStreamSynchronize(0);  // Wait for prefetch to complete

// Kernel runs with no page faults
myKernel<<<grid, block, 0, 0>>>(data, N);

// Prefetch back to host for CPU read
cudaMemPrefetchAsync(data, N * sizeof(float), cudaCpuDeviceId, 0);
cudaStreamSynchronize(0);

// Now safe to read on host
for (int i = 0; i < N; i++) {
    printf("%f\n", data[i]);
}

cudaFree(data);
```

**With prefetching:** Migration happens asynchronously while the CPU continues work. The actual GPU kernel launch sees data already resident — no page faults, no 10,000-cycle penalties.

## Memory Advice Hints

CUDA provides `cudaMemAdvise()` to give the runtime hints about memory usage patterns:

```cpp
// Prefer memory to be allocated on the GPU
cudaMemAdvise(data, N * sizeof(float), cudaMemAdviseSetPreferredLocation, device_id);

// Indicate the GPU will access this memory (enables optimizations on NVLink systems)
cudaMemAdvise(data, N * sizeof(float), cudaMemAdviseSetAccessedBy, device_id);

// Revoke location preference (allow runtime to choose)
cudaMemAdvise(data, N * sizeof(float), cudaMemAdviseUnsetPreferredLocation, -1);
```

**Common advice patterns:**
- `cudaMemAdviseSetPreferredLocation`: Tells the runtime where you want the page allocated. Use `cudaCpuDeviceId` (-1) for CPU preference.
- `cudaMemAdviseSetAccessedBy`: Indicates which GPUs will access the memory. On NVLink systems, this enables zero-copy access without migration.
- `cudaMemAdviseSetReadMostly`: Hints that data is read-only from the GPU, enabling better caching policies.

## NVLink Systems and Zero-Copy

On systems with NVLink (H100, A100, DGX), the behavior changes significantly. NVLink (GPU-to-GPU, e.g., NVLink 4.0 on H100) provides up to 900 GB/s bidirectional bandwidth between GPUs. CPU-to-GPU NVLink (Grace Hopper Superchip) provides ~900 GB/s as well, compared to PCIe Gen5's ~128 GB/s or PCIe Gen4's ~64 GB/s.

**Key difference:** On NVLink systems, when you call `cudaMemAdviseSetAccessedBy(device_id)`, the runtime may leave the page on the CPU and allow the GPU to access it directly over NVLink. This is **zero-copy** — no migration occurs, but you pay the NVLink latency per access.

**NVLink vs PCIe comparison:**
- **PCIe systems:** Always migrate pages to GPU memory on first access (demand paging). Zero-copy is not supported.
- **NVLink systems:** Can keep pages on CPU and allow GPU direct access. Migration is optional based on access patterns.

**Interview insight:** On NVLink systems, Unified Memory can actually be *faster* than explicit `cudaMemcpy` for sparse access patterns where you don't need the entire dataset on the GPU. The runtime migrates only accessed pages, avoiding unnecessary transfers.

## Profiling Unified Memory

Nsight Compute and Nsight Systems provide specific metrics for Unified Memory:

- **Migration Engine activity:** Shows when and how much data is being migrated
- **UVM (Unified Virtual Memory) metrics:** Page fault counts, migration latency
- **GPU page faults:** Number of times the GPU faulted on a memory access
- **Prefetch efficiency:** What percentage of accesses were prefetched vs faulted

**Typical workflow:**
1. Run kernel without prefetching, capture Nsight Systems trace
2. Look for "Migration Engine" bars — these indicate page fault handling
3. Add `cudaMemPrefetchAsync()` before kernel launch
4. Re-run — Migration Engine activity should disappear or be dramatically reduced

## Common Misconceptions

**❌ Wrong:** Unified Memory is always slower than explicit transfers.
**✓ Correct:** With `cudaMemPrefetchAsync()`, Unified Memory can match explicit `cudaMemcpy` performance. The overhead is from page faults, not the Unified Memory mechanism itself. If you prefetch at the right time (e.g., while CPU is doing preprocessing), you can achieve zero-fault execution with simpler code.

**❌ Wrong:** `cudaMallocManaged` memory starts on the GPU.
**✓ Correct:** Memory allocated with `cudaMallocManaged` starts on the **CPU** until first GPU access or explicit prefetch. The first GPU access triggers a page fault and migration. This matters for latency-sensitive code — don't assume data is already on the GPU.

**❌ Wrong:** Unified Memory eliminates all data movement.
**✓ Correct:** Unified Memory **still moves pages** — it just automates the movement. The runtime migrates pages between host and device as needed. The difference is transparency: you write simpler code, but the data movement still happens (and still costs bandwidth and latency).

**❌ Wrong:** Unified Memory works identically on all systems.
**✓ Correct:** Behavior depends on the system's unified memory support level:
- **Full support (Pascal+ with UVM):** Demand paging via page faults, full `cudaMemPrefetchAsync` support
- **Limited support (Pre-Pascal):** Requires explicit `cudaMemPrefetchAsync` — no automatic migration
- **NVLink systems:** Can enable zero-copy access without migration

**❌ Wrong:** `cudaFree()` on managed memory is asynchronous.
**✓ Correct:** `cudaFree()` on managed memory **implicitly synchronizes** — it waits for all outstanding operations to complete before freeing. This is safe but can introduce hidden latency if you call it immediately after a kernel.

## When to Use Unified Memory

**Good use cases:**
- **Over-subscription:** Working set exceeds GPU memory. Unified Memory spills to host memory instead of OOM.
- **Simplified code:** Multi-GPU applications where memory needs to move between GPUs frequently.
- **Dynamic workloads:** Where memory requirements are not known at allocation time.
- **CPU-GPU cooperation:** When both CPU and GPU frequently access the same data structure.

**Bad use cases:**
- **Latency-sensitive inference:** Page fault penalties can destroy tail latency SLOs.
- **Small working sets that fit in GPU memory:** Explicit transfers are simpler and more predictable.
- **Very large batch training:** The overhead of managing many page faults may exceed the benefit of simpler code.

## Key Numbers to Memorize

| Property | Value |
|-----|---|
| Page size | 4KB |
| Page fault penalty (GPU) | ~10,000 cycles |
| Minimum Unified Memory support | Compute Capability 6.0 (Pascal) |
| Max managed memory | Limited by host + device RAM |
| Prefetch overhead | Negligible (async, overlaps with CPU work) |

**Bandwidth comparison for page migration:**
- **PCIe Gen4 x16:** ~16 GB/s (both directions)
- **NVLink 3.0 (H100):** 25 GB/s per direction
- **NVLink 4.0 (B200):** Up to 50 GB/s per direction

**Practical rule of thumb:** If your working set fits in GPU memory and access patterns are predictable, use explicit `cudaMemcpy`. If the working set is dynamic or exceeds GPU memory, use Unified Memory with prefetching.

## Interview Patterns

**"When would you use `cudaMallocManaged` versus `cudaMalloc` + `cudaMemcpy`?"**
If the working set fits comfortably in GPU memory and access patterns are predictable (e.g., fixed-size model weights), explicit transfers are simpler and more predictable. Use `cudaMallocManaged` when: (1) the working set may exceed GPU memory (over-subscription), (2) memory needs to move frequently between CPU and GPU (dynamic workloads), or (3) simplifying multi-GPU code where the same pointers need to be valid across devices. In all cases, use `cudaMemPrefetchAsync()` to avoid page fault penalties.

**"What happens when the GPU accesses a unified memory page that's on the CPU?"**
The GPU traps on a page fault. The CUDA runtime's Migration Engine copies the 4KB page from host to device memory over PCIe (or NVLink), updates the page tables, and resumes the GPU. This costs ~10,000 cycles per page — for a 1GB working set, that's ~262,000 page faults and potentially 1.7 seconds of overhead. The fix is `cudaMemPrefetchAsync()` to pre-migrate pages before access.

**"How does Unified Memory behave differently on NVLink vs PCIe systems?"**
On PCIe systems, pages are always migrated to GPU memory on first access (demand paging). On NVLink systems, the runtime can optionally keep pages on the CPU and allow the GPU to access them directly over NVLink (zero-copy). This is controlled by `cudaMemAdviseSetAccessedBy`. Zero-copy is beneficial for sparse access patterns where not all pages are needed on the GPU.

**"Is Unified Memory suitable for production inference workloads?"**
Generally no, unless the working set is known to fit in GPU memory and prefetching is used. The page fault mechanism introduces non-deterministic latency spikes that violate inference SLOs. For production inference, explicit `cudaMemcpy` with pinned memory is more predictable. Unified Memory is better suited for training workloads or development where simplicity outweighs tail latency concerns.
