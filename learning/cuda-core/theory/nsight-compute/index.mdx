# Nsight Compute: The Complete Profiling Workflow

## Why this matters for interviews

When your kernel is 10x slower than expected, how do you find the bottleneck? You could guess—maybe it's register pressure? Maybe it's memory coalescing? But guessing is expensive. Professional GPU engineers use a systematic profiling workflow with Nsight Compute.

In interviews, you'll be asked: "Your matrix multiply is memory-bound, how do you verify this?" or "How do you know if your kernel is compute-bound?" Nsight Compute gives you the data to answer these questions with confidence.

## What is Nsight Compute?

Nsight Compute (`ncu`) is NVIDIA's **per-kernel** GPU performance profiler. Unlike Nsight Systems (which shows a system-wide timeline), Nsight Compute dives deep into a single kernel, collecting:

- **Roofline positioning**: Where does your kernel sit on the arithmetic bandwidth vs memory bandwidth plot?
- **Memory throughput**: L1, L2, and HBM bandwidth utilization
- **Warp efficiency**: How well are warps utilizing SM resources?
- **Compute throughput**: Instruction-level performance metrics
- **Stall analysis**: Why are warps not issuing instructions?
- **Source correlation**: Map SASS instructions back to CUDA C++ source lines

### The guided workflow

Nsight Compute's "speedup bottleneck" analysis automates the optimization prioritization:

```
1. Run kernel with Nsight Compute
2. Look at "Speedup Bottleneck" section
3. Address the top bottleneck (e.g., "Memory: 80% of time spent waiting for loads")
4. Re-profile
5. Repeat until kernel is optimal
```

This is the same workflow used by CUDA experts at NVIDIA.

## CLI usage and basic workflow

### Basic profiling command

```bash
# Full metric collection (slow, but comprehensive)
ncu --set full --export report.ncu-rep -o report ./kernel_binary

# Quick profile (just timing)
ncu ./kernel_binary

# Profile specific kernel
ncu --launch-skip 0 --launch-count 1 ./kernel_binary

# Capture only memory metrics (faster)
ncu --metric l1tex__t_sectors_pipe_lsu_mem_global_op_ld.sum ./kernel_binary
```

### Key output sections

When you run `ncu` with `--set full`, you get several sections:

**1. Roofline Analysis**
```
Arithmetic Intensity: 0.125 FLOPs/byte
Memory Bandwidth: 1200 GB/s (75% of peak)
Position: Memory-bound region
```

This tells you: your kernel is memory-bound. No amount of compute optimization will help—you need to reduce memory traffic.

**2. Memory Workload Analysis**
```
Global Load Throughput: 1200 GB/s
L1 Hit Rate: 85%
L2 Hit Rate: 45%
Global Load Efficiency: 92%
```

Global Load Efficiency of 92% means you're getting 92% of the theoretical coalesced throughput. The remaining 8% is due to uncoalesced accesses or cache misses.

**3. Compute Workload Analysis**
```
SM Throughput: 45% of peak sustained
Active Cycles: 60%
Stall Cycles: 40%
```

SM Throughput of 45% is decent for a memory-bound kernel. If this were low (<20%), you'd have compute inefficiency on top of memory bottlenecks.

**4. Warp State Statistics**
```
Warp State Breakdown:
  Active: 60%
  Scoreboard: 25%
  Execution Backend: 10%
  Instruction Fetch: 5%
```

Scoreboard stalls (25%) indicate memory-bound behavior—warps are waiting for loads to complete.

**5. Source Correlation**
```
Line            SM Throughput    Stall Cycles
kernel.cu:42    50%              1000
kernel.cu:43    20%              4000  <- CUDA memory load here
kernel.cu:44    80%              500
```

This maps performance hotspots directly to source lines. The load on line 43 is the bottleneck.

## Key metrics and what they mean

### SM throughput metrics

**`sm__throughput.avg.pct_of_peak_sustained_elapsed`**

This is the fraction of time the SM is actually doing useful work. Peak sustained is what NVIDIA advertises (e.g., 30 TFLOPS).

- >70%: Excellent compute utilization
- 40-70%: Good for memory-bound kernels
- <30%: There's room for optimization

**Important**: High SM throughput doesn't mean the kernel is optimal. A memory-bound kernel can have high SM throughput while still being far from peak memory bandwidth.

### Memory throughput metrics

**`l1tex__t_bytes_pipe_lsu_mem_global_op_ld.sum`**

Total bytes loaded from global memory through L1. Compare this to the theoretical minimum (data size × access pattern) to compute efficiency.

**`l1tex__data_pipe_lsu_wavefronts_mem_global_op_ld.rate`**

Global load throughput in bytes/sec. Compare to peak HBM bandwidth (e.g., 1500 GB/s for A100).

**`lts__t_sectors_srcmap_tex_op_mem.sum`**

L2 throughput in sectors. This is the traffic that escapes L1 and goes to L2.

### Warp efficiency metrics

**Issue Efficiency**

Fraction of cycles where at least one warp was eligible to issue an instruction.

- 100%: Perfect latency hiding (always an eligible warp)
- 50-80%: Good for most kernels
- <50%: Increase occupancy or reduce instruction latency

**Global Load Efficiency**

```
Global Load Efficiency = requested_bytes / actual_transactions × 128 bytes
```

If you request 128 bytes but the hardware does 256 bytes in transactions, efficiency is 50%. This indicates uncoalesced accesses.

- 100%: Perfectly coalesced
- 75-99%: Good
- <75%: Check for strided or misaligned accesses

### Stall reason metrics

**Scoreboard stalls**

Warps waiting for data (memory loads, remote SM communication).

- High Scoreboard = memory-bound
- Fix: Increase occupancy, use async copies, prefetch

**Execution Backend stalls**

Warps waiting for functional units (ALU, SFU, Tensor Core).

- High Execution Backend = compute-bound
- Fix: Reduce instruction count, fuse operations

**Instruction Fetch stalls**

SM cannot fetch instructions fast enough.

- Rare in practice
- Fix: Reduce register pressure, improve instruction cache locality

## The replay overhead

Nsight Compute works by **replaying the kernel multiple times**, collecting different counter groups on each pass. This introduces overhead:

- **~3-10x slowdown** vs native execution
- **Not accurate for timing**—use CUDA events for wall-clock measurement
- **Accurate for relative comparisons**—fix A vs fix B on the same kernel

### When to use which metric set

```bash
# Quick profile (just basic metrics, single pass)
ncu --set basic ./kernel

# Standard profile (roofline, memory, basic warp stats)
ncu --set standard ./kernel

# Full profile (all counters, multiple passes, 3-10x slowdown)
ncu --set full ./kernel

# Custom metrics (only collect what you need)
ncu --metric sm__throughput.avg.pct_of_peak_sustained_elapsed \
     --metric l1tex__t_bytes_pipe_lsu_mem_global_op_ld.sum \
     ./kernel
```

For optimization workflow: use `standard` for quick iterations, `full` for deep analysis.

## Common misconceptions

❌ **Wrong**: "Nsight Compute measures exact wall-clock time"

✓ **Correct**: The replay overhead (3-10x) makes timing inaccurate. Use **CUDA events** or `cudaEventRecord` for wall-clock measurement. Nsight Compute is for **relative comparisons** (e.g., "fix A is 20% faster than fix B") and **bottleneck identification** (e.g., "memory-bound vs compute-bound").

❌ **Wrong**: "All metrics can be collected in one pass"

✓ **Correct**: Hardware counter limitations require **multiple replay passes** to collect all counter groups. This is why `--set full` is slower. If you only care about memory throughput, use `--metric` to collect just those counters and skip the replay overhead.

❌ **Wrong**: "High SM active time means the kernel is optimal"

✓ **Correct**: SM can be active but **underutilizing compute**. Look at the full picture:
- SM throughput: Is the SM saturated with useful work?
- Stall breakdown: Why are warps not issuing?
- Warp occupancy: Are there enough warps to hide latency?

A kernel with 80% SM throughput but 50% Scoreboard stalls is memory-bound and could benefit from more occupancy or prefetching.

## Practical optimization workflow

### Step 1: Establish baseline

```bash
# Measure baseline timing
cuda-event-timer ./kernel

# Profile with standard metrics
ncu --set standard --export baseline.ncu-rep -o baseline ./kernel
```

Check:
- Roofline position: memory-bound vs compute-bound?
- SM throughput: >40% for memory-bound, >70% for compute-bound?
- Global Load Efficiency: >75%?
- Stall breakdown: Scoreboard > 50%?

### Step 2: Identify top bottleneck

Look at the "Speedup Bottleneck" section:

```
Top Speedup Bottlenecks:
1. Memory Workload: 75% of cycles spent waiting for memory loads
2. Warp Divergence: 10% of warps diverged on branch
3. Register Pressure: 63 registers/thread limiting occupancy
```

Address them in order.

### Step 3: Apply fix and re-profile

**If memory-bound:**
- Improve coalescing (check stride in load pattern)
- Use shared memory tiling (reduce global memory reads)
- Increase vectorization (load4 instead of load1)
- Enable L2 prefetching (cudaMemPrefetchAsync)

**If compute-bound:**
- Fuse operations (reduce intermediate memory writes)
- Use Tensor Cores (for matrix operations)
- Reduce instruction count (common subexpression elimination)
- Increase arithmetic intensity (compute more per load)

**If warp divergence:**
- Restructure conditionals (if-else to predication)
- Sort data to reduce divergence
- Use warp-level primitives (warp ballot, shuffle)

### Step 4: Verify improvement

```bash
# Profile with same settings
ncu --set standard --export fix.ncu-rep -o fix ./kernel

# Compare metrics
ncu --compare baseline.ncu-rep fix.ncu-rep
```

Check:
- Did the bottleneck decrease?
- Did overall time decrease?
- Did you introduce new bottlenecks?

## Interview patterns

**Question 1**: "Your kernel is 10x slower than expected. How do you profile it?"

**Answer**: Use Nsight Compute with `--set standard` to identify the bottleneck. Check roofline position (memory-bound vs compute-bound), SM throughput, stall breakdown (Scoreboard = memory-bound), and warp efficiency. Fix the top bottleneck and re-profile.

**Question 2**: "How do you know if your kernel is memory-bound?"

**Answer**: Three indicators:
1. Roofline plot: positioned in the memory bandwidth limited region
2. Stall breakdown: >50% Scoreboard stalls
3. Global Load Efficiency: <100% (indicates uncoalesced accesses)

**Question 3**: "What's the difference between Nsight Compute and Nsight Systems?"

**Answer**: Nsight Compute is **per-kernel**, collecting hardware counter metrics (roofline, warp efficiency, stall reasons). Nsight Systems is **system-wide**, showing a timeline of kernel execution, memory transfers, and CPU activity. Use Systems to find concurrency issues, Use Compute to optimize individual kernels.

**Question 4**: "Global Load Efficiency is 50%. What does this mean?"

**Answer**: You're getting 50% of the theoretical coalesced throughput. The load pattern is uncoalesced—threads in a warp are accessing non-contiguous addresses. Fix by checking stride: adjacent threads should access adjacent memory (stride 1).

**Question 5**: "SM throughput is 80% but the kernel is still slow. Why?"

**Answer**: High SM throughput doesn't mean optimal. If the kernel is memory-bound (high Scoreboard stalls), you need to reduce memory traffic (shared memory tiling, vectorization). If it's compute-bound but throughput is high, you're near peak but limited by arithmetic intensity—try fusing operations.

## Key numbers reference

| Metric | Good | Bad | Action |
|--------|------|-----|--------|
| SM Throughput (memory-bound) | >40% | <20% | Increase occupancy |
| SM Throughput (compute-bound) | >70% | <50% | Reduce instruction count |
| Global Load Efficiency | >90% | <75% | Improve coalescing |
| Issue Efficiency | >75% | <50% | Increase ILP/occupancy |
| Scoreboard stalls | <30% | >60% | Memory-bound, add prefetch |

| Optimization | Expected speedup |
|--------------|-----------------|
| Fix coalescing (50% → 95% efficiency) | 1.5-2× |
| Shared memory tiling | 2-10× |
| Vectorized loads (char4 vs char1) | 2-4× |
| Operation fusion | 1.2-2× |
