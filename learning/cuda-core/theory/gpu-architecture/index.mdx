# GPU Architecture & the Streaming Multiprocessor

## Why this matters for interviews

GPU architecture questions appear at every principal-level AI infrastructure interview at NVIDIA, Google DeepMind, Meta AI, Microsoft Azure ML, and any company running large-scale model training or inference. Expect "how would you calculate the peak FLOPS of an H100?", "why can't you just use a faster CPU instead?", and "what limits the number of thread blocks that can run simultaneously?" These are not warmup questions — incorrect answers indicate you haven't worked close to metal. This article gives you the precise mental model to answer all of them.

## The CPU vs GPU Design Philosophy

CPUs and GPUs represent two solutions to two different problems. A CPU core is optimized for *latency*: it uses large multi-level caches (L1–L3 totaling tens of megabytes), branch predictors, out-of-order execution engines, and speculative execution hardware to minimize the time a single thread waits for any one instruction. A GPU SM is optimized for *throughput*: it dedicates most of its transistor budget to arithmetic units and runs thousands of threads concurrently, tolerating high per-thread latency by context-switching to another thread the instant one stalls.

The consequence: a single CPU thread running sequential code will execute faster than any single GPU thread. But when the same operation must be applied to millions of independent data elements — a matrix multiplication, a convolution, an attention score computation — the GPU's aggregate throughput wins by orders of magnitude.

| Property | CPU core (e.g., Intel Xeon 8592+) | GPU SM (H100) |
|----------|----------------------------------|---------------|
| Design goal | Minimize single-thread latency | Maximize aggregate throughput |
| Caches per core | ~3 MB L2 + shared L3 | 256 KB L1/shared SRAM |
| Execution model | Out-of-order, speculative | In-order SIMT, latency-hiding |
| Concurrent threads | 2 (hyperthreading) | 2048 (64 warps × 32 threads) |
| Context switch cost | Microseconds (OS-managed) | Zero cycles (hardware-managed) |
| Branch handling | Branch prediction, speculative exec | Predicate masking, serialization |
| Clock speed | 2–4 GHz | 1.1–2.0 GHz |

GPU context switches between warps cost *zero additional clock cycles*: the entire register state for all resident warps stays in the SM's on-chip register file. The warp scheduler simply picks a different eligible warp to issue next. CPU OS-level context switches, by contrast, require saving and restoring processor state and can take microseconds.

GPUs were originally graphics pipelines (rasterization, pixel shading) repurposed for general compute with CUDA, first released by NVIDIA in 2007. The arithmetic intensity of graphics — the same shader program running on every pixel — made this repurposing natural.

## The Streaming Multiprocessor

The **Streaming Multiprocessor (SM)** is the fundamental independent compute unit of an NVIDIA GPU. Think of it as the GPU's analog of a CPU core, but built for throughput rather than latency. Each SM contains:

- **CUDA Cores** — scalar FP32/INT32 arithmetic pipelines
- **Tensor Cores** — matrix-multiply-accumulate (MMA) units for mixed-precision GEMM
- **Warp schedulers** — hardware units that select eligible warps and dispatch instructions every cycle
- **Register file** — 256 KB of 32-bit registers, partitioned across all resident warps
- **Shared memory / L1 cache** — a configurable pool of on-chip SRAM shared between the data cache and programmer-managed shared memory

**Current SM counts on production hardware:**

| GPU | Architecture | SM Count | Compute Capability |
|-----|-------------|----------|-------------------|
| H100 SXM5 | Hopper | 132 | 9.0 |
| A100 SXM4 | Ampere | 108 | 8.0 |
| RTX 4090 | Ada Lovelace | 128 | 8.9 |

An SM is *not* a warp. An SM *contains* up to 64 warps. An SM is *not* a thread block — a block is assigned *to* an SM. Multiple blocks can reside on the same SM simultaneously if resources (registers, shared memory, warp slots) permit.

## Execution Resources Inside an SM

### CUDA Cores

A CUDA Core is a pipelined scalar execution unit that performs one floating-point or integer operation per clock cycle. Each H100 SM contains 128 CUDA Cores grouped into four processing blocks of 32 cores each. CUDA Cores execute the "wide" SIMT instruction stream: element-wise activations (ReLU, GELU), index arithmetic, reductions, and any scalar FP32/INT32 work that doesn't map to matrix MMA.

**Throughput:** 1 FP32 FMA (fused multiply-add) per cycle per core = 2 FLOPs per cycle per core (one multiply + one add counted separately).

H100 total CUDA Cores: 132 × 128 = **16,896**. At a ~1.98 GHz boost clock:
`16,896 cores × 2 FLOPs/cycle × 1.98 GHz ≈ 67 TFLOPS FP32`

### Tensor Cores

Tensor Cores, introduced with Volta (cc 7.0), are SIMD execution units that perform a matrix-multiply-accumulate in a single instruction. A *warp* cooperatively executes one MMA instruction; all 32 threads contribute fragments of the input matrices and collect output fragments.

Each generation has expanded the supported fragment sizes and precision types:

| Generation | GPU | Instruction | Matrix size (M×N×K) | Dtypes |
|-----------|-----|-------------|----------------------|--------|
| 1st | Volta (cc 7.0) | HMMA | 8×8×4 | FP16→FP16/FP32 |
| 2nd | Turing (cc 7.5) | HMMA | 16×8×16 | FP16/INT8 |
| 3rd | Ampere (cc 8.0) | HMMA | 16×8×16 | FP16/BF16/TF32/FP64/INT8 |
| 4th | Hopper (cc 9.0) | WGMMA | 64×N×16 (warpgroup) | FP16/BF16/FP8/TF32 |
| 5th | Blackwell (cc 10.0) | UMMA | 2-CTA tiles | FP16/BF16/FP8/FP4 |

**Why this matters for LLM inference:** A GPT-4-scale forward pass is almost entirely GEMM. Tensor Cores deliver ~989 TFLOPS BF16 on H100 vs ~67 TFLOPS FP32 via CUDA Cores — a ~15x gap. Any inference kernel that doesn't route through Tensor Cores (via cuBLAS, CUTLASS, or WMMA intrinsics) is leaving ~93% of peak throughput unused.

Tensor Cores are *not* invoked automatically by standard FP32 arithmetic. They require explicit MMA instructions, which are accessible through:
- `nvcuda::wmma` API (warp-level, deprecated path)
- CUTLASS (C++ templates for tile-level GEMM)
- cuBLAS `cublasGemmEx` with `CUBLAS_COMPUTE_16F` or `CUBLAS_COMPUTE_32F_FAST_TF32`

### Warp Schedulers and Instruction Dispatch

Each H100 SM contains **4 warp schedulers**. Each scheduler independently selects one eligible warp per cycle and issues its next instruction. A warp is *eligible* when all source operands are ready and no prior instruction is blocking the execution pipeline.

With 4 schedulers managing 64 warps (16 warps per scheduler), the SM can sustain 4 simultaneous instruction issues per cycle. This is the hardware mechanism behind latency hiding: if 48 warps are stalled waiting on L2 or HBM, the 4 schedulers find the 16 resident warps that have data ready and keep the arithmetic units fed.

**Dual-issue:** Hopper warp schedulers can issue two independent instructions to the same warp in one cycle if the instructions have no data dependency. This is transparent to the programmer.

**Issue efficiency** (measured by Nsight Compute as `smsp__issue_active.avg.pct_of_peak_sustained_active`) quantifies how often a scheduler actually issues each cycle. 100% means no stalls; low values indicate latency that isn't being hidden, usually from insufficient occupancy or memory pressure.

### Register File

Each SM has a **register file of 65,536 32-bit (4-byte) registers = 256 KB**. This is constant across Ampere and Hopper (confirmed in both tuning guides). The register file is *partitioned* at kernel launch: if each thread uses 32 registers and the block is 256 threads (8 warps), those 8 warps consume 256 × 32 = 8,192 registers. The SM can fit `65536 / 8192 = 8` such blocks concurrently — subject also to shared memory and warp count limits.

**Register spilling:** when `nvcc` assigns more registers per thread than the SM can hold, it spills the overflow to L1 cache or, worse, global memory (local memory in CUDA terminology). A spill to global memory incurs the same ~600-cycle latency as any cache miss — it is the fastest way to destroy occupancy-dependent latency hiding. `--ptxas-options=-v` at compile time prints spill statistics; any non-zero spill count warrants investigation.

### Shared Memory / L1 Cache

The L1 data cache and shared memory **share the same physical SRAM array** on the SM. The split is configurable at runtime via `cudaFuncSetAttribute(kernel, cudaFuncAttributePreferredSharedMemoryCarveout, carveout_percent)`.

H100 supports shared memory carveouts of 0, 8, 16, 32, 64, 100, 132, 164, 196, and **228 KB** per SM (from the Hopper tuning guide). The total pool is 256 KB. A100 supports up to **164 KB** shared memory from a 192 KB pool.

The distinction matters for programming:
- **Shared memory** is *software-managed*: the programmer explicitly loads data into it and controls its lifetime. Access pattern is user-determined.
- **L1 cache** is *hardware-managed*: the hardware caches global memory loads automatically. The programmer cannot directly control which addresses are cached.

Tiling algorithms (matrix multiply, convolutions) load tiles into shared memory to reuse them across many threads, avoiding redundant global memory fetches. The L1 is useful for irregular access patterns where tiling isn't possible.

## SM Counts and Throughput Arithmetic

Walk through from first principles so you can reproduce it at any interview:

**H100 SXM5 FP32 peak (CUDA Cores):**
```
132 SMs
× 128 CUDA Cores/SM
× 2 FLOPs/cycle (FMA = multiply + add)
× 1.98 GHz boost clock
= 66.9 TFLOPS ≈ 67 TFLOPS FP32
```
Note: per the Hopper Tuning Guide, H100 (cc 9.0) has 2× more FP32 operations per cycle per SM than A100 (cc 8.0). The 1.98 GHz figure is derived from the 67 TFLOPS spec using the above formula.

**A100 SXM4 FP32 peak (CUDA Cores):**
```
108 SMs × 128 CUDA Cores/SM × 2 FLOPs/cycle × ~0.765 GHz base clock
= 21.2 TFLOPS (NVIDIA reports 19.5 TFLOPS — reflects the base vs boost clock range)
```

**H100 BF16 Tensor Core peak:**
H100 uses 4th-gen Tensor Cores with warpgroup MMA (WGMMA). NVIDIA specifies **989 TFLOPS** BF16 (without sparsity). With 2:1 unstructured sparsity exploitation: 1,979 TFLOPS. These numbers come from NVIDIA's product spec and cannot be derived cleanly from public SM microarchitecture documentation without the internal WGMMA throughput spec.

**RTX 4090 FP32 peak:**
```
128 SMs × 128 CUDA Cores/SM × 2 FLOPs/cycle × 2.52 GHz boost
= 82.6 TFLOPS FP32
```

## What Happens When a Kernel Launches

A kernel launch `<<<gridDim, blockDim>>>` creates a *grid* of thread blocks. The CUDA runtime hands the block list to the **GigaThread engine** (the GPU's work distributor), which assigns blocks to SMs. Assignment follows these constraints:

1. Each block is assigned to exactly **one SM** and runs there to completion — blocks do not migrate.
2. Multiple blocks can reside on the same SM simultaneously, subject to three resource limits:
   - **Warp slots:** H100 max 64 warps/SM → max resident threads = 2048
   - **Register budget:** 65,536 registers/SM / (registers_per_thread × block_size)
   - **Shared memory budget:** 228 KB/SM (H100) / shared_mem_per_block
3. The limiting resource determines how many blocks fit per SM; the other resources are partially wasted.

**Full utilization requires block count > SM count.** If you launch 64 blocks on an H100 (132 SMs), 68 SMs sit idle. A well-tuned kernel launches enough blocks that the wave tail (the last partial wave of blocks) still approaches 100% SM utilization. For latency-sensitive inference kernels with small batch sizes, this wave quantization is a primary source of underutilization.

## Common Misconceptions

**❌ Wrong:** GPUs are always faster than CPUs for ML workloads.
**✓ Correct:** GPUs are faster for *data-parallel* workloads with enough arithmetic intensity. Sequential code, branchy algorithms, or small batch inference (batch size 1) often runs faster on CPUs due to lower latency per operation and no PCIe transfer overhead.

**❌ Wrong:** More CUDA Cores = better GPU performance.
**✓ Correct:** For ML workloads, Tensor Core throughput and memory bandwidth dominate. The RTX 4090 has more CUDA Core TFLOPS than an A100 (~83 vs ~20) but less HBM bandwidth and fewer Tensor Core TFLOPS — making the A100 faster for large-batch training despite the CUDA Core gap.

**❌ Wrong:** GPU threads are equivalent to CPU threads.
**✓ Correct:** A GPU thread has no independent instruction cache, no branch predictor, no out-of-order execution, and no OS scheduling. It is a single SIMD lane with a private register slot. 32 GPU threads (one warp) together consume the resources of a single CPU hardware thread.

**❌ Wrong:** SM count alone determines GPU performance.
**✓ Correct:** Memory bandwidth and compute density per SM both matter. An H100 SXM5 at 132 SMs with 3.35 TB/s HBM3 outperforms an RTX 4090 at 128 SMs with ~1 TB/s GDDR6X for memory-bound kernels by ~3×, despite similar SM counts.

**❌ Wrong:** All SMs execute the same instructions simultaneously.
**✓ Correct:** Each SM executes its own independent instruction stream. Different thread blocks running on different SMs may be at completely different points in the kernel. Only threads *within the same warp* execute in lockstep.

**❌ Wrong:** Shared memory is part of the SM's L1 cache.
**✓ Correct:** They share the same physical SRAM array but are accessed through different address spaces. Shared memory is software-managed (explicit loads by the programmer); L1 is hardware-managed (caches global memory automatically). The split between them is configurable per-kernel.

**❌ Wrong:** CUDA Core count is the primary GPU performance metric for ML.
**✓ Correct:** Tensor Core throughput is the dominant metric for GEMM-heavy workloads (training, inference for transformers). CUDA Cores handle element-wise ops, which are rarely the bottleneck.

**❌ Wrong:** A CUDA thread maps 1:1 to a CUDA Core.
**✓ Correct:** A *warp* of 32 threads issues to a set of CUDA Cores. The 32 threads in a warp execute the same instruction across 32 CUDA Cores simultaneously.

**❌ Wrong:** Tensor Cores automatically accelerate FP32 code.
**✓ Correct:** Tensor Cores require explicit MMA instructions. Standard `float` arithmetic uses CUDA Cores (FP32 pipelines). Using Tensor Cores requires cuBLAS, CUTLASS, or the WMMA API.

**❌ Wrong:** Warp context switching is expensive.
**✓ Correct:** Warp switching takes *zero additional cycles*. All register state for every resident warp is stored in the SM's register file simultaneously. The scheduler simply selects a different eligible warp's PC without any state save/restore.

**❌ Wrong:** The warp scheduler executes warps in round-robin order.
**✓ Correct:** Scheduling policy is a hardware heuristic, not round-robin. The scheduler prioritizes eligible warps (those with no outstanding data dependencies) and uses internal heuristics to balance throughput.

## Interview Patterns

**"How would you calculate the theoretical peak FLOPS of an H100?"**
Start with SM count (132), CUDA Cores per SM (128), FLOPs per cycle per core (2 for FMA), and boost clock (~1.98 GHz): `132 × 128 × 2 × 1.98e9 ≈ 67 TFLOPS FP32`. For Tensor Core TFLOPS, state the verified spec (989 TFLOPS BF16) and explain it uses warpgroup MMA (WGMMA) across 4 warps per tile. Then note that ML workloads achieve Tensor Core throughput only when accessing cuBLAS/CUTLASS — not from raw `float` arithmetic.

**"Why does a GPU have so many more 'cores' than a CPU?"**
GPUs sacrifice per-thread latency for aggregate throughput. A CPU core uses most of its die area on caches, branch predictors, and OOO execution hardware to run one thread as fast as possible. An SM uses most of its die area on arithmetic units and a large register file to run 2048 threads simultaneously, hiding memory latency by switching warps in zero cycles. For workloads with abundant data parallelism (matrix ops, convolutions), the throughput advantage massively outweighs the per-thread latency disadvantage.

**"What limits how many thread blocks can run simultaneously on an SM?"**
Three independent resource limits: (1) **warp slots** — H100 max 64 warps/SM, so a block of 256 threads consumes 8 warp slots and at most 8 such blocks fit per SM; (2) **register budget** — 65,536 registers/SM divided by (registers_per_thread × block_size); (3) **shared memory** — 228 KB/SM (H100) divided by shared memory per block. The *most constrained* resource determines actual occupancy. This is why `cudaOccupancyMaxActiveBlocksPerMultiprocessor` takes all three as inputs.

**"Why is it bad to have high register usage per thread?"**
High register usage reduces occupancy: fewer blocks fit per SM, meaning fewer warps available to hide latency. If you drop from 64 resident warps to 16, the warp schedulers have only 4 warps each to choose from. Any memory stall that all 4 happen to hit simultaneously causes the scheduler to stall. The practical consequence: kernels with register spilling are nearly always memory-bandwidth limited rather than compute-limited.
