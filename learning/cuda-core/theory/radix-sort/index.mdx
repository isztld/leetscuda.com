# Parallel Sorting (Radix Sort)

## Why this matters for interviews

Sorting is the canonical example of a non-trivial parallel algorithm that defies CPU intuition. At principal-level AI infrastructure interviews, questions like "how would you sort 1 billion integers on GPU?" or "why is radix sort faster than quicksort on GPU?" test whether you understand parallel algorithm design beyond textbook sequential algorithms. The radix sort answer reveals your grasp of: (1) non-comparison-based sorting, (2) the histogram + prefix scan + scatter pattern, and (3) the work-efficiency vs step-efficiency tradeoff. Incorrect answers (e.g., "use parallel quicksort" or "just use merge sort") signal you haven't worked close to GPU hardware.

## Why Comparison-Based Sorts Fail on GPU

Classic sequential sorting algorithms—quicksort, mergesort, heapsort—all rely on *comparison-based* decisions. Each element's position is determined by comparing it to a pivot (quicksort) or merging sorted subarrays (mergesort). These algorithms have inherent *control flow divergence*: different elements follow different execution paths based on comparison results.

On a GPU, this divergence is catastrophic:
- **Quicksort**: Partitioning is inherently sequential—each element's position depends on the pivot choice, which depends on previous elements. The recursive structure creates irregular memory access and deep control flow trees.
- **Mergesort**: Merging two sorted arrays requires comparing elements and choosing which to emit next. This is fundamentally sequential—output position i depends on all comparisons up to i.

Both have O(N log N) comparisons, but on GPU the parallel depth (number of sequential comparison rounds) is O(N) in the worst case, not O(log N). The *span* (critical path length) dominates.

**Radix sort sidesteps this entirely**: it is *non-comparison-based*. It processes keys digit-by-digit, performing stable counting sorts at each digit. No comparisons between elements are needed—only histogramming and scattering.

## The Radix Sort Algorithm

Radix sort processes integer keys one digit (radix) at a time, performing a *stable counting sort* for each digit. For 32-bit unsigned integers with 4-bit digits (base-16):

```
Pass 1: Sort by bits [0:3]   (least significant digit)
Pass 2: Sort by bits [4:7]
Pass 3: Sort by bits [8:11]
...
Pass 8: Sort by bits [28:31] (most significant digit)
```

The *stability* of each counting sort is critical: if two elements have the same digit at position d, their relative order from pass d-1 is preserved. After processing all digits from least to most significant, the entire array is sorted.

### Step 1: Histogram Per Block

Each thread block processes a contiguous chunk of the input and computes a *local histogram* of the current digit values:

```cpp
// Per-block histogram for 4-bit digits (16 bins)
__shared__ unsigned int block_hist[16];
if (threadIdx.x < 16) block_hist[threadIdx.x] = 0;
__syncthreads();

int digit = (input[tid] >> digit_pos) & 0xF;
atomicAdd(&block_hist[digit], 1);
__syncthreads();
```

With 256 threads and 16 bins, there's significant contention on the atomic adds. The optimization: use **privatization**—each thread maintains its own 16-bin histogram in registers, then merges at the end.

### Step 2: Exclusive Prefix Scan Over Histograms

Once each block has computed its local histogram, we need to know *where* in the output array each block's elements should go. This requires an *exclusive prefix sum* over the block histograms:

```
Block 0 histogram: [10, 5, 8, ...]  (10 elements with digit=0, 5 with digit=1, ...)
Block 1 histogram: [12, 3, 9, ...]
Block 2 histogram: [7, 11, 4, ...]

Block-level prefix sums (per digit):
Digit 0: Block 0 -> offset 0, Block 1 -> offset 10, Block 2 -> offset 22
Digit 1: Block 0 -> offset 0, Block 1 -> offset 5,  Block 2 -> offset 8
...
```

This is the *scan* phase. We scan across blocks for each digit value. This requires:
1. A per-digit block count array (16 values for 4-bit digits)
2. Exclusive prefix scan over those 16 values
3. Add the per-block offset to the per-digit offset

### Step 3: Scatter to Output

With the computed offsets, each thread can now determine its exact output position:

```cpp
int digit = (input[tid] >> digit_pos) & 0xF;
int block_offset = block_prefix[digit][blockIdx.x];
int digit_offset = global_prefix[digit];  // Where this digit starts globally
int out_idx = block_offset + digit_offset + local_rank_in_block;
output[out_idx] = input[tid];
```

The `local_rank_in_block` is the rank of this element *within its block* for this digit value. This requires each thread to know how many preceding elements in its block have the same digit—another prefix scan, but now *within the block*.

## The Two-Level Scan Pattern

The full GPU radix sort uses a **two-level scan** pattern:

**Level 1: Per-block scan** — each block scans its own elements locally, computing per-digit offsets within the block.

**Level 2: Grid-wide scan** — a single block (or kernel) scans the block-level histograms to compute global offsets for each block's digit counts.

For large arrays (N >> block_size), this is a **three-pass algorithm**:
1. Pass 1: Each block computes local histograms and scans locally
2. Pass 2: Merge per-block histograms into global histograms, scan globally
3. Pass 3: Scatter elements to output using computed offsets

## The Work vs Step Tradeoff

Radix sort exemplifies the **work-efficiency vs step-efficiency** tension in parallel algorithms:

- **Step complexity**: O(k) where k is the number of digits (8 passes for 32-bit integers with 4-bit digits)
- **Work complexity**: O(kN) = O(8N) for 32-bit integers

Compare this to parallel merge sort:
- **Step complexity**: O(log^2 N) (log N levels, each level needs O(log N) time to merge)
- **Work complexity**: O(N log N)

For large N, radix sort is *step-efficient* (constant number of passes) but *work-inefficient* (8N > N). For small N, the overhead of 8 passes dominates. The crossover point is typically around N = 10K–100K elements.

## Production Implementations: CUB and Thrust

Writing a production-grade radix sort from scratch is complex. The NVIDIA CUB library provides `DeviceRadixSort::SortKeys` and `DeviceRadixSort::SortPairs` that implement all the optimizations:

```cpp
#include <cub/cub.cuh>

void radixSort(unsigned int* d_in, unsigned int* d_out, int n) {
    void* d_temp_storage = nullptr;
    size_t temp_storage_bytes = 0;

    // Phase 1: Query required temp storage size
    cub::DeviceRadixSort::SortKeys(
        d_temp_storage, temp_storage_bytes,
        d_in, d_out, n);

    // Allocate temporary storage
    cudaMalloc(&d_temp_storage, temp_storage_bytes);

    // Phase 2: Actually sort
    cub::DeviceRadixSort::SortKeys(
        d_temp_storage, temp_storage_bytes,
        d_in, d_out, n);
}
```

Thrust provides a simpler interface:

```cpp
#include <thrust/sort.h>
#include <thrust/device_vector.h>

thrust::device_vector<unsigned int> d_vec(n);
// ... fill d_vec ...
thrust::sort(d_vec.begin(), d_vec.end());
```

Thrust automatically dispatches to CUB for large arrays and uses sorting networks for small arrays.

## Performance Numbers

| Dataset | GPU Throughput | CPU Comparison |
|---------|---------|------|
| 32-bit ints, 1M elements | ~3–5 GB/s | CPU std::sort ~200–500 MB/s |
| 32-bit ints, 100M elements | ~5–7 GB/s | CPU std::sort ~100–300 MB/s |
| 64-bit keys, 10M pairs | ~2–3 GB/s | CPU ~100–200 MB/s |

On H100, NVIDIA's optimized radix sort achieves ~95% of HBM bandwidth for large arrays (~3.2 GB/s out of 3.35 TB/s theoretical). The limiting factor is **memory bandwidth**, not compute—the algorithm is memory-bound because each pass reads and writes the entire array.

## Common Misconceptions

**❌ Wrong:** GPU sorting is always faster than CPU sorting.

**✓ Correct:** For small N (<10K elements), CPU sorting is faster due to launch overhead and better single-thread performance. A GPU kernel launch costs ~10–20 microseconds. Sorting 1000 integers on GPU requires 8 passes × 10μs = 80μs, while CPU can sort 1000 integers in ~5μs. The crossover is typically N = 10K–100K depending on the GPU.

**❌ Wrong:** Radix sort handles all data types.

**✓ Correct:** Radix sort works natively only on *fixed-width unsigned integers*. For signed integers, you need to flip the sign bit to make negative numbers sort before positive numbers. For floats, you need bit manipulation to handle NaN, signed zero, and the fact that float comparison is not the same as integer comparison. The standard trick: for positive floats, the bit pattern is already ordered; for negative floats, you need to flip all bits.

**❌ Wrong:** Larger digits (e.g., 8-bit) are always better because there are fewer passes.

**✓ Correct:** With 8-bit digits, you only need 4 passes for 32-bit integers, but each pass requires 256-bin histograms (vs 16 bins for 4-bit). This increases shared memory usage from 16 × 4 = 64 bytes to 256 × 4 = 1024 bytes per block, reducing occupancy. The optimal digit size depends on the tradeoff: fewer passes vs. more shared memory and more contention on atomic adds.

**❌ Wrong:** Radix sort is work-efficient because O(kN) = O(N) for constant k.

**✓ Correct:** O(kN) is work-efficient only if k is small enough that kN ≈ N. For k=8 (4-bit digits on 32-bit integers), you're doing 8× the work of a sequential sort. This is fine if k << log N (e.g., N = 1M, log N ≈ 20, so k=8 is reasonable). But for small N, the constant factor dominates.

## Interview Patterns

**"How would you sort 1 billion integers on GPU?"**

Answer: Use radix sort with 4-bit digits (8 passes). Break the array into blocks that fit in GPU memory (e.g., 100M per batch). Each batch: (1) per-block histograms, (2) block-level prefix scan, (3) global prefix scan over block counts, (4) scatter to output. For 1B integers, you'd need ~4GB of input storage plus ~4GB output = 8GB. If GPU memory is smaller, use multi-pass external sorting: sort chunks to GPU memory, write to disk, then merge.

**"Why is radix sort faster than quicksort on GPU?"**

Answer: Radix sort has O(k) parallel depth (constant number of passes) while quicksort has O(log^2 N) or worse due to partitioning and merging. Radix sort's operations (histogram, scan, scatter) are all highly parallel with no control flow divergence. Quicksort's partitioning requires comparing each element to a pivot, creating branches. On GPU, branches serialize across the warp, killing parallelism.

**"What's the bottleneck in GPU radix sort?"**

Answer: Memory bandwidth, not compute. Each pass reads N elements and writes N elements (2N memory ops). With 8 passes, that's 16N memory accesses. For 32-bit integers, 1B elements = 4GB. 16 × 4GB = 64GB total memory traffic. On H100 with 3.35 TB/s bandwidth, that's 64GB / 3.35 TB/s ≈ 19ms of pure memory time. The actual sort takes longer due to synchronization and atomic contention.

## Key Numbers to Remember

| Parameter | Typical Value | Impact |
|---|---|---|
| Digit size | 4 bits (16 bins) | 8 passes for 32-bit ints |
| Optimal digit size | 4–8 bits | Tradeoff: passes vs. shared memory |
| Block size | 256–512 threads | Larger = more throughput, more shared memory |
| Crossover N (GPU vs CPU) | 10K–100K elements | Below this, use CPU sort |
| H100 radix sort throughput | ~3–5 GB/s for 32-bit ints | ~95% of HBM bandwidth |
| Temp storage (CUB) | ~2× input size | 8GB for 1B ints |
