## Background

CUDA kernels run across thousands of GPU threads simultaneously. Each thread identifies itself using two built-in variables:

- `threadIdx.x` — the thread's position within its block
- `blockIdx.x` — the block's position in the grid

The standard 1D global index formula is:

```cpp
int i = blockIdx.x * blockDim.x + threadIdx.x;
```

Because the grid may not divide evenly into the array length, always guard with `if (i < n)` before accessing memory.

## What to Implement

Implement the `vecAdd` CUDA kernel that adds two float arrays element-wise:

```
c[i] = a[i] + b[i]   for every i in [0, n)
```

Each thread handles exactly one element. The host code allocates device memory, transfers data, and launches your kernel with 256 threads per block — you only need to write the kernel.

## Constraints

- `1 ≤ n ≤ 10,000,000`
- Elements are 32-bit floats in `[-10⁶, 10⁶]`
- 256 threads per block (set by the harness)

## Example

**Input:** `n = 4`, `a = [1.0, 2.0, 3.0, 4.0]`, `b = [10.0, 20.0, 30.0, 40.0]`

**Output:** `c = [11.0, 22.0, 33.0, 44.0]`

// Host wrapper — allocates device memory, copies data, launches kernel, copies result back.
// Do NOT modify anything below this line.
void solve(const float* h_a, const float* h_b, float* h_c, int n) {
    float *d_a, *d_b, *d_c;
    cudaMalloc(&d_a, n * sizeof(float));
    cudaMalloc(&d_b, n * sizeof(float));
    cudaMalloc(&d_c, n * sizeof(float));

    cudaMemcpy(d_a, h_a, n * sizeof(float), cudaMemcpyHostToDevice);
    cudaMemcpy(d_b, h_b, n * sizeof(float), cudaMemcpyHostToDevice);

    const int blockSize = 256;
    const int gridSize  = (n + blockSize - 1) / blockSize;
    vecAdd<<<gridSize, blockSize>>>(d_a, d_b, d_c, n);

    cudaMemcpy(h_c, d_c, n * sizeof(float), cudaMemcpyDeviceToHost);
    cudaFree(d_a);
    cudaFree(d_b);
    cudaFree(d_c);
}

## Approach

Each thread independently computes one output element: `c[i] = a[i] + b[i]`. The key steps are:

1. Compute the global thread index: `int i = blockIdx.x * blockDim.x + threadIdx.x`
2. Guard against out-of-bounds access with `if (i < n)`. The last block may have threads beyond the array length when `n` is not a multiple of `blockDim.x`.
3. Perform the addition and write to output

```cpp
__global__ void vecAdd(const float* a, const float* b, float* c, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        c[i] = a[i] + b[i];
    }
}
```

### Kernel Launch Configuration

The harness launches the kernel with:

```cpp
const int blockSize = 256;
const int gridSize = (n + blockSize - 1) / blockSize;
vecAdd<<<gridSize, blockSize>>>(d_a, d_b, d_c, n);
```

The grid size uses ceiling division. `(n + blockSize - 1) / blockSize` is the standard integer idiom for `ceil(n / blockSize)`, guaranteeing the grid covers all `n` elements even when `n` is not a multiple of 256. 256 threads per block is a practical sweet spot: too few threads can't hide memory latency through parallelism; too many can reduce the number of blocks that fit on a streaming multiprocessor simultaneously.

### Performance

This kernel is memory-bound, not compute-bound. Each element requires one addition but two reads and one write. The bottleneck is global memory bandwidth, not arithmetic throughput.

The standard indexing formula ensures **coalesced memory access**: adjacent threads (within a warp of 32) access adjacent memory addresses, so the GPU services an entire warp's reads in a single 128-byte transaction rather than 32 separate ones.

## Key Insights

1. **Bounds checking is non-negotiable**: when `n` is not a multiple of `blockDim.x`, the last block has threads with indices beyond the array. The `if (i < n)` guard prevents those threads from writing to invalid memory.

2. **Ceiling division is a universal pattern**: `(n + blockSize - 1) / blockSize` appears in virtually every CUDA kernel launch. Understand it once, use it everywhere.

3. **The indexing formula ensures correctness and performance simultaneously**: `blockIdx.x * blockDim.x + threadIdx.x` gives each thread a unique global index and keeps adjacent threads on adjacent memory addresses for coalesced access.

## Common Mistakes

- **Forgetting the bounds check**: for `n = 5` and `blockDim.x = 256`, block 0 spawns 256 threads but only indices 0–4 are valid. Without the guard, threads 5–255 write to arbitrary memory locations.

- **Using floor division for grid size**: `n / blockSize` truncates, so for `n = 513` and `blockSize = 256` you get 2 blocks covering only indices 0–511, silently skipping the last element.

- **Adding unnecessary synchronisation**: each thread works entirely independently, so `__syncthreads()` inside this kernel is redundant and wastes cycles.
