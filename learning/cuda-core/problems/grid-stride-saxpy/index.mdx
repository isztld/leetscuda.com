## Background

**SAXPY** is a fundamental operation from the BLAS (Basic Linear Algebra Subprograms) Level 1 specification. The name stands for **S**calar **A**nd **X**-**Y**, representing the computation:

```
y[i] = a * x[i] + y[i]   for every i in [0, n)
```

where `a` is a scalar multiplier, and `x` and `y` are arrays of equal length. This operation appears ubiquitously in scientific computing, from linear system solvers to neural network training (e.g., learning rate updates, gradient accumulation).

While a naive CUDA implementation might assign one thread per array element (as in the `vector-add` problem), this approach has a critical limitation: it requires a grid size that scales directly with the problem size. On a typical GPU with 128 SMs and a maximum of 1024 threads per block, the maximum grid contains at most 256 blocks (1D launch limit) to 65535 blocks in other dimensions. If your array has 10 million elements, you cannot launch a grid with 10 million threads.

The **grid-stride loop** pattern solves this scalability problem. Instead of each thread handling exactly one element, each thread processes multiple elements spaced by the total grid stride:

```cpp
int stride = gridDim.x * blockDim.x;
for (int i = blockIdx.x * blockDim.x + threadIdx.x; i < n; i += stride) {
    // Process element i
}
```

This pattern ensures:
1. **Scalability**: The same kernel works for any array size, regardless of grid capacity
2. **Load balancing**: Threads in the first half of the grid handle more elements when `n` is not a multiple of the grid size
3. **Coalesced memory access**: Adjacent threads still access adjacent memory addresses in each loop iteration
4. **Flexibility**: You can choose any convenient block size (e.g., 256 or 512 threads) without worrying about exact grid sizing

## What to Implement

Implement a CUDA kernel `saxpy` that performs the SAXPY operation using the grid-stride loop pattern:

```
y[i] = a * x[i] + y[i]   for every i in [0, n)
```

Your kernel must:
1. Use a grid-stride loop so each thread processes multiple elements when `n` exceeds the total thread count
2. Compute the starting index: `i = blockIdx.x * blockDim.x + threadIdx.x`
3. Compute the stride: `stride = gridDim.x * blockDim.x`
4. Loop through elements with increment `i += stride`
5. Guard against out-of-bounds access with `if (i < n)`

The host code will launch your kernel with a grid size that may be much smaller than `n`. For example, with 256 blocks of 256 threads each (total 65,536 threads) and `n = 10,000,000`, each thread will process approximately 152 elements on average.

## Constraints

- `1 <= n <= 100,000,000` (up to 100 million elements)
- Scalar `a` is a 32-bit float in the range `[-10^3, 10^3]`
- Arrays `x` and `y` contain 32-bit floats in the range `[-10^6, 10^6]`
- Use a block size of your choice (recommended: 256 threads)
- Do not modify the host setup code in `solve()`
- Your kernel must correctly handle cases where `n` is much larger than the total number of threads

## Example

**Input:** `n = 8`, `a = 2.0`, `x = [1, 2, 3, 4, 5, 6, 7, 8]`, `y = [10, 20, 30, 40, 50, 60, 70, 80]`

- `y[0] = 2 * 1 + 10 = 12`
- `y[5] = 2 * 6 + 60 = 72`

__global__ void saxpy(float a, const float* x, float* y, int n) {
    // Calculate the global thread index (starting position for this thread)
    int i = blockIdx.x * blockDim.x + threadIdx.x;

    // Calculate the stride (total number of threads in the grid)
    int stride = gridDim.x * blockDim.x;

    // Grid-stride loop: each thread processes multiple elements
    for (int idx = i; idx < n; idx += stride) {
        y[idx] = a * x[idx] + y[idx];
    }
}

## Approach

The grid-stride loop pattern is the canonical solution for processing arrays of arbitrary size with a fixed-size grid. The key insight is that instead of each thread handling exactly one element, each thread handles multiple elements spaced by the total grid stride.

**Grid-stride loop structure:**
```cpp
int i = blockIdx.x * blockDim.x + threadIdx.x;  // Starting index
int stride = gridDim.x * blockDim.x;            // Total threads in grid
for (int idx = i; idx < n; idx += stride) {
    // Process element idx
}
```

**Why this works:**
1. **Correctness**: Every index `0` to `n-1` is visited exactly once. Thread `t` in the grid processes indices `t, t + stride, t + 2*stride, ...`
2. **Scalability**: The same kernel works for `n = 100` or `n = 100M` without changing the launch configuration
3. **Coalesced access**: In each loop iteration, adjacent threads access adjacent memory locations, maximizing memory throughput

**SAXPY computation:**
The operation `y[i] = a * x[i] + y[i]` is a fused multiply-add (FMA). On modern NVIDIA GPUs, this executes as a single instruction with FP32 precision. The FMA is critical to numerical stability compared to separate multiply and add operations.

## Complexity

- **Time complexity**: O(n / (gridDim × blockDim)) — each thread processes approximately n/stride elements
- **Space complexity**: O(1) extra space beyond the input/output arrays
- **Memory throughput**: O(3n) — each element of x and y is read once, y is written once

## Key Insights

1. **Grid-stride decouples grid size from problem size**: You can choose any convenient block size (typically 128, 256, or 512) and grid size without worrying about exact divisibility or matching n.

2. **Stride calculation is critical**: Using `gridDim.x * blockDim.x` gives the total number of threads in the grid. This is the step size for each thread's loop.

3. **Load imbalance is bounded**: When n is not a multiple of stride, some threads process one more element than others. The difference is at most 1 element per thread.

4. **This pattern scales to 3D**: For 2D/3D problems, use the same pattern with multiple indices:
   ```cpp
   int strideX = gridDim.x * blockDim.x;
   int strideY = gridDim.y * blockDim.y;
   for (int x = blockIdx.x * blockDim.x + threadIdx.x; x < width; x += strideX) {
       for (int y = blockIdx.y * blockDim.y + threadIdx.y; y < height; y += strideY) {
           // Process (x, y)
       }
   }
   ```

5. **Coalesced access is preserved**: Even with multiple iterations per thread, adjacent threads access adjacent memory in each iteration, maintaining optimal memory throughput.

## Common Mistakes

- **Using `blockDim.x` as stride**: The stride must be `gridDim.x * blockDim.x` (total threads), not just `blockDim.x` (threads per block). Using `blockDim.x` would cause threads in different blocks to process overlapping elements.

- **Off-by-one in loop condition**: Use `idx < n`, not `idx <= n`. The last valid index is `n - 1`.

- **Forgetting the bounds check**: The loop condition `idx < n` serves as the bounds check. Do not add an extra `if (idx < n)` inside the loop body — it's redundant and may confuse the compiler's vectorization.

- **Using the wrong variable in the increment**: Use `idx += stride`, not `i += stride` (unless you've named your loop variable `i`). The loop increment must use the same variable as the loop condition.

- **Launching with gridDim = 1**: While technically valid, this minimizes parallelism. A typical configuration uses 128-512 blocks for medium-sized arrays, scaling up for larger arrays.

## When to Use Grid-Stride Loops

| Scenario | Recommended |
|----------|-------------|
| Array size known at compile time and fits in grid | Optional (single-assignment is fine) |
| Array size unknown until runtime | **Required** |
| Processing multiple arrays of different sizes | **Required** |
| Building reusable library kernels | **Required** |
| Very small arrays (n < 1024) | Optional (may prefer 1:1 mapping) |

For production code, prefer the grid-stride pattern even when a 1:1 mapping would work. It future-proofs your kernel against larger inputs and is the idiomatic CUDA pattern for most algorithms.
