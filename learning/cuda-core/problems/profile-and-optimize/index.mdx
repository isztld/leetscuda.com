## Background

**Performance profiling** is the systematic process of measuring where a GPU kernel spends its time and why. Without profiling, optimization is guesswork. With profiling, you can identify the single bottleneck that, when fixed, yields the biggest speedup.

The **roofline model** provides the theoretical framework: a kernel's performance is bounded by either (1) memory bandwidth or (2) arithmetic bandwidth. The **arithmetic intensity** (FLOPs/byte) determines where a kernel sits on this model. A memory-bound kernel needs fewer FLOPs per byte moved; a compute-bound kernel needs more.

**Nsight Compute** is NVIDIA's per-kernel profiler that measures:
- **Roofline positioning**: Where does your kernel sit relative to the memory bandwidth roof and arithmetic bandwidth roof?
- **Memory workload**: What fraction of HBM bandwidth is achieved? What's the L1 hit rate?
- **Warp efficiency**: What fraction of cycles are wasted on stalls, divergence, or idle warps?
- **Stall reasons**: Why isn't the GPU doing more work? (Long scoreboard, memory throttle, execution dependency, etc.)

The canonical optimization workflow:
1. **Profile** with `ncu --set full` to get baseline metrics
2. **Identify the bottleneck** using roofline position and warp efficiency
3. **Apply a targeted fix** (coalesce memory accesses, increase occupancy, fuse operations)
4. **Re-profile** to measure improvement and confirm the bottleneck moved

A classic mistake: **non-coalesced memory access**. When threads in a warp access strided memory (e.g., `data[threadIdx.x * stride]` instead of `data[threadIdx.x]`), the GPU must perform multiple memory transactions instead of one. This can reduce global load efficiency from 100% to 10% or worse.

Understanding this workflow is essential for engineering interviews at NVIDIA, Meta, and AI infrastructure teams. They don't just want to know that you can write CUDA kernels; they want to know that you can **make them fast**.

## What to Implement

You are given a kernel that computes the **row-wise maximum** of a 2D matrix. The initial implementation has a critical performance bug: it accesses memory in a **non-coalesced pattern** that causes terrible memory throughput.

**The buggy kernel** (you'll fix it):
```cpp
__global__ void rowMaxBroken(const float* matrix, float* results, int rows, int cols) {
    int row = blockIdx.x;
    int tid = threadIdx.x;

    // BUG: Each thread in the warp accesses strided locations
    // Thread 0: matrix[row * cols + 0], matrix[row * cols + 32], ...
    // Thread 1: matrix[row * cols + 1], matrix[row * cols + 33], ...
    // This creates 32 separate memory transactions instead of 1 coalesced transaction
    float maxVal = -INFINITY;
    for (int col = tid; col < cols; col += 32) {
        float val = matrix[row * cols + col];
        if (val > maxVal) maxVal = val;
    }

    // Write result (only thread 0 per block)
    if (tid == 0) results[row] = maxVal;
}
```

**The fix**: Transpose the access pattern so that adjacent threads read adjacent memory:
```cpp
// For row-wise reduction, you cannot fix this without a different algorithm
// The correct approach is to use a proper reduction within each row
```

Actually, for row-wise maximum of a matrix, the coalesced approach is:
- Each thread block handles one row
- Threads load **contiguous** chunks of the row (not strided)
- Use a tree reduction within the block to find the max
- This ensures **100% global load efficiency** on the initial load

**Your task**:
1. Start with the broken version and run it
2. Profile it: `ncu --set full ./program`
3. In the Nsight Compute report, find:
   - **Memory Workload Analysis**: global load efficiency will be terrible (~3-10%)
   - **Roofline Analysis**: you'll see the kernel at the memory bandwidth roof
   - **Warp State Statistics**: look for "Long Scoreboard" stalls (waiting for memory)
4. Fix the kernel to use coalesced access
5. Re-profile to confirm global load efficiency > 90%

**Harness Approach**

The judge verifies your kernel by:
1. Generating a deterministic matrix of known values
2. Computing the row-wise maximum on the CPU (reference)
3. Running your `rowMax` kernel on the GPU
4. Comparing all `rows` output values within `1e-3` absolute tolerance

Input `verify R C` generates an R×C matrix where `matrix[i*C+j] = (i*C+j) % 100 + 1` and checks your output matches the CPU reference. Output is `1` if correct, `0` otherwise.

**In real-world profiling**, you would additionally use `ncu` to measure actual metrics:
```bash
# Full profiling suite
ncu --set full ./program

# Specific metrics for memory efficiency
ncu --metrics l1tex__t_sectors_pipe_lsu_mem_global_op_ld.sum,sm__warps_executed_per_sm.avg ./program

# Roofline analysis
ncu --section Roofline ./program
```

Expected metrics for a coalesced kernel:
- **Global Load Efficiency**: >90% (vs ~3% for broken pattern)
- **L1 Hit Rate**: Variable (depends on reuse)
- **SM Active Time**: Lower for memory-bound kernels

## Constraints

- Matrix dimensions: `rows * cols` where `1 <= rows <= 1024` and `1 <= cols <= 4096`
- Use `blockDim.x = 256` threads per block (8 warps)
- Each block processes exactly one row
- Results must be correct: `results[i] = max(matrix[i * cols : (i+1) * cols])`
- **Coalescing requirement**: Adjacent threads (tid 0-31, 32-63, etc.) must read adjacent memory locations
- Do not modify the provided `solve()` function
- For the harness verification, use `ncu` if available, or run with timing comparison

---solution---
__global__ void rowMax(const float* matrix, float* results, int rows, int cols) {
    int row = blockIdx.x;
    int tid = threadIdx.x;

    // Shared memory for block-wide reduction
    __shared__ float sharedMax[BLOCK_SIZE];

    // Each thread loads multiple contiguous elements (coalesced access)
    // Thread 0: cols[0], cols[8], cols[16], ... (stride = BLOCK_SIZE)
    // Thread 1: cols[1], cols[9], cols[17], ...
    // This ensures adjacent threads read adjacent memory = coalesced
    float localMax = -INFINITY;
    for (int col = tid; col < cols; col += BLOCK_SIZE) {
        float val = matrix[row * cols + col];
        if (val > localMax) localMax = val;
    }

    // Store thread's local max in shared memory
    sharedMax[tid] = localMax;
    __syncthreads();

    // Tree reduction in shared memory to find block maximum
    for (int stride = BLOCK_SIZE / 2; stride > 0; stride >>= 1) {
        if (tid < stride) {
            float other = sharedMax[tid + stride];
            if (other > sharedMax[tid]) {
                sharedMax[tid] = other;
            }
        }
        __syncthreads();
    }

    // Thread 0 writes the result
    if (tid == 0) {
        results[row] = sharedMax[0];
    }
}

## Approach

The key insight for coalesced memory access: **adjacent threads should read adjacent memory addresses**. When threads 0-31 (a warp) execute a load instruction, the GPU can combine their requests into a single memory transaction if the addresses are contiguous.

**The broken pattern** (non-coalesced):
```cpp
for (int col = tid; col < cols; col += 32) {
    float val = matrix[row * cols + col];
```
- Thread 0 reads: cols[0], cols[32], cols[64], ...
- Thread 1 reads: cols[1], cols[33], cols[65], ...
- Thread 31 reads: cols[31], cols[63], cols[95], ...

Each thread in the warp accesses addresses that are **32 floats apart**. This creates 32 separate memory transactions instead of 1 coalesced transaction. Global load efficiency drops to ~3%.

**The fixed pattern** (coalesced):
```cpp
for (int col = tid; col < cols; col += BLOCK_SIZE) {
    float val = matrix[row * cols + col];
```
- Thread 0 reads: cols[0], cols[256], cols[512], ...
- Thread 1 reads: cols[1], cols[257], cols[513], ...
- Thread 255 reads: cols[255], cols[511], cols[767], ...

Now threads 0-31 (a warp) read cols[0-31] in the first iteration. This is **contiguous memory** = one coalesced transaction. Global load efficiency is ~100%.

**The reduction**: After loading, each thread finds the max of its chunk. Then a tree reduction combines all thread results:
1. Each thread writes its local max to shared memory
2. Synchronize all threads
3. Reduce in shared memory: `sharedMax[tid] = max(sharedMax[tid], sharedMax[tid + stride])`
4. Thread 0 writes the final result

## Complexity

- **Time complexity**: O(cols / BLOCK_SIZE) per row
  - Each thread loads `cols / BLOCK_SIZE` elements
  - Tree reduction takes O(log BLOCK_SIZE) steps
- **Memory**: O(BLOCK_SIZE) shared memory per block
- **Global load efficiency**: ~100% with coalesced access, ~3% without

## Key Insights

1. **Coalescing is about stride**: The stride in your for-loop should equal `BLOCK_SIZE`, not `warpSize` (32). Using stride=32 causes each warp to access strided addresses; using stride=256 means each warp accesses contiguous addresses.

2. **Nsight Compute metrics to check**:
   - `sm__throughput.avg.pct_of_peak_sustained_elapsed`: SM active time (should be high for compute-bound, low for memory-bound)
   - `l1tex__t_sectors_pipe_lsu_mem_global_op_ld.sum`: Global load sectors
   - `sm__warps_executed_per_sm.avg`: Warp execution efficiency
   - Look at "Global Load Efficiency" in Memory Workload Analysis

3. **Roofline positioning**: A memory-bound kernel sits on the memory bandwidth roof. Moving it to the compute roof requires increasing arithmetic intensity (more FLOPs per byte).

4. **Tree reduction pattern**: The shared memory reduction uses `stride` going from `BLOCK_SIZE/2` down to 1. At each step, thread `tid` combines with thread `tid + stride`.

## Common Mistakes

- **Using stride=32 instead of stride=BLOCK_SIZE**: This is the broken pattern that causes non-coalesced access. The problem explicitly shows the buggy code with `col += 32`. Don't copy it!

- **Missing __syncthreads()**: After loading elements and before the reduction, you must synchronize all threads. Otherwise, some threads may not have written their local max yet.

- **Incorrect shared memory size**: The shared array needs `BLOCK_SIZE` entries (one per thread), not `warpSize` entries.

- **Forgetting the tree reduction**: Just loading coalescedly isn't enough. You still need to combine all thread results using a tree reduction in shared memory.

- **Not handling edge cases**: If `cols < BLOCK_SIZE`, some threads won't load anything. Initialize `localMax = -INFINITY` to handle this.

## Nsight Compute Workflow

### Step-by-Step Profiling Guide

**1. Run the baseline (broken) kernel:**
```bash
ncu --launch-skip 0 --launch-count 1 --set full ./program
```

**2. Check Global Load Efficiency:**
In the Nsight Compute report, find "Memory Workload Analysis":
- Broken kernel: ~3% global load efficiency
- Fixed kernel: ~100% global load efficiency

**3. Check Roofline Positioning:**
```bash
ncu --section Roofline ./program
```
- The broken kernel will sit on the **memory bandwidth roof** (memory-bound)
- The fixed kernel may still be memory-bound but achieves higher effective bandwidth

**4. Analyze Stall Reasons:**
In "Source Statistics" or "Stall Statistics":
- **Long Scoreboard**: Waiting for global memory loads (high in broken kernel)
- **Execution Dependency**: Waiting for previous instructions (high in fixed kernel)
- **Memory Throttle**: Stalled due to memory bandwidth limits

**5. Check Warp State:**
```bash
ncu --metrics sm__warps_executed_per_sm.avg,sm__cycles_elapsed.sum ./program
```
- Look for "Idle" vs "Active" vs "Stalled" warp cycles
- Broken kernel has high "Stalled - Long Scoreboard"
- Fixed kernel has more "Active" cycles

### Typical Profiling Results

| Metric | Broken Kernel | Fixed Kernel |
|---|---|---|
| Global Load Efficiency | ~3% | ~100% |
| L1 Hit Rate | ~10-20% | ~10-20% |
| SM Active Time | ~30% | ~30% |
| Stall - Long Scoreboard | ~50% | ~10% |
| Stall - Execution Dependency | ~10% | ~30% |
| Achieved Bandwidth | ~100 GB/s | ~1000 GB/s |

### Interpreting Results

**Memory-bound kernel** (what you have here):
- Characteristic: Low arithmetic intensity (< 1 FLOP/byte)
- Roofline: Sits on memory bandwidth line
- Fix: Improve memory efficiency (coalescing) or fuse operations

**Compute-bound kernel**:
- Characteristic: High arithmetic intensity (> 10 FLOPs/byte)
- Roofline: Sits on arithmetic bandwidth line
- Fix: Use Tensor Cores, increase instruction-level parallelism
