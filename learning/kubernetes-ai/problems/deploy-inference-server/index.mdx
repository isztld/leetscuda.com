## Background

**vLLM** is a high-throughput LLM inference engine that uses PagedAttention to maximise KV-cache utilisation. Deploying it on Kubernetes means writing a `Deployment` that schedules the pod on a GPU node, exposes the OpenAI-compatible REST API on port 8000, and includes the proper resource requests so the scheduler places the pod correctly.

On Kubernetes, GPU resources are exposed as extended resources under `nvidia.com/gpu`. A container must *request* a GPU to be scheduled on a GPU node, and should *limit* it to the same value — GPUs are not time-shared like CPU cores.

A `Service` of type `ClusterIP` makes the server reachable inside the cluster. A `readinessProbe` on `/health` (HTTP GET, port 8000) lets the kubelet hold traffic until the container is ready.

## What to Implement

Write the Kubernetes manifest (YAML) that:

1. Creates a `Deployment` named `vllm-server` in the `inference` namespace with 1 replica
2. Container image: `vllm/vllm-openai:v0.4.0`
3. Container command: `["python", "-m", "vllm.entrypoints.openai.api_server", "--model", "meta-llama/Llama-2-7b-chat-hf", "--port", "8000"]`
4. Requests and limits: `nvidia.com/gpu: 1`, `memory: 20Gi`, `cpu: "4"`
5. Readiness probe: HTTP GET `/health` port 8000, initialDelaySeconds 60, periodSeconds 15
6. Creates a `Service` named `vllm-service` of type `ClusterIP` targeting port 8000

## Constraints

- The `Deployment` and `Service` must be in the same file (use `---` YAML separator)
- Use `app: vllm` as the selector label on both `Deployment` and `Service`
- Namespace `inference` must be specified in both `metadata` sections
- Do not use `imagePullPolicy: Always` (omit it; the default is fine)

## Example

A correctly deployed vLLM server can be queried inside the cluster:

```bash
curl http://vllm-service.inference.svc.cluster.local:8000/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model": "meta-llama/Llama-2-7b-chat-hf", "messages": [{"role": "user", "content": "Hello"}]}'
```

## Approach

A minimal vLLM deployment requires two Kubernetes resources: a `Deployment` and a `Service`.

**Deployment key points:**
- `namespace: inference` in both Deployment and Service metadata
- GPU resource request must use the extended resource name `nvidia.com/gpu`
- The readiness probe uses `httpGet` so kubelet makes an HTTP GET to `/health:8000`; `initialDelaySeconds: 60` gives the model time to load before health checks begin

**Service:** A `ClusterIP` Service with matching `app: vllm` selector routes cluster traffic to the pods.

```yaml
resources:
  requests:
    nvidia.com/gpu: "1"
    memory: 20Gi
    cpu: "4"
  limits:
    nvidia.com/gpu: "1"
    memory: 20Gi
    cpu: "4"
```

## Key Insight

GPU resources in Kubernetes are **non-divisible** extended resources. Unlike CPU (millicores) or memory (bytes), a GPU is either assigned to a pod entirely or not at all. Setting request == limit is required; Kubernetes rejects a pod with mismatched GPU request/limit.

## Common Mistakes

- **Using `requests` without matching `limits` for GPUs**: Kubernetes requires these to be equal for extended resources.
- **Missing namespace on Service**: Both `Deployment` and `Service` must independently specify `namespace: inference`.
- **Wrong probe path**: The vLLM OpenAI API server exposes `/health`.
