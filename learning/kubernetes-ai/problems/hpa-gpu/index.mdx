## Background

The Kubernetes **Horizontal Pod Autoscaler (HPA)** scales the number of pod replicas based on observed metrics. Out of the box it supports CPU utilisation, but for GPU inference servers the relevant signal is **GPU utilisation** or **requests-per-second**, surfaced via the DCGM Exporter → Prometheus → `custom.metrics.k8s.io` pipeline.

The DCGM Exporter (part of the NVIDIA GPU Operator) publishes Prometheus metrics like `DCGM_FI_DEV_GPU_UTIL` (GPU utilisation 0–100). The **Prometheus Adapter** bridges Prometheus metrics into the Kubernetes metrics API, making them available to the HPA under `custom.metrics.k8s.io/v1beta1`.

A typical production target is 60–70% GPU utilisation — high enough to amortise the fixed overhead per forward pass, low enough to absorb traffic spikes without queueing.

## What to Implement

Write the `HorizontalPodAutoscaler` manifest that:

1. Targets the `Deployment` named `vllm-server` in the `inference` namespace
2. Scales between `minReplicas: 1` and `maxReplicas: 8`
3. Uses a `Pods` metric named `dcgm_fi_dev_gpu_util` with `averageValue: "60"` (percent)
4. Sets `scaleDown.stabilizationWindowSeconds: 300` and `scaleUp.stabilizationWindowSeconds: 0`

Also write a `PodMonitor` (Prometheus Operator CRD) that scrapes the vLLM pod's `/metrics` endpoint on port `8000` every 15 seconds so that DCGM metrics are collected.

## Constraints

- HPA `apiVersion` must be `autoscaling/v2`
- The `PodMonitor` must be in the `monitoring` namespace and use `namespaceSelector.matchNames: [inference]`
- Both resources must be valid YAML parseable by `kubectl apply`

## Example

Under sustained load:
- At 80% GPU utilisation → HPA scales up toward `ceil(currentReplicas * 80/60) = 2` replicas
- After 5 minutes of low utilisation → HPA scales back down (stabilisation window prevents flapping)

## Approach

**HPA manifest (`autoscaling/v2`):**
- `scaleTargetRef` points to the Deployment you want to scale
- Under `metrics`, use `type: Pods` with `pods.metric.name: dcgm_fi_dev_gpu_util` and `target.type: AverageValue` with `averageValue: "60"` — the HPA scales pods until the average GPU utilisation across all pods is 60%
- `behavior.scaleDown.stabilizationWindowSeconds: 300` prevents flapping by requiring 5 minutes of low utilisation before scaling down

**PodMonitor:**
- Must be in the `monitoring` namespace (where the Prometheus Operator watches for CRDs)
- `namespaceSelector.matchNames: [inference]` tells Prometheus to scrape pods in the inference namespace
- `podMetricsEndpoints` specifies port `8000` and path `/metrics`

## Key Insight

The stabilisation window asymmetry (`scaleDown=300, scaleUp=0`) is intentional: scale up fast (don't queue requests), scale down slow (avoid thrashing when load fluctuates). A GPU pod takes 60+ seconds to start, so premature scale-down is expensive.

## Common Mistakes

- **Using `autoscaling/v1` or `autoscaling/v2beta2`**: Custom metrics require `autoscaling/v2`.
- **PodMonitor in wrong namespace**: The PodMonitor itself goes in `monitoring`, not `inference`. The `namespaceSelector` tells Prometheus which namespace to scrape.
- **Using `targetAverageValue` (old API)**: In `autoscaling/v2`, use `target.type: AverageValue` and `target.averageValue`, not the v1-style `targetAverageValue` field.
