# Kernel Fusion

Combining multiple independent GPU operations into a single kernel is one of the most effective techniques for improving memory-bound workloads. In modern LLM training and inference, kernel fusion is the difference between acceptable and production-scale performance.

This article covers why fusion matters, the tradeoffs involved, and how to reason about fusion decisions when optimizing CUDA kernels.

---

## Why This Matters for Interviews

Kernel fusion appears in principal-level interviews across three categories:

**System design questions:** "Design a memory-efficient transformer implementation." You must identify which operations can be fused, justify fusion decisions using arithmetic intensity, and recognize the limits of fusion (register pressure, shared memory constraints).

**Kernel optimization debugging:** "This kernel is 3x slower than PyTorch." The answer often involves recognizing that multiple kernel launches with intermediate materialization is causing excess memory traffic.

**Architecture tradeoff discussions:** "Should we use CUDA Graphs or manual kernel fusion?" You need to explain that CUDA Graphs reduce launch overhead while fusion reduces per-kernel memory traffic—these are complementary, not interchangeable.

## The Arithmetic Intensity Argument

Arithmetic intensity (AI) is the ratio of FLOPs to bytes moved in a kernel:

```
AI = Total FLOPs / Total Bytes Moved (FLOPs/byte)
```

Modern GPUs have a massive gap between arithmetic bandwidth and memory bandwidth. On an H100:

- **Arithmetic bandwidth:** 989 TFLOPs/s (BF16 tensor cores)
- **Memory bandwidth:** 3.35 TB/s
- **Ridge point:** 295 FLOPs/byte

Kernels below the ridge point are memory-bound. Fusion increases arithmetic intensity by eliminating redundant memory passes.

### Example: Fused Residual + LayerNorm

Consider the `fused_residual_forward.cu` kernel from `llm.c`. The unfused version executes:

1. **Residual kernel:** `out = inp1 + inp2`
   - Reads: `inp1` (N elements), `inp2` (N elements)
   - Writes: `out` (N elements)
   - Total: 3N reads/writes

2. **LayerNorm kernel:** `normed = (out - mean) / std * weight + bias`
   - Reads: `out` (N elements), `weight` (C elements), `bias` (C elements)
   - Writes: `normed` (N elements)
   - Total: 3N + 2C reads/writes

**Unfused total:** 6N + 2C memory operations

The fused kernel executes:

1. **Single kernel:** Compute residual, compute LayerNorm stats, apply normalization
   - Reads: `inp1` (N), `inp2` (N), `weight` (C), `bias` (C)
   - Writes: `normed` (N), `residual` (N)
   - Total: 4N + 2C memory operations (intermediate residual kept in registers/SRAM)

**Fused total:** 4N + 2C memory operations

For typical LLM dimensions (N = B×T = 8×1024 = 8192, C = 768), fusion eliminates roughly 25% of memory traffic.

## Common Fusion Patterns in LLMs

### 1. Elementwise Operation Fusion

The simplest fusion: combining multiple elementwise operations into one kernel.

```cpp
// Unfused: 3 kernel launches, 2 intermediate writes
y = linear(x)           // C×N output
y = bias_add(y, b)      // read/write N elements
z = gelu(y)             // read/write N elements
w = dropout(z)          // read/write N elements

// Fused: 1 kernel launch, no intermediates
z = gelu(linear(x) + b) // read C×N weights, write N output
```

This pattern appears in transformer blocks: `linear -> bias -> gelu -> dropout` fuses to a single forward kernel.

### 2. GELU + Bias Fusion

From `gelu_forward.cu`:

```cpp
// GELU(tanh approximation):
// GELU(x) = 0.5 * x * (1 + tanh(sqrt(2/pi) * (x + 0.044715 * x^3)))

// Fused: y = GELU(x + bias)
__global__ void gelu_bias_kernel(floatX* out, const floatX* x,
                                  const floatX* bias, int N) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < N) {
        float xi = (float)x[idx] + (float)bias[idx];
        float x3 = xi * xi * xi;
        float tanh_arg = 0.79788456f * (xi + 0.044715f * x3);
        float tanh_out = tanh(tanh_arg);
        out[idx] = (floatX)(0.5f * xi * (1.0f + tanh_out));
    }
}
```

The fusion eliminates reading `x` again after bias addition. For vectorized implementations using `float4` loads, this reduces memory bandwidth by ~33%.

### 3. Attention + Softmax + Value Fusion

FlashAttention is the canonical fusion example. Standard attention executes:

```
S = Q @ K^T / sqrt(d_k)      // O(N^2) HBM write
A = softmax(S)               // O(N^2) HBM read + O(N^2) write
O = A @ V                    // O(N^2) HBM read
```

Total HBM traffic: O(N^2) for S and A materialization.

FlashAttention fuses all three operations:

```
// Tile over Q, K, V blocks
for each Q_block:
    for each K,V_block:
        S = Q_block @ K_block^T / sqrt(d_k)
        m_new = max(m, rowmax(S))
        l_new = exp(m - m_new) * l + rowsum(exp(S - m_new))
        O_block = (exp(m - m_new) * O_block + exp(S - m_new) @ V_block) / l_new
```

The key insight: online softmax maintains running statistics (m, l) and rescales partial outputs. No O(N^2) materialization. HBM traffic drops from O(N^2) to O(N).

### 4. LayerNorm + Residual Fusion

The `fused_residual_forward.cu` example fuses:

```
residual = inp1 + inp2
mean = mean(residual)
var = variance(residual)
normed = (residual - mean) / sqrt(var + eps) * weight + bias
```

Implementation strategy (kernel_v4+):

- One warp handles one token (ensures coalesced access to C-dimensional vectors)
- Use `warpReduceSum` for parallel reduction across threads
- Vectorized 128-bit loads (`Packed128<floatX>`) to amortize load latency
- Streaming writes with cache hint (`store128cs`)

```cpp
// kernel_v4: vectorized, single-pass stats
__global__ void fused_residual_forward_kernel4(floatX* residual, floatX* normed,
    floatX* mean, floatX* rstd, const floatX* inp1, const floatX* inp2,
    const floatX* weight, const floatX* bias, int N, int C) {

    using x128 = Packed128<floatX>;
    int idx = blockIdx.x * blockDim.y + threadIdx.y;
    if (idx >= N) return;

    // Each warp handles one token (C elements)
    float sum = 0.0f;
    float sum_sq = 0.0f;

    // Phase 1: compute residual, accumulate stats
    for (int c = threadIdx.x * x128::size; c < C; c += 32 * x128::size) {
        const x128 in1 = load128cs(inp1 + c);
        const x128 in2 = load128cs(inp2 + c);
        x128 out;
        for (int k = 0; k < x128::size; ++k) {
            out[k] = (floatX)((float)in1[k] + (float)in2[k]);
            sum += (float)out[k];
            sum_sq += (float)out[k] * (float)out[k];
        }
        store128(residual + c, out);
    }

    sum = warpReduceSum(sum);
    sum_sq = warpReduceSum(sum_sq);

    float m = sum / C;
    float v = sum_sq / C - m * m;
    float s = rsqrtf(v + eps);

    // Phase 2: apply normalization (zigzag loop for register pressure)
    // ...
}
```

## Common misconceptions

❌ **Wrong**: "Fusion always improves performance"

✓ **Correct**: Fusion only helps memory-bound kernels. If each operation is compute-bound (above the roofline ridge point), fusion does not improve throughput — consider a GEMM at AI = 500 FLOPs/byte on H100 (ridge = 295): fusing it with a bias addition adds FLOPs but the kernel is already compute-saturated. Fusion trades memory bandwidth for compute; if the kernel is compute-bound, you're adding FLOPs without reducing memory traffic, which may hurt performance.

❌ **Wrong**: "Fusion increases register pressure significantly"

✓ **Correct**: Fusion is a tradeoff, not a free upgrade. While fusion can increase live variables, careful design mitigates this:
- Process elements in tiles (reduce live state)
- Use streaming loops (compute + write, then reuse registers)
- Apply vectorized loads (amortize register usage across 4 elements)

The kernel_v6 in `fused_residual_forward.cu` uses shared memory to cache weights and bias, trading SRAM for register pressure. Tradeoffs include:
- Register pressure vs occupancy
- Shared memory usage vs occupancy
- Kernel complexity vs maintainability

❌ **Wrong**: "CUDA Graphs and kernel fusion are the same"

✓ **Correct**: They solve different problems. CUDA Graphs capture a sequence of kernel launches and replay them with reduced overhead; this is distinct from fusion:
- **CUDA Graphs:** Reduce launch overhead (typically ~10-20 microseconds per launch)
- **Kernel Fusion:** Reduces per-kernel memory traffic (can be 2x-10x bandwidth improvement)

For memory-bound workloads, fusion matters more. For latency-sensitive inference with small batches, CUDA Graphs matter more. Use both when possible. Example: A transformer block with 10 unfused kernels:
- CUDA Graphs save: 10 × 15μs = 150μs launch overhead
- Fusion saves: 6N memory reads (e.g., 6 × 8192 × 4 bytes = 192KB per layer)

For inference at batch 1, Graphs may dominate. For training at batch 2048, fusion dominates.

❌ **Wrong**: "Online softmax is approximate"

✓ **Correct**: Online softmax is numerically exact. The algorithm maintains running statistics (m, l) and rescales partial outputs, producing numerically exact results (within floating-point precision). FlashAttention's key insight: online softmax enables chunked computation without materializing the full attention matrix. The algorithm is exact—this is why FlashAttention can be used in training where gradients must match naive attention.

## Interview patterns

**Question**: "Design a transformer forward pass that fits 1B parameters in 80GB GPU memory."

**Answer**:
1. Identify fusion opportunities: `QKV_proj -> bias -> gelu` can fuse; `attention -> softmax -> @V` can fuse via FlashAttention
2. Quantify memory savings: unfused softmax materializes O(N^2); FlashAttention uses O(N)
3. Tradeoff analysis: FlashAttention backward recomputes attention weights (2x FLOPs) vs storing O(N^2) matrix
4. Mention gradient checkpointing: trade FLOPs for activation memory (AI increases)

**Question**: "A kernel doing `y = relu(linear(x) + bias)` is 3x slower than expected. Diagnose."

**Answer**:
1. Check memory footprint: likely separate kernels for linear, bias_add, relu
2. Identify bottleneck: memory-bound (AI ~ 1/6 for SAXPY-like ops)
3. Measure intermediate traffic: each materialization reads/writes N elements
4. Propose fusion: single kernel reduces memory ops from 5N to 3N (40% improvement)

**Question**: "For batch size 1 inference, should I prioritize CUDA Graphs or kernel fusion? Why?"

**Answer**:
1. Compute latency: small batches mean fewer elements per launch
2. Launch overhead: 10-20μs per kernel launch becomes significant
3. Memory bandwidth: less memory traffic per step, so fusion benefits are smaller
4. Recommendation: both, but prioritize Graphs first (linear scan of 10 kernels = 150μs overhead)

### Fusion Heuristics

| Scenario | Recommendation | Rationale |
|----------|---------------|-----------|
| Elementwise ops (relu, gelu, bias) | **Fuse** | Memory-bound, trivial compute overhead |
| GEMM + elementwise | **Fuse if** AI < ridge point | GEMM dominates, fusion adds minimal compute |
| Attention + softmax | **Fuse (FlashAttention)** | O(N^2) → O(N) memory traffic |
| Sequential matmuls | **Do not fuse** | Each matmul is compute-bound |
| Inference batch < 8 | **Prioritize CUDA Graphs** | Launch overhead dominates |
| Training batch > 128 | **Prioritize fusion** | Memory bandwidth dominates |

## Summary

Kernel fusion is the practice of combining multiple GPU operations into a single kernel to reduce intermediate memory traffic. For memory-bound workloads, fusion can provide 2x-10x speedups by eliminating redundant HBM reads/writes.

Key takeaways:

- **Fusion increases arithmetic intensity** by eliminating intermediate materializations
- **FlashAttention is the canonical fusion example**, reducing attention memory traffic from O(N^2) to O(N)
- **Fusion has tradeoffs**: register pressure, shared memory usage, kernel complexity
- **CUDA Graphs and fusion are complementary**: Graphs reduce launch overhead; fusion reduces memory traffic
- **Interview questions** test your ability to identify fusion opportunities and justify decisions using arithmetic intensity

When designing GPU kernels for LLMs, always ask: "What intermediate data am I materializing that could be kept in registers or SRAM?" The answer often points directly to a fusion opportunity.
