# Hopper TMA & Warpgroup MMA

## Why this matters for interviews

TMA and WGMMA questions signal that the interviewer is probing the frontier of GPU kernel optimization. These are not interview topics for general CUDA engineers — they appear at NVIDIA, Meta AI, Microsoft Azure ML, and Google TPU teams building production LLM inference kernels. When an interviewer asks "How does FlashAttention 3 use TMA?" or "Explain warp specialization with WGMMA," they are testing whether you have actually written Hopper kernels or only studied Ampere-era techniques.

Understanding TMA and WGMMA is essential because they represent a fundamental architectural shift in how Hopper exposes the memory hierarchy to programmers. Previous generations required manual coordination of 32 threads to load a tile into shared memory. TMA offloads this to dedicated hardware. Similarly, WMMA (introduced in Volta) operated on a single warp; WGMMA operates on 4 warps collectively, enabling the async pipelining that powers FlashAttention 3's performance breakthroughs.

## The Hopper Architecture Advantage

Hopper (H100, compute capability 9.0) introduced two hardware units that fundamentally change how we program high-throughput LLM kernels: the Tensor Memory Accelerator (TMA) and warpgroup-level matrix multiplication (WGMMA). These units work in tandem: TMA handles async bulk copies from global memory to shared memory, while WGMMA processes tiles of data at 128-thread granularity.

**The performance impact:** FlashAttention 3 on H100 achieves ~50% of theoretical attention throughput vs. FlashAttention 2's ~30% on A100. This 1.5-2× improvement comes from TMA's ability to prefetch K/V tiles while WGMMA processes previous tiles, hiding global memory latency behind computation.

## Tensor Memory Accelerator (TMA)

### What is TMA?

The Tensor Memory Accelerator is a dedicated hardware engine in Hopper (and Blackwell) GPUs that offloads multi-dimensional memory copy coordination from the GPU's scalar pipelines. Instead of programming explicit loops where 32 threads each compute their own address and load a 4-byte value, the programmer describes the copy via a TMA descriptor (shape, stride, swizzle) and issues a single TMA instruction.

**TMA descriptor creation on host:**
```cpp
#include <cuda/tensor_map>

// Create a 2D tensor map for a matrix of shape (N, d)
cudaTensorMap_t tensorMap;
cuTensorMapEncodeTiled(
    &tensorMap,                // output descriptor
    cudaTensorMapDataTypeF32,  // data type
    2,                         // rank (2D)
    globalPtr,                 // pointer to global memory
    {N, d},                    // global shape
    {128, 64},                 // tile shape (in elements)
    {d, 1},                    // global strides (in elements)
    {32, 8},                   // box shape for reduction
    {0, 0}                     // box offsets
);
```

The TMA descriptor encodes:
- **Global memory base address** and data type
- **Tensor shape** (up to 5D)
- **Strides** between dimensions
- **Tile shape** for the copy operation
- **Swizzle pattern** for shared memory (avoids bank conflicts)

### TMA Copy Instructions

TMA copies use the `cp.async.bulk.tensor` instruction family:

```cpp
// 2D tensor copy: copies a 16×32 tile from global to shared memory
// The copy is asynchronous — issuing thread can continue work
cp.async.bulk.tensor.2d.shared::cluster.global.mbarrier::complete_txs_bytes(
    [smem_base],      // shared memory base pointer
    [globalPtr],      // global memory pointer (via tensor map)
    {16, 32},         // tile shape
    [mbarrier];       // mutex barrier for completion signaling
);
```

Key characteristics:
- **Async execution:** The issuing thread returns immediately; TMA hardware performs the copy
- **mbarrier signaling:** Threads can wait for completion using `mbarrier.try_wait`
- **Multicast support:** TMA can broadcast one copy to multiple thread blocks in a cluster
- **Swizzle:** Shared memory address is XORed with bits of the thread index to avoid bank conflicts

### TMA vs. Traditional Async Copy

Ampere introduced `cp.async` for async copies from global to shared memory, but TMA is fundamentally different:

| Feature | `cp.async` (Ampere) | `cp.async.bulk.tensor` (Hopper TMA) |
|---------|---------------------|-------------------------------------|
| Issue granularity | Single warp (32 threads) | Single thread (TMA engine) |
| Address computation | Each thread computes its own | Descriptor-based, hardware computes |
| Max copy size | 16 bytes per instruction | Up to 256 bytes per instruction |
| Multidimensional | No (flat address range) | Yes (1D-5D tensors) |
| Swizzle support | No (manual in kernel) | Yes (automatic via descriptor) |
| Multicast | No | Yes (to Thread Block Cluster) |

**Address computation savings:** For a 16×32 tile load, traditional code requires 32 threads × 4 cycles = 128 thread-cycles of address computation. TMA requires one instruction from one thread, with the TMA hardware computing all addresses internally. This reduces register pressure (no need to store per-thread addresses) and frees arithmetic bandwidth for other work.

### TMA in FlashAttention 3

FlashAttention 3 demonstrates TMA's power through warp specialization. The thread block is partitioned into:

- **Producer warps (1-2 warps):** Load K/V tiles from global memory to shared memory using TMA
- **Consumer warps (2-3 warps):** Execute WGMMA operations on data already in shared memory

While consumer warps execute WGMMA on tile N, producer warps prefetch tile N+1 using TMA. This overlap hides the global memory latency entirely, limited only by the L2 cache bandwidth.

```cpp
// Simplified FlashAttention 3 TMA prefetch pattern
// Producer warp:
while (tile_index < num_tiles) {
    TMA_load_async(K_tile[tile_index + 1], smem_K);
    tile_index++;
}

// Consumer warps (interleaved):
for (int i = 0; i < num_tiles; i++) {
    mbarrier_wait(K_tile[i]);      // Wait for TMA to complete
    wgmma_compute(K_tile[i], Q);   // Process while next tile loads
}
```

## Warpgroup Matrix Multiplication (WGMMA)

### What is a Warpgroup?

A warpgroup is a set of 4 contiguous warps where the first warp's rank is a multiple of 4. In PTX:

```cpp
int linearIdx = (%tid.x + %tid.y * %ntid.x + %tid.z * %ntid.x * %ntid.y);
int warpRank = linearIdx / 32;
int warpgroupId = warpRank / 4;
```

For an 8-warp thread block:
- **Warpgroup 0:** Warps 0, 1, 2, 3 (warpranks 0-3)
- **Warpgroup 1:** Warps 4, 5, 6, 7 (warpranks 4-7)

The warp-rank alignment restriction exists because Hopper SMs contain four subunits, each with its own warp scheduler and Tensor Core. Warpgroups map to these subunits for efficient dispatch.

### WGMMA Instructions

WGMMA operates on 4 warps (128 threads) collectively. A single `wgmma.mma_async` instruction processes a much larger tile than WMMA:

| Instruction | Thread Granularity | Tile Shape (M×N×K) | Data Types |
|-------------|-------------------|-------------------|------------|
| WMMA (Ampere) | 1 warp (32 threads) | 16×16×16 | FP16, BF16, TF32 |
| WGMMA (Hopper) | 4 warps (128 threads) | 64×256×16 | FP16, BF16, FP8 |

**WGMMA async MMA example (PTX):**
```ptx
// wgmma.mma_async.sync.aligned.m64n256k16.f32.bf16.bf16
// Produces: 4 warps accumulate a 64×256 FP32 result
// Consumes: A = 64×16 BF16 in registers, B = 16×256 BF16 in shared memory

wgmma.mma_async.sync.aligned.m64n256k16.f32.bf16.bf16(
    %reg[C0..7],   // 8 FP32 registers per warp for accumulator (64×256 distributed)
    %reg[A0..15],  // A register fragment (64×16 distributed across 4 warps)
    [%smem_B],     // B in shared memory (16×256)
    %pred
);
```

The 64×256 output is distributed across 128 threads. Each thread accumulates 64×256 / 128 = 128 FP32 elements in its registers.

### WGMMA Async and Pipeline Barriers

The `async` in `wgmma.mma_async` means non-blocking execution. Other warps can execute while WGMMA runs in the background. This enables the **warp specialization** pattern:

```ptx
// Producer warps: load data
cp.async.bulk.tensor.2d.shared::cluster.global.mbarrier::complete_txs_bytes(
    [smem_A], [global_A], {64, 16}, [barrier_A]);

// Consumer warps: wait and compute
mbarrier.try_wait(p0, [barrier_A]);
@p0 wgmma.mma_async...([C], [A], [smem_B]);

// Fence and commit for synchronization
wgmma.fence();
wgmma.commit_group();

// Wait for all prior WGMMA operations to complete
wgmma.wait_group(0);  // Wait for all groups before this one
```

Pipeline barrier primitives:
- `wgmma.fence()`: Marks a synchronization point in the pipeline
- `wgmma.commit_group()`: Commits all pending WGMMA operations
- `wgmma.wait_group(n)`: Waits for the first n groups to complete (0 = all)

### Warp Specialization Pattern

Warp specialization is the key to hiding latency in Hopper kernels. The thread block partitions warps into distinct roles:

**FlashAttention 3 Forward Pass:**
```
Thread Block (160 threads = 5 warps):
├─ Producer Warps (2 warps = 64 threads)
│  ├─ Load Q tile from global → shared (using TMA)
│  ├─ Load K tile from global → shared (using TMA)
│  └─ Load V tile from global → shared (using TMA)
│
└─ Consumer Warps (3 warps = 96 threads)
   ├─ Execute WGMMA: Q @ K^T → attention scores
   ├─ Apply softmax (in register)
   └─ Execute WGMMA: scores @ V → output tile
```

The producer warps load tile N+1 while consumer warps process tile N. This requires:
- **Double buffering** in shared memory (two buffers per matrix)
- **Barrier synchronization** between producer and consumer
- **Predicate execution** to handle the final tile without overfetching

### WGMMA Tile Shapes for H100

H100 supports the following WGMMA shapes:

| Data Type | M | N Range | K | Notes |
|-----------|---|---------|---|-------|
| BF16/FP16 | 64 | 8-256 | 16 | Most common for LLMs |
| FP8 | 64 | 8-256 | 32 | Double K for FP8 (8-bit) |
| FP8 E4M3 | 64 | 8-128 | 64 | Special FP8 variant |
| FP8 E5M2 | 64 | 8-128 | 64 | Special FP8 variant |

The N dimension can be any multiple of 8 from 8 to 256. This flexibility allows tuning for different attention head sizes and sequence lengths.

## FlashAttention 3: TMA + WGMMA Integration

FlashAttention 3 (Dao et al. 2024) demonstrates the full power of TMA + WGMMA integration. Key innovations:

### 1. Warp Specialization for Memory/Compute Overlap

FlashAttention 3 partitions the thread block into producer and consumer warps. Producers prefetch K/V tiles using TMA while consumers execute WGMMA on current tiles. This overlap is only possible because:
- TMA copies are asynchronous (don't block the issuing warp)
- WGMMA is async (doesn't block consumer warps)
- Barriers coordinate completion without busy-waiting

### 2. Loop Order Optimization

FlashAttention 2 (A100) iterates over query tiles externally and KV tiles internally. FlashAttention 3 (H100) can iterate over KV tiles externally because TMA loads are so fast. This changes the parallelization strategy:

**FA2 (A100) parallelism:**
- Outer loop: query tiles (parallel across thread blocks)
- Inner loop: KV tiles (sequential within block)

**FA3 (H100) parallelism:**
- Outer loop: KV tiles (prefetched by TMA)
- Inner loop: query tiles (processed by WGMMA)

The new order enables better utilization when batch size × num_heads is small (common in inference).

### 3. Reduced Shared Memory Traffic

FA2's split-K strategy required 4 warps to communicate partial sums via shared memory. FA3's split-Q strategy lets each warp work independently, eliminating inter-warp shared memory sync. Combined with TMA's descriptor-based loads (no manual address computation), shared memory traffic drops ~2×.

### 4. Performance Numbers

**H100 FlashAttention 3 (batch=2, seq=8192, heads=32, head_dim=128):**
- Forward pass: ~335 TFLOPS/s (52% of H100 FP16 peak)
- Backward pass: ~168 TFLOPS/s (50% of forward throughput)
- Memory efficiency: 9.8× fewer HBM accesses vs. standard attention

## Common Misconceptions

**❌ Wrong:** TMA is just async copy (cp.async from Ampere) with a fancier name.

**✓ Correct:** TMA is a separate hardware engine with a descriptor-based programming model. `cp.async` is warp-issued and computes addresses per-thread. TMA is thread-issued and computes all addresses internally from the descriptor. TMA also supports multicast and swizzle, which `cp.async` does not.

**❌ Wrong:** TMA works on all CUDA compute capabilities.

**✓ Correct:** TMA requires compute capability 9.0+ (Hopper and Blackwell). Ampere (8.0-8.9) and earlier do not have TMA hardware.

**❌ Wrong:** TMA automatically handles tiled copying for any memory layout.

**✓ Correct:** TMA requires the programmer to correctly set up the TMA descriptor with shape, strides, and tile shape. Incorrect descriptors cause undefined behavior. The descriptor is not a magic bullet — it must match the memory layout.

**❌ Wrong:** WGMMA is just WMMA for 4 warps in a loop.

**✓ Correct:** WGMMA is a fundamentally different instruction class designed for async pipelining. WMMA is synchronous and warp-local. WGMMA is async, operates across 4 warps, and integrates with TMA barriers. The SASS generated is HGMMA (Hopper GEMM), not HMMA.

**❌ Wrong:** WGMMA can be used with any thread block size.

**✓ Correct:** WGMMA requires exactly 4 warps (128 threads) per warpgroup. For a single-warpgroup kernel, use exactly 128 threads. Multi-warpgroup kernels use 256, 384, etc., but each WGMMA operation involves exactly 4 warps.

**❌ Wrong:** Reducing FLOPs always makes kernels faster.

**✓ Correct:** FlashAttention has more FLOPs than standard attention (due to recomputation in backward), but it is faster because it drastically reduces HBM traffic. Memory bandwidth is typically the bottleneck, not compute, for attention.

## Interview Patterns

**"Explain how FlashAttention 3 uses TMA and WGMMA together."**

Start with the problem: attention requires O(N^2) operations but is memory-bandwidth limited because standard implementations materialize the N×N attention matrix. FlashAttention tiles the computation so each tile fits in SRAM (H100: 228KB shared memory per SM).

On Hopper, FlashAttention 3 partitions the thread block into producer and consumer warps. Producers use TMA to asynchronously load K/V tiles from global memory to shared memory. Consumers use WGMMA to compute Q@K^T and @V. The async nature of both TMA and WGMMA enables overlap: while consumers process tile N, producers prefetch tile N+1.

Mention the synchronization: mbarrier objects signal TMA completion, and wgmma.wait_group ensures WGMMA results are ready before use. This is warp specialization — distinct warps perform distinct roles, coordinated by hardware barriers.

**"What is the IO complexity of FlashAttention and why does it matter?"**

From FlashAttention 1 Theorem 2: computing exact attention requires Θ(N^2 × d^2 / M) HBM accesses, where N is sequence length, d is head dimension, and M is SRAM size. Standard attention requires Θ(N × d + N^2) HBM accesses (the N^2 term from the attention matrix). For typical values (d=128, M=228KB), FlashAttention achieves ~9× fewer HBM accesses.

This matters because H100's HBM3 bandwidth (3.35 TB/s) is the bottleneck, not its 40 TFLOPS FP16 Tensor Core throughput. Reducing HBM traffic by 9× directly translates to 9× faster attention for long sequences.

**"How does WGMMA differ from WMMA in practice?"**

WMMA operates on 1 warp (32 threads) with tile shape 16×16×16. WGMMA operates on 4 warps (128 threads) with tile shapes like 64×256×16. WGMMA is async and integrates with TMA barriers for pipelining. In SASS, WMMA generates HMMA instructions, WGMMA generates HGMMA instructions.

For LLM kernels, WGMMA is preferable because:
1. Larger tiles amortize the fixed cost of starting the Tensor Core pipeline
2. Async execution enables warp specialization with TMA producers
3. 64×256 tiles match typical attention output dimensions better than 16×16

|-----------|--------------|
| TMA copy max size | 256 bytes per instruction |
| TMA multicast | Up to 8 thread blocks in a cluster |
| WGMMA tile (BF16) | 64×N×16, where N ∈ {8,16,...,256} |
| WGMMA tile (FP8) | 64×N×32, where N ∈ {8,16,...,256} |
| WGMMA threads | Exactly 128 (4 warps) |
| Shared memory per SM | 228 KB |
| Max warps per SM | 2048 |
| HBM3 bandwidth | 3.35 TB/s |
| FP16 BFLOPS (Tensor Core) | 989 TFLOPS |

**FlashAttention 3 (H100) performance:**
- FA3 vs FA2: 1.7× forward throughput improvement
- At seq=8192: ~335 TFLOPS/s achieved
- Memory bandwidth achieved: ~2.8 TB/s (84% of peak)

**TMA vs cp.async address computation:**
- cp.async: 32 threads × ~4 cycles = 128 thread-cycles per 16-byte load
- TMA: 1 thread × 1 cycle = 1 thread-cycle per 256-byte load
- Savings: ~128× fewer cycles for address computation

## When to Use TMA and WGMMA

**Use TMA when:**
- Targeting H100 or Blackwell (cc 9.0+)
- Loading multi-dimensional tensors in tiled fashion
- You need async copies overlapped with computation
- You want automatic swizzle to avoid bank conflicts
- Building attention, convolution, or reduction kernels

**Use WGMMA when:**
- Targeting H100 or Blackwell (cc 9.0+)
- Implementing GEMM or attention inner loops
- You can structure your kernel with exactly 128 threads per warpgroup
- You need async MMA with barrier synchronization
- Building custom LLM inference kernels

**When to stick with WMMA (Ampere):**
- Targeting A100, RTX 3090, or earlier GPUs
- Thread block size cannot be 128 (e.g., legacy code constraints)
- Simpler synchronization model is preferred over raw throughput
